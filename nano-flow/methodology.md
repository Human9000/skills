# 团队配置分析方法

## 四个决定性变量

从信息流配置表导出：

| 变量 | 判定方法 | 影响的管理动作 |
|------|---------|-------------|
| **枢纽结构** | 看"信息从哪来/到哪去"列——全经过一个人 = 星形，成员间有直连 = 网状 | 星形→按周推进/不设Sprint/枢纽保护；网状→可设Sprint/可并行 |
| **单点依赖** | 看"备份人"列——全是"无" = 高单点依赖 | 接口约定强制执行、优先补备份人 |
| **注意力碎片化** | 看"时间"列——有兼职/一人多角色 = 碎片化严重 | 兼职信息主动推送、接口变更电话确认 |
| **新人比例** | 看"经验"列——有🆕 = 需要学习期 | 新人2周保护期、不独立负责跨角色接口 |

## 管理方法推导

| 团队画像 | 必须做 | 不要做 |
|---------|--------|--------|
| 星形 + 高单点 + 兼职 + 新人 | 按周推/Sprint；全部接口写约定；枢纽3h保护；新人2周学习期；兼职主动推送 | 不设燃尽图；不排>3周计划；不让新人独立接口；不让枢纽做琐事 |
| 星形 + 低单点 | 同上，接口约定只覆盖关键接口 | — |
| 网状 + 低单点 | 可设2周Sprint；可并行开发 | 接口约定仍然必须写（成熟团队也会假设不一致） |

## 三个监控信号（不看图）

- 融合问题列表条目涨还是跌？（连续2周↑ → 注意枢纽精力）
- "没有排查策略"的问题占比 > 50%？（→ 能力或资源缺口）
- 本周有接口违约吗？（连续2周有 → 估算太乐观）

## 问题流转路径

遇到问题 → 群里@人（卡在哪/需要谁/什么时候要）→ 当天响应给方向 → 2天没排查策略 → 记入个人周报"卡在哪" → 周五PM用AI融合 → 周会决议 → 已解决归档
