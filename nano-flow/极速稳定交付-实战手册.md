# 极速稳定交付 —— 实战手册

> 不管你用 Scrum 还是 Kanban，跟 PMBOK 还是敏捷宣言——项目管理的终极问题只有一个：
> **"给定现在这支团队，怎么用最快最稳的方式把事做成？"**
>
> 这本手册回答这个问题。它不关心你的迭代长度是 1 周还是 2 周，
> 不关心你的燃尽图贴没贴对位置，不关心你有没有开 Sprint Review。
> 它只关心：**在当前这群人的配置下，什么动作能让项目跑得最快、最不会翻车。**

---

# 引言：为什么大多数项目管理在浪费时间

## 一个场景

团队用 Scrum 跑了三个月。站会每天开，看板每周更新，燃尽图贴在墙上。Sprint Review 有 PPT，Retro 有纪要。

三个月后核心开发者离职。你翻开项目文档——燃尽图完美贴合理想线、Retro 纪要里写着"加强沟通""提升代码质量"、看板上的卡片全部在"已完成"列。

然后你问了一句："所以他写的那个模块，现在还有谁能接手？"

没有人。

这就是形式化敏捷的终极症状：**所有仪式都在跑，但项目的核心风险——单点依赖、接口假设不一致、真正的难点没有被发现——没有一个仪式触及到。**

## 根本问题

大多数项目管理方法论回答的是"标准情况下该怎么做"。但项目从来不在标准情况下运行。你的团队不是方法论教材里的示例团队——有人是全栈枢纽、有人是兼职、有人昨天刚入职。

方法论应该跟着团队走。不是团队跟着方法论走。

## 这本手册的定位

这本手册基于两个来源：

1. **RST 敏捷项目框架**（六阶段生命周期：启动→规划→执行→监控→收尾→维护）——提供结构骨架
2. **nano-flow 极简协同体系**（接口约定 + 阻塞升级 + 难点即任务）——提供务实肌肉

骨架告诉你"什么时候该看什么"，肌肉告诉你"具体做什么"。

## 一个贯穿全书的核心理念

**给定当前团队，最小动作集合是什么？**

不是"敏捷应该怎么做"，不是"PMBOK 怎么规定的"。是：刘豪扛了三个模块、新同事还在学习、任振国是兼职——在这个配置下，什么动作是必须做的、什么动作可以做但会拖慢速度、什么动作纯粹是浪费。

这本书每一个章节都会先问这个问题，再给答案。

# 第一章：认清你的团队 —— 方法论跟着人走

## 1.1 没有"标准团队"

任何项目管理方法论都假设了一个"标准团队"——全栈、全职、同地、人数适中、角色清晰。

你的团队大概率不是这样。

所以第一步不是选方法论，是**认清你手里是一支什么团队**。

## 1.2 团队的四个决定性变量

画一张表，回答四个问题：

| 变量 | 问自己 | nanoengine-ecg 的答案 |
|------|--------|---------------------|
| **单点依赖** | 哪个人如果请假一周，哪块工作就完全停滞？ | 刘豪——编译器+Runtime+算法全链路只有他懂。固件在新同事独立之前也是单点。 |
| **注意力碎片化** | 谁同时在做好几个项目？ | 任振国（兼职，主业占大头）、李雪利（兼职）。刘豪一个人四个角色，角色切换本身就是碎片化。 |
| **新人比例** | 团队里有几个需要学习期的人？ | 新同事刚入职，MCU 开发从零开始。石天麟虽资深但 RISC-V/NPU 是新方向。 |
| **枢纽结构** | 团队信息流是星形（一个人是枢纽）还是网状（多对多）？ | 星形——所有接口都经过刘豪。非刘豪成员之间不直接协调。 |

这四个变量直接决定了你应该怎么管理。不是"Scrum 规定迭代长度 1-4 周所以我们要选一个"——是你的团队结构决定了你该用什么样的节奏和仪式。

## 1.3 不同配置的适配策略

### 星形枢纽 + 高单点依赖（nanoengine 当前配置）

这是最常见的小团队配置：一个人懂全局，其他人围绕他。

- **不要做：** 多角色并行开发（接口假设会悄无声息地漂移）、让枢纽人物同时做太多事（他卡住了所有人都卡住）
- **必须做：** 枢纽人物每天和各方向同步一次（15 分钟足够）、所有接口必须有书面约定（枢纽人不在场时另一个人也能继续做）、新人优先降低单点依赖（不是优先产出新功能）
- **迭代节奏：** 不要固定 Sprint 长度。按"周"推进比按"Sprint"更自然。里程碑型的硬节点（"MCU 端到端跑通"）才是真正的节奏驱动。

### 网状 + 低单点依赖（理想成熟团队）

每个方向都有人能独立推进，角色之间可以直接协调。

- **可以做：** 多角色并行、正式的 Sprint 节奏、燃尽图追踪
- **仍然必须做：** 接口约定（成熟团队也会假设不一致）、阻塞升级（成熟团队也会藏问题）、难点清单（成熟团队也会遇到计划外的困难）

> **关键结论：** 不管什么配置，nano-flow 的四件事（方向 + 接口 + 阻塞 + 难点）是底盘。团队越成熟，可以在这个底盘上加更多工具。团队越不成熟、单点依赖越高，越应该只用这四件事。

### 兼职成员多的配置

兼职成员的本质不是"人不够"——是**信息同步的频率天然比全职低**。

- **不要做：** 让兼职成员负责和其他角色的接口（他们的信息滞后会导致接口假设漂移）
- **必须做：** 兼职成员的输入和输出都经过全职枢纽人物中转、接口约定尤其重要（兼职成员一周后才看到变更，如果没人通知就按旧约定做了）
- **沟通规则：** 涉及兼职成员的接口变更，必须主动推送（飞书私聊 / @消息），不能放在群公告里等他们自己看

## 1.4 枢纽保护原则

如果团队有一个星形枢纽（像刘豪），保护这个人就是保护项目。

| 原则 | 做法 |
|------|------|
| 枢纽不做琐事 | 环境搭建、文档格式化、数据整理——任何不需要枢纽判断力的事，让别人做 |
| 枢纽的连续时间是不可侵犯的 | 每天至少 3 小时连续编程/设计时间。所有需要枢纽"看一眼""确认一下"的事，集中到下午一个时段 |
| 枢纽不做唯一执行者 | 每一个模块必须有一个"备份人"——不要求备份人能做到枢纽的水平，但能在枢纽请假时不让项目停滞 |

# 第二章：方向 —— 一句话让所有人判断标准相同

## 2.1 方向不是愿景，是判断标准

愿景是"我们要做超低功耗 AI ECG"。方向是"当超低功耗和更高精度冲突时，我们选哪个"。

一个有效的方向表述，必须让团队在面对任何"做 A 还是做 B"的选择时，不需要请示管理层就能自己判断。

## 2.2 方向一句话的写法

格式固定：

> 面向 **[谁]**，我们做 **[什么]**，让他们能 **[做什么以前做不了的事]**。区别于 **[竞品/替代方案]** 的关键是 **[我们的独特优势]**。

nanoengine-ecg 的版本：

> 面向可穿戴设备固件集成工程师，我们提供不到 $5 的通用 MCU AI ECG 推理引擎 SDK，让他们在 STM32F429 上实现 12 导联心电 AI 实时分析——推理 < 2ms、内存 < 35KB、占空比 < 0.5%、200mAh 续航 > 7 天。区别于竞品专用芯片方案的关键是：通用硬件 + 极低资源 + 超低功耗。

注意这句话里有数字（$5、2ms、35KB、0.5%、7 天）。数字不是用来汇报的——是当两个需求冲突时，"用哪个数字做判断标准"。

比如：有人提出"要不要支持更大的模型来提升精度"——看方向一句话：内存 < 35KB。如果更大的模型会超 35KB，答案就是不做。不需要开会。

## 2.3 方向校准：红绿笔法（20 分钟）

这是方向对齐的唯一有效方法。不是开会讨论，是**每个人亲手改**。

1. PM 把方向一句话写在一个 md 文件里（5 分钟）
2. 每个人用 ~~删除线~~ 划掉不同意的词句，用 **加粗** 写替代方案（10 分钟）
3. PM 合并所有人修改（5 分钟）
4. 如果还有人不同意 → 重复步骤 2

**为什么不开会讨论？** 因为口头讨论中，资历浅的人不好意思反对资历深的人。笔迹不会撒谎。当你亲手划掉一个字，你是在为自己不同意的东西负责。

## 2.4 取舍填空

方向最核心的本质是**舍弃**。一个没有舍弃的方向是愿望清单——在资源冲突时没有判断力。

每人填空（至少填两行）：

> "我们选择做 ____，即使这意味着不做 ____。"

填不出来 → 说明方向还不够清晰 → 回到红绿笔。

## 2.5 三条红线

从发起人那里拿到三条不可越过的边界。每条格式固定：

> "关于 ____，我们不能越过 ____。"

例如：
- 关于时间，MCU 阶段不得晚于 2026/09/30
- 关于质量，C 输出 vs PyTorch 参考 = ZERO ERROR
- 关于范围，MCU 阶段不碰芯片后端

**不超过三条。** 超过三条等于没有红线。什么都重要等于什么都不重要。

**发起人必须签字。** 没有签字的红线在项目中期遇到压力时会被遗忘。

# 第三章：接口 —— 最高杠杆的投资

## 3.1 项目中最高频、最昂贵的返工类型

A 在写代码，B 也在写代码。A 的产出是 B 的输入。两人各自做了自己的假设，两周后一对接——对不上。返工 3-5 天。

这不是能力问题。这是**没有人在开工前把假设对齐**。

## 3.2 接口约定：10 分钟节省 2-5 天

任何两个人的工作有交接面时，花 10 分钟回答 5 个问题，写下来，双方确认：

1. **上游是谁？下游是谁？**
2. **交付什么？**（具体到文件名、函数签名、struct 定义、数据格式）
3. **交付物的格式/规范是什么？**（能多具体就多具体——不是"一个数组"，是 `int8_t output[12][125]`）
4. **交付时间？**
5. **下游怎么验证交付物没问题？**（比如：用上游给的测试向量跑一遍 + 输出匹配）

双方确认签字。任何一方要改接口 → 更新约定 + 通知对方 + 重新确认。

## 3.3 什么时候必须写接口约定

以下任一条件满足就必须写：

- 两个人的工作有**时间先后**（A 做完 B 才能开始）
- 两个人的工作**可以并行但需要接合**（A 的输出格式决定 B 怎么读）
- 其中一个人是**兼职**（信息同步天然滞后，书面约定是唯一防漂移的手段）
- 其中一个人是**新人**（新人对"默认应该怎么做"没有经验，需要明确的规范）

## 3.4 接口约定的防漂移机制

接口变了但对方不知道——这是最隐蔽的返工来源。

**规则：** 任何接口变更，上游必须**主动推送**给下游——飞书私聊或 @消息。不能放在群公告里等对方自己看。下游确认收到后，更新接口约定文档。

如果下游是兼职 → 额外规则：变更通知后 24 小时内未确认 → 电话或当面确认。

## 3.5 接口和文档的区别

文档是对世界的描述。接口约定是两个具体的人之间的承诺。

- 文档可以过期（而且常常过期）。接口约定不过期——因为在有人违约的那一刻，下游就会在群里 @上游。
- 文档是写给"将来可能看的人"的。接口约定是写给"下周就要对接的那个人"的。

**不要写大文档。写接口约定。**

# 第四章：问题 —— 即时抛出、即时响应、2 天没排查策略就升级

## 4.1 为什么取消站会

站会的原始设计意图是好的——每天快速同步，不让阻塞过夜。但在实践中它退化了：

- **信息衰减：** 下午 3 点遇到的问题，存到第二天早上 9:15 才说——过了一晚上，细节已经忘了大半
- **汇报化：** 变成了"轮流向 PM 汇报"，不是"团队一起发现问题"
- **延迟响应：** 一个本可以下午 3:30 解决的事，因为"等明天站会再说"，白白丢了 18 个小时

**站会的本质需求不是"同步"，是"不要让问题藏起来"。** 这个需求不需要一个固定会议来满足。

## 4.2 替代方案：群聊 @ + 2 天排查策略时钟

遇到问题 → **立刻在群里 @人**，说清楚三件事：
1. 卡在哪（具体到可复现）
2. 需要谁
3. 什么时候要

被 @ 的人**当天必须响应**。不一定是当天解决，但当天必须说："我知道了，预计 X 时间处理" 或 "我想到一个排查方向：先试 A，如果不行试 B"。

**"有排查策略但需要时间"是正常的——不进列表。**
**"2 天了连从哪入手都不知道"才是需要升级的——进列表。**

这个区分的意义：前一种情况是技术攻坚的正常节奏。后一种情况意味着问题超出了当前响应人的能力或权限——他卡住了，但不知道找谁能帮自己，或者不好意思再 @别人。这就是需要项目经理介入的信号。

## 4.3 问题升级时钟

| 时间线 | 状态 | 动作 |
|--------|------|------|
| @了之后 24h 内 | 对方已响应，讨论出排查方向 | 正常推进。不进列表。 |
| @了之后 24h | 对方未响应 | 催刘豪。刘豪去问"看到 @了吗？需要帮忙吗？" |
| @了之后 2 个工作日 | 有排查策略，排查中 | 正常推进。不进列表。 |
| @了之后 2 个工作日 | **仍然没有排查策略**——连"从哪入手"都说不清 | ⚠ 记入 02-问题列表。问题超出当前响应人的能力/权限范围。 |

**"2 天没排查策略"和"2 天没解决"是两回事。** 后者是正常的技术工作——找根因、设计修复方案、验证修复，加起来可能要一周。前者是信号：当前负责的人搞不定了，但还没人帮他。

## 4.4 问题列表的用法

见 `02-问题列表.md`。核心要素：

| 字段 | 为什么重要 |
|------|----------|
| 问题描述（具体到可复现） | "MCU 延迟高"不是问题描述。"Conv1d 算子在 M4 上实测延迟 8ms，CPU 基线 1ms，8 倍差距"才是 |
| 当前有没有排查策略？ | **这是进列表的核心判断字段。**"没有"意味着需要周会讨论 |
| 影响范围 | 谁被卡住了？什么事做不了？用来判断优先级 |

## 4.5 问题文化

团队能不能顺畅地说出"我卡住了，而且我不知道怎么排查"，是项目健康度的第一指标。

建立这种文化的方法：**项目经理先示范。** 当刘豪在群里 @人 说"我在 XX 上卡了 2 小时，试了 A 和 B 都不行，有人遇到过类似问题吗？"——这是在示范：卡住是正常的、说出来是安全的、求助是高效的。

# 第五章：难点 —— 计划的敌人不是变化，是你的假设

## 5.1 两种难点

| | 计划书上的难点 | 真正的难点 |
|---|-------------|----------|
| 来源 | 立项时分析出来的 | 做了才发现的 |
| 特征 | 抽象、笼统（"MCU 性能可能不达标"） | 具体、意外（"M4 的 DSP 指令在某内存布局下反而比标量慢"） |
| 谁来解决 | 已经分配了责任人 | 可能需要跨角色讨论 |
| 危险程度 | 低——你已经看见了 | 高——你之前没看见，而且它可能卡住不止一个人 |

搞死项目的从来不是第一种。是第二种。

## 5.2 难点和问题列表的关系

**难点 ≠ 问题列表中的问题。**

- **问题列表中的问题（第四章）：** 在群里 @了人、2 天没排查策略、需要升级到周会决策。关注的是"谁卡住了、怎么解决、谁来做"——是**执行层面的阻塞**。
- **难点（本章）：** 做了才发现的、之前没预见到的技术或协作障碍。关注的是"我们之前为什么没预见到、下次怎么避免"——是**认知层面的教训**。

一个问题进了列表、解决了之后，如果它揭示了一个之前没预见到的系统性困难，它就是"难点"。如果只是日常的协调问题（"张三没回消息""接口文档少写了一个字段"），它就不算难点。

## 5.3 难点的识别标准

一个被解决的问题同时是"难点"，需要满足：
1. **你之前没预见到它会这么难** — 如果你在项目计划里已经写了"MCU 延迟可能超标"，那它超标了是你验证了一个已知风险，不是难点
2. **它卡住了不止一个人** — 如果只影响一个人且那个人自己解决了，不叫难点
3. **解决它的过程中学到了一个可复用的经验** — "以后所有类似场景都应该先做 X 再开始 Y"

## 5.4 难点的记录方式

不单独建文件。在 `02-问题列表.md` 的"已解决"表中，教训一栏如果满足上述三个标准，用 **加粗** 标注。周会时顺便看一眼有没有新的难点该被提炼出来。

> "M4 的 arm_convolve_1x1 在某些内存布局下性能退化 3 倍。**教训：所有算子在 M4 上实测之前，不假设 DSP 扩展一定有效。**"

# 第六章：阶段与节奏 —— 什么时候该看什么

RST 定义了六个阶段：启动 → 规划 → 执行 → 监控 → 收尾 → 维护。每个阶段承担不同的功能，跳过或压缩会付出代价。

这本手册不去掉这个结构——结构本身是有价值的。但把每个阶段的内容从"方法论规定的仪式"替换为"在当前团队配置下取得最快最稳结果的最小动作"。

## 6.1 启动阶段：花 1 天，省 3 个月

**目标：** 让每个人脑子里对"我们要做什么、为什么做、做好的标准是什么"有相同的心理画面。

**最小动作：**

| 动作 | 多久 | 产出 | 谁参与 |
|------|------|------|--------|
| 方向红绿笔 + 取舍填空 | 30 分钟 | 方向一句话 + 取舍记录 | 全部专职成员 |
| 三同心圆干系人地图（找出隐性关键人物 + 沉默否决者） | 30 分钟 | 干系人清单 + 影响力评分 | 项目经理 + 发起人 |
| 三条红线 + 发起人签字 | 30 分钟 | 签字的红线文档 | 项目经理 + 发起人 |
| 恐惧清单（匿名便利贴） | 20 分钟 | Top 5 恐惧（拍照） | 全部成员 |
| 团队协议（沟通/决策/冲突/反馈 四个约定） | 30 分钟 | A4 纸协议，每人签字 | 全部成员 |

**总投入：** 约 2 小时 20 分钟（分两段，半天内完成）。
**跳过代价：** 三个月后你会听到"我以为项目是要做 X""为什么没人告诉我 Y 也要参与"。启动省一天，后期用十天补。

**缩减版（团队 < 3 人或紧急启动）：** 至少做方向红绿笔 + 三条红线。其余可以跑起来再补。

## 6.2 规划阶段：不规划完，规划框架

**目标：** 不是产出一份无法改变的详细计划，而是建立一套"当变化发生时团队还能做出正确判断"的框架。

**规划阶段最大的陷阱：** 花太多时间在"预测未来"上。你预测的未来大概率是错的——需求会变、技术会卡、人会有状况。与其预测，不如设计一个能快速响应变化的框架。

**最小动作：**

| 动作 | 多久 | 产出 |
|------|------|------|
| 角色依赖图（谁依赖谁的产出？） | 20 分钟 | 依赖关系图 → 驱动接口约定清单 |
| 风险预映射（只画传导路径，不列风险清单） | 30 分钟 | 2-3 条关键传导路径 + 每个最上游风险的早期信号 |
| 里程碑设定（不是 Sprint 计划，是"什么时候必须看到什么"） | 20 分钟 | 3-5 个硬节点 + 每个节点的验收标准 |

**不做的：**

- 不画用户故事地图（MCU 阶段需求清晰，用户只有一种角色，不需要）
- 不排 Sprint Roadmap 超过 3 周（排了也会变）
- 不做团队能量五维投票（不如直接看群里有没有人 @了没被响应）

**什么时候需要完整规划（扩展到 rst-toolkit）：**

- 用户角色 ≥ 2 种 → 用户故事地图
- 团队 ≥ 7 人 → 正式 Sprint Roadmap
- 迭代 ≥ 6 个 → 燃尽图追踪
- 有外部汇报需求 → 四象限监控面板

## 6.3 执行阶段：问题即时 @、每周一次会

**目标：** 不是"按计划推进"，是"问题不藏、接口不漂、每周清零"。

**日常节奏：**

```
有问题的任何时刻
    │
    ▼
群里 @人 ──→ 被 @ 的人当天响应
    │              │
    │              ├─ 讨论出排查策略 → 按策略推进（不进列表）
    │              │
    │              └─ 2 天了连排查方向都没有 → 记入问题列表
    │
    └─ 解决了 → 记一条教训（如果值得）

每周五 15:30-16:30 周会
    ├─ 打开问题列表，逐条过
    │   每个问题：解 / 搁置 / 调目标
    ├─ 接口约定检查（有违约吗？有变更没通知的吗？）
    └─ 每人说一件"下周最担心卡住的事"
```

**没有每日站会。没有 Sprint。** MCU 阶段不到 3 个月，按周推进。周会是唯一的固定会议——它的唯一目标是**把问题列表清到零**。

## 6.4 监控阶段：不看图，看三个信号

**目标：** 在问题从"可轻松修复"变成"不可逆"之前发现它。

**传统监控的问题：** 燃尽图、累积流图、SPI/CPI——这些图表维护成本高、解读需要经验、容易被美化。对 5 人团队来说，维护这些图表的时间不如用来直接看代码和聊天记录。

**三个真实信号（每周五周会时过一遍）：**

| 信号 | 怎么看 | 异常阈值 |
|------|--------|---------|
| **问题列表活跃数** | 问题列表里活跃问题在涨还是跌？ | 连续 2 周上涨 → 项目在变复杂，注意枢纽人物精力 |
| **"没有排查策略"问题的比例** | 活跃问题中，多少是"连从哪入手都不知道"的？ | > 50% → 团队在某些领域缺乏对应的能力或资源 |
| **接口违约次数** | "这周有接口到了交付时间但没交付吗？" | 连续 2 周有违约 → 估算太乐观或接口太多 |

这三个信号不需要任何图表。每周五花 5 分钟口头过一遍。

**什么时候引入数据监控：**
- 团队 > 7 人 → 项目经理肉眼扫不过来
- 管理层要求正式汇报 → 四象限监控面板

## 6.5 收尾阶段：三张 A4 纸 + 一周缓冲

**目标：** 让下个项目从"之前犯过的错误"中受益，而不是重复它们。

**最小动作：**

**1. 方向验证（1 小时）：**
邀请真实用户来试用，不是团队内部验收。问三个问题：
- "这个功能解决了你最开始时最痛苦的问题吗？"
- "如果满分 10 分，你现在打几分？"
- "差的那几分是什么？"

如果用户说"还行""挺好的"然后停顿——那不是满意，是礼貌。

**2. 三张 A4 纸归档（2 小时）：**
- **决策日志（5-10 个关键决策）：** 选项 A vs B → 选了哪个 → 为什么（真实约束和妥协，不要美化）
- **偏差记录（2-3 个方向偏离事件）：** 什么时候发现 → 怎么发现 → 纠正多久 → 重来会在哪里更早干预
- **团队工作模式：** 最佳节奏、并行/串行规律、特别提示（一句话）

**3. 团队能量恢复（1 周）：**
不是放假——是做三件事：
- **技术债清偿（3 天）：** 核心路径上的 TODO 清理干净
- **学习缓冲（2 天）：** 每个人学一件和项目相关但当时没时间深入的事
- **团队散伙饭（半个晚上）：** 两个问题——你最骄傲的瞬间？你接下来想做什么？

## 6.6 维护阶段：从冲刺切到长跑

**目标：** 不要用做项目的节奏做维护——那是"敏捷疲劳"最常见的来源。

**切换标志：** 一个可观测的事件，比如"SDK 交付给首个厂商满 30 天且无 P0 缺陷"。

**节奏调整：**
- 主动 @ 响应 → 不变（仍然是即时 @，当天响应）
- 周会 → 月度检查（30 分钟，看用户反馈关键词、MTTR 趋势、加班情况）
- Sprint → 不存在了。改为 Kanban 连续流

**度量切换：**

| 项目期你盯什么 | 维护期你盯什么 |
|-------------|-------------|
| 做完多少功能 | 用户有没有在骂我们（反馈关键词趋势） |
| 进度快还是慢 | Bug 修复有没有变慢（MTTR 趋势） |
| 计划完成率 | 团队有没有在加班（维护期加班比项目期更危险） |

# 第七章：不同团队配置的实战变体

以下四张表覆盖最常见的四种团队配置。找到最接近你的那一张，直接套用。

## 7.1 星形枢纽 + 高单点依赖 + 有新人（nanoengine 当前）

| 维度 | 做法 |
|------|------|
| 节奏 | 按周推进，不设 Sprint。里程碑硬节点驱动。 |
| 站会 | 每天 15 分钟。从阻塞开始，不从进度开始。 |
| 规划 | 只规划未来 2-3 周。远期只保留方向性描述。 |
| 接口 | 两个人在开工前必须先写接口约定。所有人必须写。 |
| 阻塞 | 24/48h 时钟升级。枢纽人阻塞 = 全员阻塞，最高优先级解除。 |
| 新手保护 | 新手前 2 周不排产出任务。学习期 + 导师制。新手不独立负责跨角色接口。 |
| 兼职成员 | 兼职成员的输入输出经过枢纽人中转。接口变更必须主动推送，不能等他们自己看。 |
| 周回顾 | 30 分钟。只看：难点新增/解决？接口违约？下周最可能卡住的事？ |
| 不做的 | 燃尽图、Sprint Planning/Review/Retro、能量投票、用户故事地图 |

## 7.2 网状 + 低单点依赖（理想成熟团队）

| 维度 | 做法 |
|------|------|
| 节奏 | 2 周 Sprint（成熟团队对方向的判断更准确，2 周是效率/反馈的最优平衡点） |
| 站会 | 每天 15 分钟。从右往左看板，只讨论卡住的卡。 |
| 规划 | Sprint Planning 70 分钟。可引入 T 恤尺码估不确定度。 |
| 接口 | 仍然必须写接口约定。成熟团队的接口假设也会不一致——而且因为双方都很自信，可能更晚发现。 |
| 监控 | 可以引入燃尽图 + 累积流图。注意：燃尽图只用来诊断，不用来催进度。 |
| 周回顾 | Sprint Retro 45 分钟。标准三问。只选一个改进项。 |

## 7.3 纯新人团队（所有人都在学习期）

这是最高风险配置。不要用正常团队的节奏去跑。

| 维度 | 做法 |
|------|------|
| 节奏 | 不要 Sprint。每周一个明确的最小目标（"这周每个人都独立跑通一个算子"）。 |
| 学习期 | 前 2 周不排产出任务。第 3-4 周排 50% 学习 + 50% 产出。 |
| 导师 | 必须有至少一个能回答技术问题的人。如果没有 → 先招这个人，再让新人入职。 |
| 接口 | 新人之间不要设定接口依赖。每个人都做独立的任务，由导师负责集成。 |
| 阻塞 | 新人遇到阻塞 1 小时内必须求助——不等站会。鼓励"过度求助"，惩罚"闷头硬搞"。 |

## 7.4 分布式团队（异地/远程）

| 维度 | 做法 |
|------|------|
| 站会 | 视频站会，开着摄像头。共享屏幕看同一份每日同步记录。 |
| 接口约定 | **必须写。** 分布式团队没有"走过去问一句"的奢侈。所有接口约定必须书面。 |
| 异步沟通 | 核心时间段（比如北京时间 14:00-17:00）所有人必须在线可响应。其他时间异步。 |
| 文档 | md 文件 + git 仓库。不用飞书文档（无法离线看、无法 grep）。所有 md 文件在 git 仓库里，每个人本地有一份。 |
| 社交 | 每月一次线下碰面。如果做不到 → 每周五加 15 分钟非正式视频聊天（不谈工作）。 |

# 附录 A：模板速查

以下模板全部以独立 md 文件形式存放在 `nano-flow/` 文件夹：

| 模板 | 文件 | 什么时候用 |
|------|------|----------|
| 项目状态（唯一真相源） | `00-项目状态.md` | 每天看一次。方向变了随时更新 |
| 接口约定（5 个问题） | `01-接口约定.md` | 任意两人要对接工作前 |
| 问题列表（真问题追踪） | `02-问题列表.md` | @人 2 天后还没排查策略时 |
| 周会记录（清零问题） | `03-周会记录.md` | 每周五下午 |

## A.1 方向一句话模板

```
面向[谁]，我们做[什么]，让他们能[做什么以前做不了的事]。
区别于[竞品/替代方案]的关键是[我们的独特优势]。
```

## A.2 接口约定模板

```
编号：IF-[编号]
上游（产出方）：[谁]
下游（使用方）：[谁]
交付什么：[具体文件/数据/格式]
格式/规范：[能多具体就多具体]
交付时间：[日期]
验证方式：[下游怎么确认交付物正确]
如果延期：[下游可以先做什么 / 谁被通知]
上游确认：[签名/日期]
下游确认：[签名/日期]
```

## A.3 问题进入列表的判断标准

```
不是"问题解决了吗"——是"连排查策略都还没有吗"。

有排查策略、正在排查中 → 正常推进，不进列表。
@了人、2 天过去了、连"从哪入手"都说不清 → 进列表。
```

## A.4 周会流程

```
1. 打开问题列表，逐条过。每个问题：解 / 搁置 / 调目标
2. 接口约定检查（3分钟）：有违约吗？有变更没通知的吗？
3. 每人说一件"下周最担心卡住的事"
4. 每月最后一次加：方向检查
```

## A.5 三张 A4 归档模板

**决策日志（每条三行）：**
```
选项 A vs 选项 B → 选了哪个 → 为什么（真实约束和妥协）
```

**偏差记录（每条四句）：**
```
什么时候发现偏航 → 怎么发现 → 纠正多久 → 重来会在哪里更早干预
```

**团队模式（一段话 + 一句特别提示）：**
```
最佳节奏 / 并行串行规律 / 决策习惯 / 
"下一个和这个团队合作的人最应该知道的一件事：____"
```

# 附录 B：症状 → 诊断 → 行动速查

| 你看到的现象 | 可能的原因 | 第一行动 |
|------------|----------|---------|
| 站会每人轮流说"还在做"，没有阻塞报告 | 心理不安全感——不敢说卡住了 | PM 先说自己卡在哪里，示范脆弱性 |
| 同一张卡连续 3 天在"进行中"不移动 | 存在未报出的阻塞 | 私下问负责人"有什么需要我帮忙的吗"，不问"为什么还没做完" |
| 集成时发现模块对不上 | 接口假设不一致 | 立即要求所有跨角色对接都写接口约定 |
| 周报全绿但直觉告诉你有问题 | 信息失真积累——坏消息被过滤了 | 不看周报。随机坐到一个开发者旁边，让他把当前版本跑给你看 |
| 兼职成员的产出经常需要大幅返工 | 信息同步滞后——兼职成员按旧约定做事 | 接口变更必须主动推送，不在群公告里等他们自己看 |
| 核心开发者离职后项目停摆 | 单点依赖，知识没备份 | 立即做知识地图——每个模块标出"只有一个人知道全貌"的红点 |
| 团队越来越累但产出没增加 | 在制品太多，上下文切换消耗了能量 | 限制 WIP——每人同一时间只做一件事 |
| 难点清单太长，几个月没清理 | 要么在系统性回避难点，要么清单里塞了不该塞的东西 | 只保留"不解决无法推进"的 🔴 项。其余归档 |

# 附录 C：什么时候引入更多工具

nano-flow 的 4 个文件是底盘。以下信号告诉你什么时候该从 rst-toolkit 引入更多工具：

| 信号 | 引入什么 |
|------|---------|
| 团队超过 7 人，站会上信息量太大 | 引入燃尽图 + 累积流图（视觉化替代纯文字） |
| 有正式的月度管理层汇报需求 | 引入四象限监控面板（SPI/CPI + 信息准确度/团队能量） |
| 用户角色 ≥ 2 种，需求依赖复杂 | 引入用户故事地图（横轴用户旅程 + 纵轴优先级） |
| 项目进入维护期，从冲刺切长跑 | 引入维护期月检记录（用户反馈关键词 + MTTR + 加班趋势） |
| 需要预测交付日期（给管理层承诺） | 引入吞吐量统计 + Cycle Time 分布 |
| 团队能量明显下降但不知道为什么 | 引入团队能量五维匿名投票 |

**原则：** 只在现有工具不够用时才加新的。每加一个新工具，问自己："这个工具能让团队的决策更快还是更慢？如果更慢，不加。"
