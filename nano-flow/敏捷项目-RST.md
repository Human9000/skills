# 敏捷项目实战手册——从方向校准到持续交付的完整实操指南

**刘豪**

---

# 导言

## 这本手册解决什么问题

假设这样一个场景：你所在的组织做敏捷开发已经一两年，站会每天都在开，看板用起来了，迭代也在一轮一轮地跑。但你越来越觉得哪里不对劲——团队很忙，但"忙不出活"；周报每次都很漂亮，计划完成率接近100%，但产品上线后用户根本不用。你开始怀疑：是不是敏捷本身有问题？

答案是否定的。问题不在敏捷，在于你把敏捷当成了一套"照着做的流程"，而不是一套需要理解背后逻辑的思维框架。你只知道"站会要问三个问题"，但不知道为什么问这三个问题——所以当标准流程不适用时（比如团队分布在不同时区、需求变化比迭代还快、干系人根本不看你的燃尽图），你就失焦了。你不知道哪些东西可以调整、哪些东西调整了就会失效。

这是目前敏捷实践中最普遍的问题：照做型敏捷。团队在执行"敏捷仪式"但没有敏捷思维。站会在开，评审在走，但信息失真依然在积累，团队能量依然在透支，方向依然在悄悄漂移——只是没有人用对的方式去看见这些。

## 根本原因

照做型敏捷的根源是：团队只学到了"怎么做"，没有学到"为什么"。知道Sprint是两周一个周期，但不知道为什么要固定时间盒——所以当项目压力大的时候，第一个被砍掉的就是Sprint Review。知道Daily Scrum要问"昨天做了什么今天做什么"，但不知道为什么问——所以站会变成了轮流汇报，信息失真的积累并没有被阻止。

在非标准情况下的判断力，恰恰来自对"为什么"的理解。当你知道Sprint的核心目的是"强制反馈节奏"而非"每两周交付一次"，你就会知道即使项目很忙，Review也不能砍——因为砍掉反馈回路，团队就会在封闭中继续做假设，风险比延迟交付高得多。

这本手册用一套核心洞察来解释敏捷实践背后的原理。它不是另一套管理方法论，它是一套描述"项目实际在发生什么"的语言。用这套语言，你可以看见甘特图上看不见的东西——信息在哪里失真、能量在哪里消耗、方向在哪里漂移。

## 怎么使用

这本手册有两种使用方式：

**第一遍通读（约3小时）。** 从导言读到附录，理解完整的逻辑链条——为什么敏捷要划分六个阶段、每个阶段解决什么问题、这些核心洞察如何解释这些做法的有效性。通读的目标不是记住每一个步骤，而是建立"当项目出现问题时，我知道应该往哪个方向找答案"的判断框架。

**日常查阅。** 项目进行到哪个阶段，就翻到对应章节，按步骤执行。每个步骤都有具体的操作方法，不需要在执行时再去理解背后的理论。如果你在执行中遇到了手册步骤和你的实际情况不符的情况，回到"为什么"段落，理解这个步骤背后的原理，然后根据原理自己判断如何调整。

## 一句话概括

这是一本以敏捷开发完整生命周期为主线、用一套日常语言解释每一步深层目的和意义的实操手册。解决的不是"敏捷该怎么做"，而是"敏捷为什么这样设计，以及当标准做法不适用时你该怎么判断"。

# 第零章：六个核心洞察——看懂项目真正在发生什么

本章用六段话介绍六个核心洞察。每个洞察直接用项目管理场景举例解释。理解这六个洞察，你就有了分析任何项目问题的基础语言。

在后续章节中，这六个洞察会反复出现在"为什么"段落里。如果你在读后续章节时对一个洞察感到模糊，可以回到本章重新理解。

---

### 洞察一：方向与取舍——不在PPT上，在资源冲突时你实际选择了什么

每张PPT上的项目愿景写得都很漂亮，墙上贴的使命宣言也很动人。但那些都不算数。项目真正的方向只有一件事能定义：当资源发生冲突时，你实际选择了什么。

选择"多做功能"而不是"优化用户体验"，你的项目方向就是"多做功能"。选择"按时交付"而不是"保证质量"，你的项目方向就是"按时交付"。这些选择暴露出来的，才是你真正的方向。PPT上写的只是愿望。

问题在于：你贴在墙上的方向和团队实际决策中暴露出来的方向是不是同一个？如果不一样，团队就会出现"每个人都觉得自己在往项目目标努力，但每个人努力的方向互相抵消"的现象。方向清晰的标志不是"团队能背出项目愿景"，而是"当两个需求冲突时，不用请示管理层，团队能自己判断应该优先做哪一个"。

**项目管理场景举例：** PO在Sprint Review上说"我们的目标是提升用户体验"，但下一个Sprint中，一个用户体验优化和一个新功能开发撞期了——PO选择了新功能。项目真正的方向不是PPT里写的"提升用户体验"，是PO在资源冲突时选择了"多做功能"。

---

### 洞察二：信息失真——每传递一层就衰减一层

信息每经过一个层级传递一次，就会发生一次失真——不是有人在故意撒谎，而是每个人在传递信息时都会无意识地加上自己的理解、省略自己认为不重要的细节、强调自己关注的部分。经过三四层传递之后，原始信号和最终信号可能已经是两回事。

你作为项目经理感觉到的那种"周报全绿但直觉告诉我项目偏了"的直觉，就是信息失真积累到一定程度后发出的信号。直觉不是玄学，是你的潜意识捕捉到了正式汇报渠道之外的真实信号——某个开发在站会上说了"还在做"但语气多了一点犹豫，某个干系人连续两次没有回复邮件。

信息失真的危险在于：它不会自己暴露。失真越严重的团队，汇报越漂亮、表面上越没有问题。

**项目管理场景举例：** 开发对PM说"功能开发完成了，正在测试"——这是一个被压缩了三次的描述。原始信号是："代码写完了但还有三个单元测试没过，集成环境部署时遇到了问题，不过我觉得应该今天能搞定。"经过三层压缩变成了"正在测试"——这是一个从"确实还有问题但我在处理"变成"一切正常"的失真。PM看到的是"一切正常"，真实世界是"还有问题需要处理"。这就是报喜不报忧的源头。

### 洞察三：团队能量——加班是在预支明天的精力

大多数项目经理只算"人天"——团队有5个人、每人每天8小时、两周迭代一共80人天。但实际产出从来不是80人天。为什么？因为只算了时间，没算能量。

团队的能量不只是时间，它包括：注意力（一个人同时做几个项目时注意力是分裂的）、信任（两个从没合作过的团队在接口处天然需要更多时间建立默契）、学习空间（新技术的学习成本不是零）、犯错勇气（一个害怕犯错的团队不会主动承担有风险的工作）。加班不是在增加产出，是在预支明天的精力来填补今天的缺口。

当团队能量被消耗到临界线以下，人还在座位上，但已经停止了有效思考——这是最隐蔽的产出消失。成员会出现在所有会议上，做完了所有汇报，但真实产出趋近于零。

**项目管理场景举例：** 一个迭代安排得满满当当，看起来团队刚好能完成。但团队成员有三个同时在处理一个线上事故的排查、有两个在准备下周的架构评审、有一个在带新人——这些都没有出现在迭代计划里。迭代结束时进度绩效指数只有0.7，团队很沮丧。不是能力问题，是团队能量在没有人注意的情况下被透支了。

### 洞察四：恢复和学习能力——核心人走了多久能恢复？同一个错误会犯第二次吗？

任何一个项目都会遇到冲击——核心人离职、需求大变、技术方案被推翻。团队之间在这时候的分水岭不在于是否会遇到冲击（都会遇到），而在于遇到冲击之后：有些团队能快速找到新的协作方式，从中学习，下次遇到类似冲击时恢复得更快。有些团队遇到冲击之后，第一次需要两周恢复、第二次需要三周、第三次就散掉了——因为每次冲击只增加了恐惧，没有增加韧性。

恢复能力高的团队有几个共同特征：知识不是藏在某一个人的脑子里——代码做了Review，关键决策有日志记录，新人能通过文档而不是靠问人了解项目；有自动化的测试和部署流程——不会因为某个人请假就没人知道怎么发版；团队愿意承认错误并改进，而不是把错误藏起来。

恢复能力低的团队也有一个共同特征：所有关键环节都是单点依赖——"只有张三知道怎么做"，"只有李四能部署"。这部分人一旦离开，团队直接停摆。

**项目管理场景举例：** 前端核心开发者突然请假两周。恢复能力高的团队：后端开发者之前参与过前端代码的Review，可以提合并请求；测试团队有自动化的端到端测试覆盖关键路径；团队有前一个迭代的决策日志可以查。恢复能力低的团队：没有人知道前端项目的构建流程，测试是纯手动的，有一个关键决策"藏在离职员工的脑子里"——两周后项目基本停滞。

### 洞察五：真正影响项目的人——不是title最高的，是大家做决定前会去问谁

每个项目中都有这样一个人：他不一定是职位最高的——他可能是团队中最资深的开发，也可能是客服部最能影响产品方向的经理，甚至是一个不在项目组织架构图上的专家。但所有人在做重大决定之前，都会先问一句他的意见。这个人一句话能改变项目的走向，但他的影响力不写在任何正式文档中。

这类关键人物中有正向的——能凝聚团队、帮助决策；也有负向的——能通过消极方式让项目停滞。而且关键人物的坐标会变化——一个被冷落的关键人物可能随着时间从支持者变成阻力。

项目管理中最大的风险暴露之一就是：你花了大量时间管理组织架构图上的人，但没有花时间管理真正能影响项目走向的人。

**项目管理场景举例：** 项目经理每周和CTO对齐一次，项目状态报告每两周向VP汇报一次。但团队在做一个关键的架构决策时，所有人不约而同地先打电话问一个两周前刚从团队转去数据部门的资深工程师——他才是这个决策的真正关键人物。CTO不是。VP不是。是那个电话里的人。

### 洞察六：最小的真实行动——不确定的情况用最小的真实行动来验证，而不是开三轮讨论会

团队在面对高度不确定的情况时有两个常见选择：要么停下来开更多的讨论会和分析会，直到"考虑清楚了"再做；要么直接蒙一个方向开始干。两个选择都很糟糕——前者浪费大量时间在"分析瘫痪"中，后者浪费大量时间在"做错了要返工"中。

正确做法是走一条中间路径：用最小的真实行动来验证假设，而不是用讨论来模拟决策。做一个快速原型、找一个用户聊半小时、用一个周末做一个可以点按的模拟界面。每个这种小实验产生的真实反馈，比十次会议上的"我觉得""我认为"更有价值。

**项目管理场景举例：** 团队不确定新设计用户能不能接受。选项A：开三次评审会，画更多的UI稿，让大家投票。选项B：花两天做一个可点击的原型，找五个目标用户各测15分钟。选项B就是正确做法——成本更低、反馈更真实、即使失败也只损失了两天。选项A看起来"安全"但消耗了更多时间，而且输出的只是"会议室里的假设"，不是"真实世界的反馈"。

- **跨行业适配：** NDM的"最小"在不同行业定义不同——软件=几天原型；硬件=仿真验证/3D打印样机；建筑=BIM模拟；医药=细胞实验到动物实验的渐进验证。共同原则：用行业允许的最小成本获得真实反馈。

### 第零章之1.5：标准的敏捷项目框架是什么样的——Scrum、Kanban与敏捷宣言

在开始动手做项目之前，你需要先知道一件事：敏捷不是一个"流程"。你没法像安装软件一样装一套敏捷到团队里，然后它就自动运转了。敏捷是一套三层结构——原则、框架、实践。

最底层是**敏捷原则**（敏捷宣言的四个价值观和十二条原则）。原则是不变的——它们告诉你"好的项目管理为什么这样做"。中间层是**框架**（Scrum、Kanban等）。框架是可选的——你可以根据项目类型选一个最适合的。最上层是**实践**（站会、看板、迭代规划、用户故事）。实践是可调整的——同一个团队在不同项目阶段可以用不同的实践。

很多团队的问题出在哪？他们把最上层的实践当成了敏捷本身——以为开了站会就是敏捷了，用了Scrum就是敏捷了。但当他们发现站会开成了汇报、Scrum跑成了赶工的时候，他们不知道哪里出了问题，因为没有人告诉过他们底层的原则是什么。这一节就是补上这个缺口——让你在看到"标准框架"的同时，也理解"这些框架为什么这样设计"。

#### 一、敏捷宣言——四个价值观和十二条原则

2001年，17位软件行业的思想家聚在一起，写下了敏捷软件开发宣言。这份宣言只有四句话，但它改变了整个行业对"怎么做项目"的理解。

**四个价值观：**

1. **个体与互动 高于 流程与工具。** 意思是：流程和工具是为了帮助人的，不是反过来。如果某个流程让团队更累了但没产出更好的结果，不要死守流程。最好的沟通方式是两个人面对面聊十分钟，而不是在工具上开一个三天的审批流程。

2. **可工作的软件 高于 详尽的文档。** 意思是：给用户一个能用的东西，比给用户一份完美的需求文档有价值得多。文档不是不重要——只是优先级在"能用"之后。一个能跑但文档粗糙的产品，比一个文档精美但还没做出来的产品有用一百倍。

3. **客户合作 高于 合同谈判。** 意思是：你和你客户的关系应该是共同解决问题，而不是甲方乙方地互相提防。当需求变了，最好的方式是坐下来一起讨论怎么调整，而不是拿出合同说"当初你签的就是这个，现在不能改"。

4. **响应变化 高于 遵循计划。** 意思是：计划有用，但变化来的时候，调整计划比死守计划更明智。不做计划不对——但做了计划之后发现环境变了还不改，是更严重的错误。

这四个价值观不是"不要流程、不要文档、不要合同、不要计划"。它们的原话里都带着"高于"这个词——这不是非此即彼，是"在冲突的时候优先选哪一个"。

**十二条原则中与本书实操最相关的几条（日常语言版）：**

**原则1：尽早交付、持续交付。** 不要等到所有功能都做完了再给用户看。做一个功能就交付一个，让用户尽早用上。你早一周拿到用户反馈，你就少浪费一周在做错误的事情上。本书的迭代节奏设计（步骤2.2）和Sprint的固定时间盒，都是为了让"尽早交付"变成可执行的节奏。

**原则2：欢迎需求变化。** 不要害怕需求变。在项目后期欢迎变化也没问题——关键是要有一套能快速响应变化的机制。这就是为什么你需要好的规划框架和迭代节奏（第一步拆分、第二步校准），而不是因为怕变化就把需求写死了再开工。本书规划阶段的风险场地图（步骤2.4）和Sprint Planning中的方向检查（步骤3.1），都是让变化能被主动管理而不是被动承受。

**原则3：业务和开发每天都在一起工作。** 产品负责人（PO）不是"每个迭代开始时发一份需求给团队然后就消失了"的角色。他应该每天都在团队附近，随时回答团队的疑问。本书的步骤1.1（方向校准）和步骤3.1（Sprint Planning）的成功，都依赖PO的持续参与。

**原则4：受激励的个体。** 给团队信任、支持和自主权，他们自然就会做出好的东西。不需要微观管理，不需要打卡，不需要填工时表。你给团队一个清晰的方向（启动阶段的工作）、一个合理的节奏（规划阶段的迭代设计）、一个安全的反馈空间（执行阶段的站会和Retro），然后相信他们会自己找到最好的方式去实现。本书的团队能量五维评估（步骤2.3）和能量恢复（步骤5.3），都是这个原则的具体落地。

**原则5：面对面的沟通。** 信息失真的最小化方式是"两个人直接对话"。越多的中间人、越多的邮件层级、越多的汇报文档，信息的损失就越大。本书的每日站会（步骤3.2）从右往左看板的做法、迭代回顾会中的安全空间原则（步骤4.4），都是为了保护和促进面对面的直接沟通。

**原则6：团队定期反思如何更有效。** 这是敏捷的自我改进机制。不是做完一个项目才反思，是每个迭代结束都反思——今天比昨天好在哪，哪里还需要改进。这就是迭代回顾会（Retro，步骤4.4）存在的根本原因。

这四个价值观和六条原则，是所有敏捷框架的底层操作系统。你看后面的Scrum和Kanban时会发现，每一个事件、每一个角色、每一个工件，都能追溯到某一条原则。这不是巧合——框架是原则的具体化。

#### 二、Scrum框架——三个角色、五个事件、三个工件

Scrum是当前最主流的敏捷框架。它是一个用固定节奏驱动工作的机制。

**三个角色：**

- **产品负责人（PO）。** 负责决定"做什么、为什么做、优先做什么"。PO是团队和用户之间的桥梁——他理解用户的需求，把它翻译成团队能执行的任务，然后按优先级排序。PO不是"老板"——老板告诉你"做什么"，PO告诉你"用户需要什么"。PO最重要的能力不是管理，是做决定——当两个需求冲突的时候，他能说"先做这个，那个下个迭代再做"。听上去简单，但实际上是最难的角色，因为你需要不停地拒绝别人的需求。

- **Scrum Master（SM）。** 负责保护团队、移除障碍、促进Scrum流程的正确执行。SM不是项目经理——项目经理管人，SM管流程。当一个外部干系人想在迭代中间插入一个紧急需求时，SM会说"这个需求可以放到下一个迭代的Planning里讨论"——他保护团队的节奏不被打破。当一个开发被一个技术问题卡了两天时，SM会去协调资源帮他解决。一个好的SM是"让团队可以专注做事的人"。

- **开发团队。** 跨职能的、自组织的3-9人小团队。"跨职能"的意思是团队里拥有从设计到开发到测试到部署的所有技能——不需要依赖外部团队才能完成一个功能。"自组织"的意思是团队自己决定"怎么做"——不是由外部的人指派任务，而是团队站在一起说"这个迭代里，谁最适合做什么"。团队太大（超过9人）会导致沟通成本爆炸，团队太小（少于3人）会导致技能覆盖不足。

**五个事件（所有事件都有固定的时间盒，不能超时）：**

1. **Sprint（迭代）。** 核心单位。1-4周（通常2周）的一个固定时间盒。每个Sprint结束时要产出一个"可用的、潜在可交付的"产品增量。Sprint的核心不是"每两周交付一次"，而是"每两周获得一次来自真实世界的反馈"——这个区别决定了你把Sprint当成"催进度的手段"还是"校准方向的机制"。

2. **Sprint Planning（迭代规划）。** 每个Sprint开始时的方向校准。PO从需求列表里挑出最重要的目标，团队把这些目标拆成具体任务，然后团队确认"这个Sprint结束时，用户能做这件事"。好的Planning产出的不是一个任务清单，是一个"这周我们要解决谁的问题"的共同理解。

3. **Daily Scrum（每日站会）。** 每天15分钟。不是汇报，是同步。核心问题不是"你昨天做了什么"，而是"从昨天到今天，有什么东西阻止了工作的流动"。本书的站会做法（从右往左看板）就是对这个原则的最佳实践。

4. **Sprint Review（迭代评审）。** 每个Sprint结束时，把做出来的东西展示给真实用户或干系人看。不是"演示"，是"收集反馈"。这一小时中最重要的声音不是团队的，是用户的。用户说"这个不是我要的"——那不是失败，是你在Sprint结束时就发现了问题，而不是在项目上线六个月后才发现。

5. **Sprint Retrospective（迭代回顾）。** 每个Sprint结束时，团队内部反思"这个Sprint我们哪里做得好、哪里需要改进"。只选一个改进项，下个Sprint落地。不是吐槽大会，不是批评大会——是"如何在下个Sprint变得更好一点点"的团队讨论。

**三个工件：**

1. **Product Backlog（产品需求列表）。** 项目所有需求的唯一来源。它是一个动态的、按优先级排序的列表——PO持续地根据用户反馈和市场变化调整排序。"唯一来源"的意思是：任何要开发的东西，都必须先出现在这个列表里。没有的东西就是"不在当前项目的范围内"。

2. **Sprint Backlog（迭代需求列表）。** 本次Sprint要做的具体任务列表，加上如何实现这些任务的计划。它是团队对"这个Sprint我们要完成什么"的承诺。

3. **Increment（增量）。** 每个Sprint结束时产出的可用的产品增量。"可用"的意思是——达到了团队定义的"完成"标准（测试通过了、文档写好了、可以部署了）。"完成"的定义很重要：如果团队把"代码写完了"叫完成，但测试还没做、部署还没跑，那就不是真正的增量——只是一个半成品。

**Scrum完整节奏表（以两周Sprint为例）：**

| 时间 | 周一 | 周二 | 周三 | 周四 | 周五 |
|------|------|------|------|------|------|
| **第一周** | 开发工作 | 开发工作 | 开发工作 | Daily Scrum 9:15-9:30 | Daily Scrum 9:15-9:30 |
| | Daily Scrum 9:15-9:30 | Daily Scrum 9:15-9:30 | Daily Scrum 9:15-9:30 | 开发工作 | 开发工作 |
| **第二周** | Daily Scrum 9:15-9:30 | Daily Scrum 9:15-9:30 | **Sprint Planning 9:00-10:30** | Sprint Review + Retro 16:00-17:00 | (下个Sprint开始) |
| | 开发工作 | 开发工作 | 开发工作 | 开发工作 | |

这张表展示了固定节奏的含义：每天9:15站会（15分钟），每两周周期结束时做Planning（下个迭代的规划，约90分钟）和Review加Retro（本周的评审加回顾，约1小时45分钟）。剩下的所有时间都是开发时间，不需要协调、不需要请示——团队自己决定做什么。

#### 三、Kanban——可视化、在制品限制、流管理

Kanban不同于Scrum，它不定义角色、不定义事件。Kanban只做一件事：让"工作是如何流动的"变得完全可见。

**四个核心实践：**

1. **可视化工作流。** 把工作的每个阶段画成一列，每项工作写在一张卡片上，贴在对应的列里。你看到的不只是"每个人在做什么"，而是"整个系统的工作状态"——哪一步在堆积、哪一步是瓶颈、哪一步被卡住了，全部一目了然。一张图，一个团队的协作状态就完全暴露在那里。

2. **限制在制品（WIP）。** 给每个工作阶段设定一个最大卡片数。比如"进行中"列最多5张卡——这意味着团队最多只能同时做5件事。当达到上限时，不能再从左边拉新的工作进来。这个机制强制解决的问题是：同时做太多事情导致每件事都做不完。限制在制品后，团队被迫把已经开始的做完，才能开始新的事情。结果是——完成的事情反而变多了。

3. **管理流动。** 持续跟踪卡片从开始到完成需要多久（Cycle Time）。不是"催卡片走快点"，而是找到让卡片流动变慢的原因并消除它。是评审流程太复杂？是在制品限制被打破了？还是某个环节的人手总是不够？管理流动的本质是系统性消除瓶颈，而不是逼个体加更多的班。

4. **持续改进。** Kanban没有固定的回顾事件，但它的改进机制是持续内嵌在实践中的——每次你看到一个瓶颈、调整了一个在制品限制、发现了一个流动障碍，你就在做一个改进。Kanban的改进不是"每迭代一次才讨论改进"，而是"任何时候发现问题，随时改进"。

**Kanban和Scrum的关系：不是非此即彼。**

这是一个常见的误解。很多团队以为要在Scrum和Kanban之间选一个，然后就必须忠于一个框架。实际上，最好的团队往往两者都用：用Scrum的固定节奏（Sprint）来驱动反馈循环，用Kanban的可视化和在制品限制来管理工作流。这种混合方式叫做"Scrumban"——它用Scrum的时间盒保证周期性校准，用Kanban的看板和在制品限制保证流动效率。

**什么情况下用纯Kanban比Scrum更合适：**

- **运维/支持类工作。** 如果你的工作是响应线上告警、处理用户工单、修复Bug——你无法把需求打包成Sprint，因为工作是被事件驱动的，不是被规划驱动的。此时Kanban的连续流模式更合适。

- **需求持续涌入，无法按Sprint打包。** 有些项目类型决定了需求源源不断地进来，比如一个API平台的开发。API团队很难说"这周我们只做这三件事"——因为外部依赖方随时可能提出新的集成需求。Kanban让团队能随时拉入新需求，同时用在制品限制确保不超载。

- **团队需要极大灵活性。** 如果Scrum的固定节奏让团队感到束缚——不是因为团队不自律，而是因为工作性质决定了节奏不能固定（比如市场驱动的快速响应团队）——那么Kanban可能更适合。

#### 四、这个框架和本书六个阶段的关系

你现在可能在想：Scrum有五个事件、三个角色、三个工件，Kanban有四个核心实践——那这些和本书的六个阶段（启动、规划、执行、监控、收尾、维护）是什么关系？

答案是：Scrum和Kanban告诉你"做什么事件"（站会、评审、回顾），本书告诉你"怎么做每个事件、为什么这样做"。它们是两个层面的东西。

下面这张映射表可以帮你理解：

| Scrum/Kanban事件 | 对应本书的阶段和步骤 |
|-----------------|-------------------|
| Sprint Planning | 执行阶段步骤3.1——怎么做一次好的迭代规划（从方向校准到任务分解到能量检查） |
| Daily Scrum | 执行阶段步骤3.2——怎么做一次不浪费时间的站会（从右往左看板、只讨论卡住的卡） |
| Sprint Review | 执行阶段Sprint结束时的反馈环节——结合监控阶段的燃尽图（步骤4.1）和累积流图（步骤4.2）一起看数据 |
| Sprint Retrospective | 监控阶段步骤4.4——怎么做一次能产生真正改进的回顾（只选一个改进项、确保落地） |
| Product Backlog管理 | 规划阶段步骤2.1（用户故事地图）——怎么把需求组织成一个有优先级的列表 |
| Sprint Backlog | 执行阶段步骤3.1（Sprint Planning中的任务分解）——怎么把目标拆成具体任务 |
| Kanban看板 | 执行阶段步骤3.3——怎么设置看板列和在制品限制，让工作流动起来 |
| Sprint / 迭代节奏 | 规划阶段步骤2.2——怎么决定迭代长度、怎么固定仪式时间窗口 |
| 能量管理（非Scrum标准事件） | 规划阶段步骤2.3（团队能量预算）+ 收尾阶段步骤5.3（能量恢复）——Scrum框架本身没有这个环节，但它是项目成功的关键 |
| 风险预映射（非Scrum标准事件） | 规划阶段步骤2.4（风险场地图）——Scrum框架本身没有这个环节，但本书补充了它 |

总结成一句话：**Scrum和Kanban给了你一张"什么时间做什么事"的日程表；本书给了你"做这些事的时候具体怎么操作、为什么这样操作"的操作手册。**

从启动到收尾，这六个阶段覆盖了一个完整项目生命周期中的所有关键节点。Scrum的五个事件覆盖了其中三个阶段（执行和监控）的核心节奏。本书在其他阶段（启动、规划、收尾、维护）补充了Scrum没有细讲但同样关键的工作。所以它们不是矛盾的——它们是互补的。你可以在Scrum的框架下跑完启动、规划、执行、监控、收尾、维护的完整周期，每个阶段用本书的具体步骤来指导你的操作。

# 第零章之二：敏捷为什么这样划分阶段——每个阶段的目的、目标与重点

这是整本手册最重要的前置章节。如果你只读一章，请读这一章。

敏捷把项目生命周期划分为六个阶段：启动 -> 规划 -> 执行 -> 监控 -> 收尾 -> 维护。这个划分不是人为随意的，而是对一个复杂系统"从无到有、从有到稳"这个自然过程的结构化表达。理解每个阶段为什么存在、承担什么功能、跳过或压缩会付出什么代价——比记住每个阶段的具体步骤重要得多。

## 阶段总览

| 阶段 | 团队状态 | 核心动作 | 一句话目标 |
|------|---------|---------|-----------|
| 启动 | 一群散的个人，有资源没对齐 | 明确方向、划定边界、识别关键人物 | 确保所有人朝同一个方向 |
| 规划 | 方向清晰但路径有大量空白 | 分解需求、设定迭代节奏、评估团队能量 | 建一套能持续做出好决策的框架 |
| 执行 | 在固定节奏中持续产出 | 按迭代推进、每日校准方向、交付增量 | 用节奏驱动产出，用反馈保方向 |
| 监控 | 在执行的同时退后观察 | 采集真实信号、发现偏差、在不可逆前干预 | 在问题变严重之前看见问题 |
| 收尾 | 增量完成了，但活儿还没完 | 验证成果、归档经验、恢复团队能量 | 善终，不在下一次重复同样的错误 |
| 维护 | 系统上线，进入运营 | 从冲刺切换到稳态、持续小改进 | 用符合维护期的方式做维护 |

---

## 阶段一：启动——为什么不能"先干起来再说"

**为什么需要这个阶段：** 项目中最昂贵的问题不是技术难题，不是资源不足——是"每个人以为大家在做同一个项目，但实际上每个人对成功的理解不一样"。启动阶段被跳过或压缩，意味着项目从一开始就在积累信息失真。方向不一致的成本是项目中最隐性的成本：A团队在开发功能A，B团队在做接口对接功能A，但A和B对"功能A"的理解差了半个维度——这种偏差会在项目中期以"返工"和"冲突"的形式爆发。

时间上看，这些问题的爆发点在项目后期甚至验收阶段，但根源在启动阶段。启动阶段的投入是整条项目曲线的投入产出比最高点——用一天校准方向，可以避免后期数周的返工。这就像盖房子：地基的投入只占房屋总成本的10%左右，但如果地基歪了，后面所有的投入都是浪费。

**核心目标：** 让每个团队成员和关键干系人对"我们要做什么、为什么做、做好的标准是什么"有相同的理解。注意关键词是"相同"——不是相同的PPT文档，是相同的心理画面。你问团队里的任何一个人"这个项目的成功标准是什么"，他能用自己的话说出来，内容和其他人说的对得上。

**阶段重点：**

- 项目方向必须通过集体参与才能对齐——负责人写好然后群发给大家是通知，不是对齐。通知只会制造"我好像知道该做什么但我不确定"的隐性信息失真。
- 真正影响项目的人的识别，比显性干系人的管理优先级更高。"不在项目名单上的那个人"如果在启动阶段没有被识别出来，项目中期他的阻力会比任何一个显性干系人的阻力大一个数量级。
- 红线（硬边界）不是限制——是解放。没有红线的团队，在每一个微小决策上都需要请示；有红线的团队，在红线以内拥有充分的自主判断空间。

**如果跳过或压缩启动阶段：** 三个月后你会反复听到这些声音——"我以为这个项目是做X的""为什么没人告诉我Y部门也要参与""这个需求我们当时不知道"。每一句话对应启动阶段省掉的一个动作。启动省一天，后期用十天补。这是整个项目周期中最糟糕的时间投资。

---

## 阶段二：规划——为什么不一次性规划完

**为什么需要这个阶段：** 传统项目管理的规划假设是"未来是可预测的，所以我们可以提前做一份详尽计划然后严格执行"。这个假设在稳定环境中成立，但在大多数项目中不成立——需求会变、市场会变、团队自己也会在项目过程中学到新东西而改变之前的认知。

敏捷规划的出发点不是"预测未来"，而是"设计一个能适应变化的框架"。所以敏捷规划做两件事：宏观上设计足够清晰的结构（用户故事地图、迭代节奏、团队能量预算），微观上只对未来近期做详细规划。远期只保留方向性的、粗粒度的计划。

一份详细的迭代计划在执行两周后，其中的大部分假设已经过时。不是规划做错了，是环境变了。敏捷规划的应对方式不是"不做规划"，而是"高频次地重新校准"。规划不是一次性的产出物，是一个持续的活动。

**核心目标：** 不是产出一份无法改变的详细计划，而是建立一套"让团队在变化中仍能做出良好决策"的框架——优先级清晰的Backlog、健康的迭代节奏、被主动管理的团队能量储备。

**阶段重点：**

- 任务之间的信息交换点是规划中最需要关注的对象。两个模块之间的依赖关系、两个团队之间的交接点——这些接口处的信息失真积累速度是整个项目中最快的。不同的人对接同一个模块，各自的假设不同，等到集成时才发现对不上。
- 迭代长度没有"最优值"——第一个迭代周期就是校准期。前三个Sprint的数据决定合适的迭代长度，而不是在项目开始前靠"分析"算出来。有些团队一周一个迭代效果最好，有些三周最好。靠实际运转来试。
- 团队能量储备中最容易被忽略的维度是新团队之间的"信任存量"。两个从未合作过的团队共同开发一个模块，前几周的效率天然比合作过的团队低——这不是能力问题，是默契需要时间建立。规划时不考虑这个因素，就会把正常磨合期误判为团队能力不足。

**如果跳过或压缩规划阶段：** 团队在迭代3到4左右会遇到三个典型问题——"我们接下来该做什么？"（优先级不清晰导致方向丢失）、"接口处连接不起来"（模块之间的依赖没有被标注和主动管理）、"不是做不完就是做完后大改"（团队能量预算没做或做虚了）。

## 阶段三：执行——为什么用固定时间盒而不是"做到完成为止"

**为什么需要这个阶段：** Sprint存在的根本理由不是"每两周交付一次"——而是"每两周获得一次来自真实世界的反馈"。这个区别极其关键：前者的目标是产出，后者的目标是校准。固定时间盒的核心功能是强制了一个"必须停下来、拿出你手里的东西、给别人看、听反馈"的节奏。这个节奏本身是项目最核心的风险管理机制。

这就像在黑夜里走路：固定时间盒就是每隔一段时间必须停下来，用手电筒照一下前方的路。如果你不停下来看路，你走得再快，也可能在下一个弯道掉进坑里。你走得越快，掉进坑里的后果越严重。

假设你在做一个产品，连续做了两个月，然后第一次给用户看。如果这两个月中所有假设——用户需求、技术方案、市场条件——恰好全部成立，你没有浪费时间。但如果其中任何一个假设是错的，你浪费的是两个月。Sprint把"不知道假设是否成立"的时间窗口压缩到两周以内。即使假设错了，你最多损失两周。

**核心目标：** 以固定节奏持续交付可用的增量，在每个周期结束后收集来自用户和团队的真实反馈，用于校准下一个周期的方向。

**阶段重点：**

- Sprint Goal比Sprint Backlog重要得多。Goal是这个Sprint结束时用户能做什么，Backlog是实现Goal的手段。没有Goal，Backlog就只是一个"任务列表"——你完成了所有任务但可能什么都没有真正交付给用户。
- Daily Scrum的核心机制不是"汇报进度"，是"每天清零信息失真"——不让任何信息不对称或阻塞超过24小时不处理。如果有人卡在某个问题上超过一天没有说出来，那这一天的努力就是在错误的方向上。
- 在制品限制是保护团队能量不被过度消耗的硬性机制。没有在制品限制的看板只是一个高级任务列表，不是流管理系统。同时做太多事情的结果是每件事都在半成品状态，一件事都没交付。

**如果跳过或压缩Sprint节奏（比如取消Review直接进入下一个Sprint）：** 团队在封闭环境中持续基于假设做决策，直到上线后用户说"这不是我想要的"。每个被跳过的Review都是积累在两个Sprint之间未被校准的信息失真。节省的一两小时Review时间，代价可能是数周的返工。

- **跨行业适配：** 执行阶段的"Sprint"在不同行业节奏不同。软件可以1-2周，硬件通常需要4-12周的里程碑窗口，建筑/医药以月为规划单位。核心不变的是：固定节奏+强制反馈。

## 阶段四：监控——为什么不能只看"做完了多少"

**为什么需要这个阶段：** 监控和执行在时间线上是并行的，但它们在认知模式上是两个完全独立的活动。执行的心态是"向前推进"——保持速度、克服障碍、把任务打勾。监控的心态是"退后观察"——停下来、看数据、问"方向对不对"。两种心态很难由同一个人在同一时间完成，所以项目需要一个独立的监控职能（不一定是专门的人，但一定是专门的思维切换）。

大多数项目的问题不是"做得不够快"，而是"做了不该做的事"。监控就是为了在错误方向上消耗的资源超标之前叫停或者修正。敏捷项目之所以容易出"忙碌但无效"的状态，恰恰是因为执行效率很高，但监控被压缩成了"燃尽图看一看"的形式主义。

为什么这个做法？因为项目的数据像汽车的仪表盘：你不能只盯着速度表（完成了多少），你还需要看油表（团队还有多少能量）、水温表（是否过热）、导航（方向是否正确）。只看一个指标，就相当于看着速度表开车——你能开得很快，但可能开错路或者开到没油。

对燃尽图的正确使用方式不是"看看线是不是偏了然后催团队"，而是"发现偏了之后问：是什么导致了偏？是估算问题？需求变化？还是信息失真？"

**核心目标：** 发现"项目的真实状态和预期状态之间的偏差是什么导致的"——是方向需要调整（目标变了）、信息失真加剧了（报喜不报忧更严重了）、还是团队能量在系统性下降（团队越来越疲惫）。

**阶段重点：**

- 燃尽图的价值不在"总量"，在"变化趋势"。完成速度的变化趋势比当前完成量本身早数天给出预警信号。曲线的斜率在变平，比完成百分比低了几个百分点更重要。
- 累积流图和燃尽图必须同时看——一个告诉你还剩多少工作量，一个告诉你工作卡在哪个环节。两张图一起才能定位问题出在哪里。
- 团队能量水平和信息准确度这两个领先指标，比传统进度和成本指标敏感性高得多。但当团队能量开始下降时，它会在几周后才体现在进度数字上。监控的意义就在于提前几周发现这些信号。
- 任何单个指标都会说谎。交叉验证是监控的基本功——至少两个独立来源的数据同时指向同一个结论，才能确认一个判断。

**如果监控沦为"看燃尽图然后催进度"：** 团队会开始美化燃尽图——在迭代过程中悄悄修改剩余工作量的估算，让曲线看起来更漂亮。这是最恶性的失真循环：监控本身制造了更多的信息失真，让决策者离真实状态越来越远。

- **跨行业适配：** 监控工具随行业变化。燃尽图适用于可拆解为小任务的项目；硬件/建筑更适合里程碑趋势图和实物完成百分比。FDI（信息准确度）是所有行业的通用领先指标。

## 阶段五：收尾——为什么不能"做完了就散了"

**为什么需要这个阶段：** 收尾是敏捷项目中最容易被忽略的阶段。习惯了"一个Sprint接一个Sprint"节奏的团队，很容易在项目交付后直接跳入下一个项目。但收尾阶段承担三个不可替代的功能。

第一，诚实地面对"我们到底做成了什么"。项目启动时我们说要解决用户的一个问题，现在问题真的解决了吗？不是团队自己打分，不是测试用例全部通过——是真实用户说的。如果差距存在，记录差距，这不是失败——这是学习。跳过这一步，你永远不知道自己有没有做成。

第二，经验沉淀——这个项目中被验证有效的做法和被证伪的假设需要被保留下来。不是写没人读的总结报告，是用最轻量的方式保留核心信息：关键决策的"为什么"、最严重的信息失真事件、团队的工作模式特征。沉淀不是给文档库添砖加瓦，是让下一个项目团队不必从零积累判断力。

第三，团队能量恢复——团队成员在一个项目中消耗的精力、注意力、信任、学习勇气，需要一次系统性的恢复。一个项目结束后不恢复就直接进入下一个项目，团队的能量储备会在两三个项目后见底。连续跳过三次能量恢复，核心人才会帮你执行"能量恢复"——离职。这就像跑完马拉松不放松不休息，直接去跑下一个马拉松。你可能撑得住一次，但撑不住第三次。

**核心目标：** 让项目善终——验证成果是否真实解决了问题、归档经验让下一次更好、恢复团队能量让下一次还能跑起来。

**阶段重点：**

- 成果验证必须有真实用户参与，不是团队内部验收。团队的自我评价和用户的真实感受之间通常有显著差距。内部验收只能验证"做完了没有"，不能验证"做对了没有"。
- 归档的质量不由字数决定。三张A4纸（决策日志、信息失真事件记录、团队工作模式特征）比一份五十页无人阅读的总结报告有价值得多。问一个简单的问题：三个月后新来的团队，看这些材料能不能快速理解这个项目的关键教训？
- 团队能量恢复是最容易被跳过、但跳过后果最严重的环节。它不产出"可见的交付物"——但没有它，下一次项目的产能从起点就被打了折扣。管理者看不到它所以认为它不重要，恰恰是最大的陷阱。

**如果跳过收尾阶段：** 下一个项目启动时，团队带着上一个项目的疲劳和未处理的信息失真，没有从上一个项目的经验中学到任何东西，关键决策的"为什么"只存在于已经离开的人的记忆中。这不是"项目结束了"，这是"问题被平移到了下一个项目"。

---

## 阶段六：维护——为什么维护不能按项目的方式管理

**为什么需要这个阶段：** 维护阶段和前面五个阶段的本质差异在于：目标变了。项目期的目标是在有限时间内交付一个完整的成果；维护期的目标是长期保持系统健康并持续做小改进。管理节奏必须跟着目标变。

为什么这个做法有效？因为不同的目标需要不同的节奏。项目像百米冲刺——集中所有力量在短时间内冲到终点。维护像慢跑健身——持续的、低强度的、注重长期健康的运动方式。如果让一个百米运动员用冲刺的方式去跑马拉松，他撑不过三公里。

同理，如果用项目冲刺的方式来管理维护——每两周一个Sprint、每个Sprint有Deadline压力、用工作量完成量来衡量团队表现——团队会在没有真正紧迫截止日期的情况下持续处于高压状态。这就是"敏捷疲劳"最常见的来源：用做项目的节奏做运营。

维护阶段需要从冲刺节奏切换到稳态节奏。迭代周期拉长、仪式简化、指标从"产出量"转向"稳定性"。对用户来说，系统稳定的运行和偶尔的小改进，比"每周一个新功能但偶尔崩一次"好得多。

**核心目标：** 系统上线后保持稳定运行，以可持续的节奏进行增量改进，防止系统在无人注意中逐渐恶化。

**阶段重点：**

- 维护期的核心指标不是产出量，是稳定性（平均修复时间、平均恢复时间）和用户满意度。如果你在维护期还用工作量作为考核指标，你还没有真正切换到维护模式。
- 信息监控必须比项目期更轻量。每周站会和每日站会不适用于维护期——月度快速检查和季度深度检查就足够了。维护期也要管理团队能量——只是方式不同，不再是冲刺式的紧张，而是关注长期可持续。
- 维护期最有价值的工具是"问题热点图"——标注哪些模块最容易出问题、哪些代码段让新人修改时最容易犯错、哪些文档和实际运行的代码差距最大。这张"热点图"决定技术债清偿的先后次序。

**如果维护阶段用项目期的管理方式：** 团队会一直处于"冲刺"状态，即使实际上没有紧迫的截止日期。这是维护期团队能量持续消耗的最大原因。"敏捷疲劳"本质上就是"用做项目的节奏做运营"的后果。系统在稳定运行，但团队的心跳一直在做项目，迟早会绷不住。

# 第一部分：项目启动——让所有人朝同一个方向跑

项目启动不是开个kick-off会、发个立项通知就结束了。启动阶段唯一要做的事情是：确保每个参与项目的人，脑子里对"我们要做什么、为什么做、怎么做"的画面是一模一样的。不是相似，是一模一样。这个阶段省掉的时间，会在项目中期以十倍的成本还回来。

---

## 步骤1.1 定义项目真正的方向

### 怎么执行：

**第一步：核心团队围在白板前，PO先写出三个问题的答案**

召集不超过7人的核心团队（PO、SM、关键开发者、设计师），每人拿一支马克笔站在白板前。Product Owner先在白板上写出三个问题的答案：

- 谁在用我们的产品？他们现在最大的痛苦是什么？
- 做完这个项目之后，他们能做什么现在做不了的事？用用户自己的话写，不要用技术语言。比如写"在手机上点两下就能看到库存"而不是"后端对接WMS系统实现实时库存查询"。
- 如果六个月后我们只做对了一件事，那件事是什么？

PO写的时候其他人不说话。写完之后，所有人默读一遍。

**第二步：红笔划掉不同意的，绿笔写下替代方案**

每人拿一支红笔，直接在白板上划掉自己不同意的词、句子、甚至整个段落。不许口头争论——笔就是你的嘴。划完之后拿绿笔，在被划掉的地方旁边写你的替代方案。

这个过程中会有人想说话。制止。写成文字的力量比口头争论大十倍——写下来意味着你要为自己写的答案负责。

**第三步：合并所有人的红绿笔结果**

PO把红笔划掉的部分和绿笔替代的部分汇总起来，形成一版新的答案。重新写一遍白板。然后重复第二步——再每人拿红笔划掉自己不同意的。两轮之后，剩下的就是团队所有人没有异议的内容。这就是项目真正方向的初稿。

**第四步：做取舍测试**

在方向初稿下面，每个人写三行填空：

- "所以我们选择做______，即使这意味着不做______。"
- "所以我们选择做______，即使这意味着不做______。"
- "所以我们选择做______，即使这意味着不做______。"

如果三行都填不出来，说明方向还不够清晰——你没有做真正的取舍。一个没有舍弃的方向不是方向，是愿望清单。回到第三步，重新审视方向表述，直到每个人都能填出至少两行。

**第五步：把方向做成实体卡片贴在墙上**

把确认后的方向手写在一张A4硬卡纸上（不要打印，手写的过程会让它更有重量），贴在团队工作区的正中央墙上。每个Sprint Planning开始前，先读一遍这张卡片。

### 为什么这么执行：

不清晰的方向是项目最大的隐形杀手。当每个人脑袋里对"成功"的想象不一样时，所有人都在拼命做自己认为对的事——但合力的结果可能是零，甚至为负。这就好比让五个人推同一辆车，每个人往不同的方向使劲。你越用力，车越不动。项目中最贵的成本从来不是技术债或返工，而是"每个人都在做，但做的是不同的项目"。

白板加红绿笔的做法之所以有效，是因为它绕过了"嘴上说同意但心里不认同"的陷阱。开会的时候大家不好意思当面反对——没有人想在第一天就当刺头。点头容易，但划掉一个字需要勇气。笔迹不会撒谎。当你让别人亲手划掉他不认同的词时，你得到的是真实的校准，不是虚假的共识。这就是为什么"不许说话、只许动笔"这个规则如此重要——它把虚假的礼貌过滤掉了。

取舍测试（"我们选择做X，即使意味着不做Y"）抓住了方向最核心的本质：真正的方向一定包含舍弃。什么都想要等于没有方向。你不能在创业时既冲市场份额又保极致利润还要求快速扩张——这三个方向在资源有限时天然冲突。项目也一样。如果你说不出"我们选择不做哪些事"，说明你的方向还不够具体。到了项目中期，资源冲突时每个人都会按自己认为的优先级做选择——那时候你才发现大家从来没有真正同意过。

### 目的：

让团队在面对任何"做A还是做B"的决策时，有相同的判断标准。

### 重点：

- 方向不是PO一个人写了发给大家——那是通知，不是校准。必须所有人参与修改。即使PO写的初稿已经很好，大家也必须亲手用笔改动过。改动的动作本身参与了校准。
- "所以我们选择做______，即使这意味着不做______"是方向的试金石。如果填不出来，说明你没有做取舍。做不出取舍的方向在项目中期没有指导意义——遇到冲突时每个人都会按自己的优先级做选择。
- 方向会变，但不能悄悄变。任何时候如果团队发现方向需要调整——来自用户反馈、来自市场变化、来自技术约束——必须用同样的白板流程重新校准。悄悄改方向等于没有方向。

---

## 步骤1.2 画出真正影响项目的人

### 怎么执行：

**第一步：画三个同心圆**

在白板上画三个同心圆，每个圆代表一层干系人：

- **最内圈**：写"每天参与决策的人"。通常是5人以下——PO、SM、技术负责人、设计师。这些人每天都在项目里，每个人的决定直接影响项目走向。
- **中圈**：写"不每天参与但能改变项目走向的人"。包括项目发起人、资源持有者、部门主管、客户代表、关键供应商。他们不参加每日站会，但他们在某个时刻说的"是"或"不"可以决定项目的生死。
- **外圈**：写"受项目影响但在决策中没有声音的人"。包括最终用户、客服、运维、合规审核员。他们不被邀请参会，但项目交付后每天承受结果。

在圆圈里写出每个人的名字或角色。不要写职位（"王总监"），写清楚"上层业务线VP张总，批预算的人"。

**第二步：用红色星号标出隐性关键人物**

在中圈和外圈中，用红色星号标注出"不在项目组名单里，但你做重大决定前会想去问一句的人"。这些人可能没有正式的项目职位，但团队的直觉告诉你要先问他们。这个直觉就是信号。

典型的隐性关键人物：上一个版本的技术负责人（新人不了解历史决策原因）、客户方的运营主管（不是签合同的人，但每天用产品）、数据合规专员（法律上审批，但系统设计阶段不在场）。

**第三步：写出每个红色星号的人的核心关切**

对每个标了红色星号的人，在白板旁边写一句话：这个人的什么利益、恐惧或目标可能与项目方向一致或冲突。比如"合规专员——他关心的不是功能多强，是所有用户数据流转路径必须有审计记录。我们的方案如果跳过了这个，他必然会叫停。"

**第四步：给每个人影响力评分**

在红色星号旁边标注1-5的影响力评分。评分的标准是：这个人如果说"不"，项目停摆的概率有多大。

- 5分：他一个人就能让项目中止。
- 4分：他能让项目推迟一个月以上。
- 3分：他的反对会导致重大返工。
- 2分：他的建议需要听取但不致命。
- 1分：影响有限。

**第五步：特别标注沉默否决者**

在所有红色星号中，再圈出一种特殊的人——"沉默否决者"：他们不参加会议、不回复邮件、对你的项目没有任何反馈。你以为他们没意见。不，他们只是没表达。当你把方案拿给他们签字的时候，他们才会说"不行"。

找到这些人，在启动后两周内，安排一次面对面的谈话（不是邮件，不是电话，是面对面）。谈话的目标不是说服他们同意你，是搞明白：他们为什么会否决？他们关心的那件事在不在你的规划里？

### 为什么这么执行：

项目的最大风险往往不是技术也不是资源，而是一个你没有考虑到的人，在关键节点说了一句"这个方案我不认可"。这类人的特点就是：他们不在一线，但他们的一个态度就能让整个团队的工作方向发生偏移，甚至叫停整个项目。他们没有正式的项目头衔，但你心里清楚——做重大决定之前，你得先问问他们的意思。

这就好比下棋：你不能只盯着自己的棋子，你得看清棋盘上所有能影响局面的子——包括对方阵营里那些看起来不起眼但关键时候能"将"你一军的子。三个同心圆的方法，就是逼着你在项目开始前，先把棋盘上所有能影响结果的人看清楚。

最内圈是每天都在棋盘上的人——你当然看得到他们。中圈是那些偶尔落子的人——你容易在规划中忘记他们。外圈是最容易被忽略的——他们从来不动棋子，但棋盘上的每一步都影响他们的利益。等到他们发现自己被影响了再来说话，往往已经是在项目的中后期。

最危险的是"沉默否决者"。这类人在项目过程中一言不发，你以为他们同意了——不，他们只是还没到说话的时候。等你把方案做好、资源投进去、团队运转起来了，拿去给他们签字。他们说"不行"。为什么之前不说？因为之前你没问。为什么没问？因为你在三个同心圆里根本没写上他们。这个操作的价值就在于此：在只花一个对话成本的阶段，把"以后要花数周返工才能解决的问题"提前发现。

### 目的：

知道谁在帮项目加速、谁可能让项目减速、谁的信号应该被提前聆听。

### 重点：

- 最容易漏的是两类人：实际使用产品的一线用户（他们的声音常常被"产品经理代表用户"给过滤掉），和运维与客服（他们会在产品上线后承受最大冲击，但规划阶段通常不在场）。
- 影响力评分不是用来讨好人——给定高分的目的是提醒你"这个人的意见你需要主动管理"。不要等到他来找你，你要在问题发生前就去找他。
- "沉默否决者"这块不提前做，项目中期一定会被一个你没想到的人拦下。这不是概率问题，是"你画了图就不会被漏掉，没画就一定会被漏掉"的必然。

## 步骤1.3 画出项目的三条红线

### 怎么执行：

**第一步：和项目发起人开30分钟会议**

约项目发起人（出钱或对项目成果最终负责的人），只开30分钟。不开1小时——30分钟刚好够聚焦。会议只有一个主题："什么情况下你会叫停这个项目？"

**第二步：把发起人的回答提炼为三条红线**

每个人说"这个项目不能失败"的时候，心里的"失败"定义是不一样的。你要把发起人心里的失败定义挖出来，写成三条红线。每条红线的格式必须是：

**"关于______，我们不能越过______。"**

例如：
- "关于预算，总开发成本不能超过280万元。"
- "关于时间，2026年12月31日前必须上线核心交易功能，否则年度合同作废。"
- "关于质量，用户支付链路的数据错误率必须为零。如果支付数据出错，项目立即暂停调查。"
- "关于范围，本次项目不涉及组织架构调整。如果发现需要调整组织架构，必须重新立项。"

**第三步：确认红线内的自主决策权**

问发起人第二个问题："在这三条红线以内，团队有没有自主决策权？"

如果发起人说"有"，把这句话写下来。如果发起人说"要看情况"——追问"看什么情况"，把条件和情况都写清楚。比如"预算在10万元以内的调整，团队自主决定。超过10万需要发起人审批。时间上如果提前完成可以自己决定发布窗口，但如果延期超过两周必须提前上报。"

**第四步：发起人签字，贴在墙上**

把三条红线和自主决策权范围打印在A4纸上，发起人签字。贴在方向卡片旁边。任何人都能看到。

### 为什么这么执行：

红线的作用不是限制团队，是解放团队。这是一个很反直觉但特别重要的事情：没有边界的时候，人反而更不敢动。没有明确红线时，团队做每一个决策都要猜测"这个事要不要向上请示"——一个决定的成本从1分钟变成半天甚至一周。你就好比给小孩画了一个活动范围——"院子以内随便玩"。知道了边界在哪，孩子在院子里怎么疯都行，因为他知道边界以内是安全的。没有边界的话，他反而畏手畏脚哪里都不敢去。红线就是这个"院子边界"——边界明确了，边界内的探索才能自由。团队不需要在大方向不明确的情况下战战兢兢地做决策，他们知道"在红线以内，我做的决定就是有效的"。

为什么要发起人签字？因为签字的动作意味着"我说过的话算数"。没有签字的红线是口头说说——项目中期的压力下，口头承诺是最容易被忘记的。到了项目中期预算紧张了，发起人可能已经不记得自己说过"280万以内可以自主决定"。但白纸黑字贴在墙上，谁都赖不掉。签字代表一种正式承诺。

为什么红线不超过三条？因为超过三条等于没有红线。五条以上的"红线"意味着什么事都要请示——那就不是红线，是束缚清单。红线的价值在于它的稀缺性——只有最关键的边界才被提升为红线。什么都重要等于什么都不重要。让发起人自己排序，选出他最在意的三个边界，其余的交给团队自己判断。

### 目的：

给团队一个"可以在里面安全奔跑"的空间，同时给发起人一个"放心放手"的依据。

### 重点：

- 红线超过三条等于没有红线。五条以上意味着"所有事情都要请示"。如果发起人列出了七条，帮他合并成三条。
- "质量"这条不能写成"高质量交付"——那是废话。质量红线必须可验证。怎么写："用户支付链路的数据错误率必须为零"——这就是可验证的。如果写不出可验证的质量红线，说明团队和发起人对"什么叫做好"还没有共识。——在启动阶段解决这个分歧，比在验收阶段发现好十倍。
- 红线必须书面签字。口头红线在项目中期遇到压力时会被遗忘或推翻。
- **跨行业适配：** 红线的优先级因行业而异。软件通常预算/时间优先；硬件/制造行业质量安全红线最高；医药/器械行业合规红线压倒一切。做红线设定时先问"什么错误在这个行业是不可接受的"。

## 步骤1.4 启动工作坊——把一群人变成一个团队

### 怎么执行：

**第一步：安排完整的半天，关闭所有通知**

安排一个完整的半天（4小时），中间安排15分钟茶歇。所有人关掉电脑、关掉手机通知。不许迟到，不许早退。不是"尽量参加"，是"这是项目启动的唯一一次工作坊，不在等于项目开始就没有你"。

**第二步：严格按四段议程推进**

**第一段——方向校准（45分钟）**

把步骤1.1产出的方向卡片贴在所有人能看到的地方。每人轮流向团队说一句话：

"这个方向对我意味着，我的工作重点是______，我可以放下______。"

重点听"我可以放下"的部分。如果有人说"我放不下"或"我什么都放不下"——这意味着方向还不够清晰。这个人不知道什么是不重要的。在方向上还有没有对齐的地方，停下来继续校准，不要推进议程。

**第二段——共创团队协议（60分钟）**

拿出A1大白纸，分成四个区域，每个区域回答一个问题：

1. **怎么沟通？**——团队内部用哪个沟通渠道（企业微信、Slack、飞书？工作消息和非工作消息分开吗？紧急情况用什么通道？多长时间内回复？下班后和周末发消息合不合适？）
2. **怎么决策？**——哪些决策一个人决定即可（比如选哪个前端框架、代码风格）？哪些决策需要团队共识（比如架构选择、技术栈切换）？哪些决策必须上报发起人（超预算、延期两周以上、范围变更）？
3. **怎么处理冲突？**——意见不一致时，第一步做什么？（先各自写论据，然后开会，还是直接找第三方仲裁？）
4. **怎么给反馈？**——反馈当面给还是书面给？限定多长时间内给出？接收反馈的人不能反驳（先听，消化了再说）。

答案写在大纸上。每人签字。签了就意味着"我同意按这个规则合作"。

**第三段——恐惧清单（60分钟）**

每人拿三张便利贴，匿名写下"这个项目中最让我担心的三件事"。不署名，不讨论。收齐后贴在白板上，团队集体把它们分组归并。合并成几个大类。

然后团队一起投票：哪些恐惧是我们可以通过项目管理来缓解的？哪些是我们无法控制但可以提前准备预案的？

注意：如果所有人写的都是"担心进度不够快"——这是心理安全感还没建立的信号，团队在写"安全的答案"。此时项目经理或Scrum Master应该先写一个自己真实的恐惧作为示范。比如"说实话，我担心我之前带的项目都是靠加班撑过来的，这个项目我不想再走那条路，但我不确定不加班能不能做好。"

**第四段——第一周行动（30分钟）**

每人说出自己未来一周唯一一件最重要的事。只一件事。不是"我要做A、B、C三件事"——是"明天到公司，我先解决的那一件事是什么"。

这件事必须满足：它是一周的起点，如果这件事没做成，其他事都会受影响。

散会时，每个人都确切知道明天早上到公司先做什么。

**第三步：整理产出物**

工作坊结束后，整理四件产出物：
1. 方向卡片（确认版）
2. 团队协议（A1纸）
3. 恐惧清单Top5（拍照存档）
4. 第一周个人行动卡（每人一张）

拍照存档所有产出物。团队协议贴在方向卡片旁。

### 为什么这么执行：

高效团队和普通团队的分水岭不在于能力——而在于是否在项目开始时就建立了共享的心理模型。所谓"共享的心理模型"，就是每个人不需要讨论就能知道"在这种情况下，我的队友会怎么反应"。这种默契不是在同一个办公室里自然产生的，它需要刻意建立。

恐惧清单这个环节特别关键。未被表达的恐惧不会消失，它只会在项目压力大时以另一种形式释放出来——拖延、隐瞒坏消息、表面上同意但心里不打算做。这就像一根弹簧，你一直压着它不让它动，但它积蓄的能量不会凭空消失。压得越紧，反弹的力量越大。当一个人把担心匿名写在纸条上、被团队看见时，"只有我一个人在担心"的孤独感就消失了——那根弹簧的压力就松了。恐惧从"不能说的秘密"变成了"我们可以一起面对的问题"。

启动工作坊本质上是一次"团队对齐仪式"。表面上是开会，实际上是让每个人的脑袋运行在同一个频率上。团队协议的价值不是那一张纸写得多漂亮——而是当协作中出现摩擦时，你可以说"我们当时约定了这样做"，而不是"你为什么这样做"。协议是执行阶段的"交通规则"，不是事后追责的工具。

恐惧清单则是对团队中"台面下的问题"的一次排查。很多项目死在"大家都知道有问题但没人说"上。恐惧清单把这个"大家都知道但没人说"抬上了台面。说出来，它就不再是问题了——至少不再是暗伤。

### 目的：

项目开始时，每个成员都确切知道：我们要去哪里（方向）、我们怎么协作（团队协议）、我最担心什么（恐惧清单）、我今天做什么（行动卡）。

### 重点：

- 如果启动工作坊的时间被压缩到2小时以内——不是时间不够，是对启动不够重视。不要妥协。妥协意味着"我们不在乎能不能对齐"。
- 恐惧清单如果变成了形式主义（每个人都写"担心进度不够快"这类安全答案）——说明心理安全感还没有建立。领导先写一个自己真实的恐惧，示范脆弱性。团队协作中有个规律：第一个说真话的人决定了之后能说多真。
- 团队协议最大的价值不是写的时候，而是出现冲突时可以被拿出来说"我们当时约定了这样做"。如果协议写了就收在文件夹里再也不看，它就没有价值。必须挂在墙上。

---

## 启动阶段检查点

每一条用"是/否"回答。有一条"否"就不能进入规划阶段。

1. 每个核心团队成员能用自己的话说出项目真正的方向（不是背，是用自己的话——"我们这个项目就是要让一线销售在手机上三分钟完成报价申请"）。
2. 你知道谁是这个项目最关键的三个隐性关键人物，以及他们的核心关切各是什么。
3. 三条红线已书面确认，发起人签了字，贴在墙上。
4. 团队协议每个人都签了字，贴在墙上。
5. 恐惧清单Top5不是"安全的客套话"——里面包含至少一条真正让人不安的内容。
6. 每个人都知道明天早上到公司的第一件事做什么。

# 第二部分：项目规划——不只分解任务，更设计节奏

启动阶段结束意味着"方向对齐了、边界清楚了、关键人物找到了"。但方向对齐不代表计划清晰。第二阶段要做的是：把"我们想做什么"变成"我们怎么去做"，而且不只是分解任务——是设计一套让团队能持续有效运行的框架。

---

## 步骤2.1 用户故事地图——把需求变成用户旅程

### 怎么执行：

**第一步：邀请真实用户参与**

在安排工作坊之前，先花时间找至少一位真实用户。如果没有真实用户——用最接近用户的人：客服主管、销售冠军、用户研究员。没有用户代表就不要做用户故事地图。内部视角猜用户需求做出来的地图从一开始就是信息失真的源头，你根本不需要画。

**第二步：铺牛皮纸，画横轴——用户旅程**

在长条桌上铺一大张牛皮纸（或用拼接的白板纸代替）。横轴从左到右是用户从"第一次听说这个产品"到"成为忠实用户"的完整旅程。典型的阶段包括：了解产品→注册/下载→首次使用→日常使用→遇到问题→寻求帮助→持续使用→推荐给别人。

在每个阶段下面，用便利贴写出用户在做什么。写用户的行为和感受，不是功能描述。比如"我在同事的朋友圈里看到这个工具的截图"而不是"广告系统推送"。用用户的语言——你越是用用户的原始词汇，你越接近真实需求。

**第三步：画纵轴——优先级分层**

纵轴分三行，从上到下：

- **第一行（最高）：** 如果没有这个，这个版本就没有意义。这就是MVP——用户能完成最核心的价值闭环的最少功能。
- **第二行（中间）：** 做了这个版本，产品会明显更好。重要但不生死攸关。
- **第三行（最下面）：** 以后再做，不影响核心价值——nice to have。

每个行为便利贴，团队讨论它应该在第几行。讨论的焦点是：用户没有这个步骤，他还能不能完成他想做的事？

**第四步：标出用户痛点与惊喜点**

用两种颜色的便利贴覆盖在行为卡片上：
- **红色便利贴（痛点）：** 用户在使用中觉得麻烦、卡顿、困惑、生气的地方。比如"每次要输入长达20位的订单号"——红色。
- **绿色便利贴（惊喜点）：** 用户会觉得"哇，这个体验太好了"的地方。比如"扫个码订单信息自动填入"——绿色。

红色最多的区域就是最高优先级——先解决最痛的地方。

**第五步：画依赖关系线**

在两张便利贴之间画一条线，标注它们之间的依赖关系。比如"提交订单"这张便利贴依赖"登录"这张便利贴——这条线上就标注"提交订单依赖登录：如果登录方式变化，提交订单的数据接口需要变"。

依赖关系线的目的是找出：哪些需求之间有时间先后和协作关系。两个需求由不同的人做，接合处在哪里？谁负责对齐？

**第六步：评估风险**

对每条依赖关系线做风险评估。三种标记：
- **绿色线（低风险）：** 接口清晰。两边都明确知道对接方式。
- **黄色线（中风险）：** 接口存在但不稳定。还有不确定的地方。
- **红色线（高风险）：** 接口模糊。双方对彼此的理解不一致，或者还没有讨论过怎么对接。

高风险接口的用户故事在迭代规划时必须优先安排——它们需要在早期被验证，不能等到集成阶段才发现对不上。

### 为什么这么执行：

传统的需求列表有一个根本问题：它把需求拍平成一个清单，丢失了两个最关键的信息。第一，需求之间的先后依赖关系——用户必须先做A才能做B，A的交付节奏决定了B的交付节奏。第二，需求之间的拼接点——两个需求由不同的人做时，它们在哪里对接、谁负责对齐。

用户故事地图用横轴保留时间顺序（用户旅程）、纵轴保留优先级、依赖关系线保留协作风险。它让团队从一个"功能列表"变成一个"用户旅程的电影剧本"——每个人都能看到用户在什么时间、什么顺序、经历什么场景。这比任何PRD文档都直观。

一张用户故事地图最有价值的不是你拆出了多少张卡片，而是你看到了卡与卡之间的拼接点在哪里。这些拼接点才是项目真正的风险汇聚处。想象一下：A团队在前端开发"提交订单"页面，B团队在后端开发"订单处理"接口。两边同时开工，各自做各自的假设——A假设"后端返回的格式是这个样子的"，B假设"前端会这样传数据"。两个星期后一对接，发现两边对不上。这就是项目中最高频、最昂贵的返工类型。

依赖关系线的评估就是在提前标注这些拼接点。你不需要预测所有风险——你只需要找出那些"如果两边理解不一致就会出大问题"的接口，在早期把它们对齐。发现接口模糊比发现需求多花了多少钱重要得多。

### 目的：

产出一张团队共享的、用户视角的、包含优先级和协作风险标注的项目全景图。

### 重点：

- 没有真实用户参与的用户故事地图是内部视角的"猜用户想要什么"——这是信息失真的源头。不要用"我们就是用户"来回避，你不是。你是开发者，你天然地看到的是"怎么实现"，而用户看到的是"什么有价值"。
- 依赖关系线是最容易被跳过的环节——大家忙着贴便利贴，不愿意停下来分析接口。但它恰恰是整张地图最有价值的部分——它告诉你风险藏在哪里。宁愿少贴十张功能便利贴，也要把依赖关系线画清楚。
- 用户故事地图不是一次画完就不变了。每个版本发布后，用不同颜色的便利贴标注"用户实际行为与预期不符的地方"——下一版本规划时优先看这些。
- **跨行业适配：** B2B工业品应改为"用户任务分析图"（终端操作工+采购决策者+维护团队三类用户的角色分析）；硬件产品需增加物理约束列（功耗/尺寸/散热）；建筑行业应改为"价值流图"。接口依赖线概念所有行业通用。

## 步骤2.2 迭代节奏设计——找到团队的最优频率

### 怎么执行：

**第一步：回答三个问题决定迭代长度**

不默认"两周一个迭代"。用三个问题来决定：

1. **需求变化的频率是多少？**
   - 需求每周都在变（客户每周有新想法）→1周迭代。太长的话等你做完迭代一半需求可能又变了。
   - 需求每两周调整一次→2周迭代。稳定性和响应速度的平衡点。
   - 需求相对稳定，一个月才调整一次→3到4周迭代。给团队更多深度工作时间。

2. **团队的反馈饥饿度是多少？**
   - 新人多的团队需要更频繁地确认方向对不对→1周迭代，快速纠偏。
   - 成熟团队对方向判断更有把握→2周迭代没问题。更长的周期也可以。
   - 关键规则：如果迭代结束时Retro大家说不出"这个迭代比上迭代好在哪"→可能迭代太短，还没做出有意义的进展。

3. **仪式时间占迭代总时间的比例是否健康？**
   - 把所有仪式加起来（Sprint Planning + Daily Scrum x 天数 + Review + Retro），算总数。这个总数不能超过迭代总时间的15%。
   - 比如一个2周迭代（160小时可用时间），仪式时间应该不超过24小时（平均每天1.7小时）。如果超过这个数→要么迭代太短，要么仪式太冗长。

**第二步：固定仪式时间窗口**

决定了迭代长度之后，把仪式时间固定到日历里。例如：
- 每周三上午9:00-10:30：Sprint Planning
- 每周三下午16:00-17:00：Sprint Review + Retrospective
- 每天上午9:15-9:30：Daily Scrum

固定到日历是不允许挪动的。挪一次就会挪第二次。固定的仪式节奏本身就是一种稳定器——团队知道什么时候该做什么事，不需要每次临时协调。

**第三步：前3个Sprint标记为节奏校准期**

前3个Sprint不做绩效评估，不做速度比较，不做团队排名。只做一件事：观察节奏是否合适。

3个Sprint结束后，做一次专门的"节奏Retrospective"。讨论一个核心问题：
"我们现在的迭代节奏（长度和仪式）让团队感觉是'刚刚好'、'太赶了'、还是'有时候不知道该干什么'？"

根据回答决定是否需要调整迭代长度。

### 为什么这么执行：

迭代长度的本质是"团队多快能发现自己偏航了"。太短的时候——你还在规划阶段没回过神来又要评审了，团队永远在做表面功夫，深度工作时间被会议切割得支离破碎。最典型的症状是：开发人员感觉每天不是在开会就是在准备开会，一个完整的功能需要跨越三个迭代才能做完。

太长的时候——从偏航到被发现之间的时间窗口太大，等到发现问题时已经偏了很远，修正成本已经很高。就好比开车：你每100公里才看一次导航，如果一开始就走错了方向，等发现时已经多跑了50公里冤枉路。如果每5公里看一次导航，走错路时最多多跑2-3公里就能纠正。

合适的迭代长度是：信息积累到一个"可以被发现但还没造成伤害"的程度时，正好是迭代结束的时候。这个最优值不是计算出来的，是试出来的。每个团队的"最优节奏"不一样，跟你团队的成熟度、项目的复杂程度、需求的变化频率都有关系。所以前3个Sprint必须是节奏校准期——你要用真实数据来感受，而不是用理论推导。

迭代的固定节奏本身就是一种稳定机制。为什么？因为人类讨厌不确定性。如果每天都在变——"今天开会吗？不知道""明天迭代结束吗？看情况"——团队会把大量精力消耗在"协调"上，而不是"做事"上。当节奏固定下来，团队可以提前安排自己的工作节奏，在会议和深度工作之间找到平衡。

### 目的：

找到团队在"足够稳定来完成有意义的工作"和"足够频繁来及早发现偏差"之间的最优平衡点。

### 重点：

- 不要一开始就定死迭代长度。标准答案"2周"对很多团队不是最优解。有些1周更好，有些3周更好。
- 仪式时间占比超过15%——是会议过载，不是敏捷。立刻精简。精简的方法不是取消仪式，是缩小参与范围：不是每个人都必须参加每次会议。Planning核心开发者必须参加，设计师可以在异步查看；Review邀请干系人和用户参加，开发者在完成关键任务后参加。
- 如果一个迭代结束Retro时，团队说不出"这个迭代比上个迭代在哪方面更好"——可能迭代太短，还没来得及深入就结束了。延长迭代长度试试。
- **跨行业适配：** 如果反馈周期>1个月（硬件/建筑/医药），改为分层频率：
  - **物理层：** 里程碑驱动（月/季度），关键节点做gate review。
  - **逻辑层：** 仿真/验证驱动（周），迭代的是设计和验证，不是实物。
  - **管理层：** 月度复盘，检查假设是否仍然成立。
  对于纯软件之外的行业，前3个"Sprint"改为前1-2个"阶段"作为节奏校准期。

## 步骤2.3 团队能量预算——不只算人天，算能量

### 怎么执行：

**第一步：打开日历，做时间审计**

把团队过去两周的日历打开，统计每个人的时间分配。不需要精确到分钟，大致分类即可：

- **核心产出时间：** 写代码、做设计、写文档、做调研、写测试。这些是直接产出的活动。
- **会议时间：** 所有会议，包括站会、评审、讨论、1对1。
- **中断处理时间：** 临时需求、线上问题、同事求助、各种"紧急"但不属于迭代计划的事情。
- **空白时间：** "不知道接下来该做什么"的时间——需求没有澄清到位，任务没有分配清楚，在等安排。

如果"核心产出"占比低于60%——团队的能量正在被大量无效消耗。规划阶段的首要任务不是排更多功能，是清理这些消耗源。

**第二步：做团队能量五维评估**

每个维度给出红黄绿评级：

1. **时间余量（绿/黄/红）**
   - 每个迭代是否有20%的缓冲时间不被任何任务占用？绿。
   - 勉强留了10%的缓冲，但第一个需求变更就吃掉了。黄。
   - 迭代塞得满满的，一个气泡都塞不进去了。红。

2. **关键人备份（绿/黄/红）**
   - 每个关键技能领域（前端、后端、测试、运维）是否有至少一个人能接手紧急工作？绿。
   - 部分领域有备份，但后端只有一个人懂数据库迁移逻辑。黄。
   - 完全单点依赖——某个人请假，某个领域的工作就得停下来。红。

3. **学习缓冲（绿/黄/红）**
   - 新工具、新技术、新业务领域的学习时间是否明确纳入了规划？绿。
   - 口头说要学但迭代里没有留时间——"边做边学"。黄。
   - 完全没考虑学习时间，默认"他们应该已经会了"。红。

4. **信任存量（绿/黄/红）**
   - 跨团队或跨部门的接口人之间是否已经有过面对面合作经验？绿。合作过的团队效率比没合作过的高40%以上。
   - 认识但没合作过，或者只在线上沟通过几次。黄。
   - 完全不认识，第一次合作就是对接接口。红。

5. **注意力余量（绿/黄/红）**
   - 每人专注一个项目，没有额外干扰。绿。
   - 有人同时做两个项目。黄。
   - 有人同时做三个以上项目。红。

**第三步：红色维度优先修复**

任何评级为红色的维度，在下一个迭代中优先处理。修复一个红色维度比新增一个功能优先级更高。

为什么？因为红色维度不会自动变好——它只会随着时间推移恶化。信任问题会变成沟通摩擦，学习缺失会变成技术债，单点依赖会在那个人请假时变成灾难。不修复红色维度就排新功能，相当于地基裂缝上盖楼层。

**第四步：整张评估表贴在团队墙上**

把五维评估结果做成一张表，贴在团队墙上。每次迭代规划时先看一眼它——本周期的红色项有没有改善？有没有新的红色项出现？

### 第五步：用团队匿名投票校准评估——防止"安全废纸"

前面四步有一个隐蔽的风险：如果做评估的人（通常是你，项目经理）本身缺乏对团队真实状态的感知能力，这张五维评估表可能被填成全绿的"安全废纸"。你以为团队能量很足，实际上成员在暗地里透支。这不是项目管理工具失效了——是采集工具的人没有触达真实。

解决这个问题的方法不是让项目经理"更有同理心"（那不可训练），而是把评估的数据源从"项目经理一个人的判断"改为"项目经理的初判 + 团队匿名投票的交叉验证"。具体操作：

在每个迭代的回顾会上，增加一个5分钟的匿名投票环节。发5张电子投票卡（或纸质便利贴折起来投进盒子），每张对应一个维度：

1. **"上周我有至少两次连续90分钟不被打断的工作时间。"** ——对应时间余量。同意/不同意。
2. **"如果核心模块的负责人明天请假一周，有另一个人知道该怎么做。"** ——对应关键人备份。同意/不同意。
3. **"这个迭代我学到了至少一件对工作有帮助的新东西。"** ——对应学习缓冲。同意/不同意。
4. **"跨团队对接时，我不需要反复确认对方是否理解了我的需求。"** ——对应信任存量。同意/不同意。
5. **"除了这个项目，我同时参与的其他项目或临时任务没有超过一个。"** ——对应注意力余量。同意/不同意。

统计每个维度的"同意"比例。**如果某个维度同意率低于60%，无论你项目经理自己的初判是什么颜色，该维度自动降一级：你判绿→实际黄，你判黄→实际红。** "自动降级"这条规则的价值在于：它去掉了项目经理自我说服的空间。你不能说"我觉得还好吧"——60%的同事告诉你不好，数据说了算。

同时，这张投票数据直接成为五维评估表上的"团队校准线"一栏。团队墙上贴的评估表变成五列：维度 | 项目经理初判 | 团队同意率 | 校准后评级 | 本迭代行动。这给了所有人一个透明的对照——项目经理看到的和团队感受到的，差距有多大。

**如果连续两个迭代团队投票同意率持续走低但项目经理初判一直是绿——这不是评估系统坏了，是项目经理的感知系统出了问题。** 这时需要做的是把评估权部分转移给团队：让团队成员轮值在迭代规划时做初判，项目经理只做审核。

### 为什么这么执行：

为什么人力充足、预算到位，项目还是推不动？因为传统规划只算了看得见的资源——人天、预算、工时。没有算那些看不见但同样关键的东西——团队注意力、信任存量、学习空间、犯错需要的时间。这些看不见的"隐性能量"才是真正产出的前置条件。

就像一个水库——你在计划灌溉面积时不能只算放出去的水，还必须确保水库本身还有蓄水量。如果水库已经快干了，你计划再多的灌溉面积也没有意义。当团队所有的精力都已被当前任务锁定时，任何意外——需求变更、关键人请假、技术难点——都会直接导致交付崩塌，因为没有余量来吸收冲击。

五个维度中的"信任存量"是最被低估的。两个没有合作过的团队被安排做接口开发，前2到3个迭代的效率天然比合作过的团队低40%以上。这不是态度问题，不是能力问题——是信任需要时间才能建立。两个陌生人第一次配合，光是理解对方的沟通习惯、工作方式、判断标准，就需要好几周。你在规划中不给这个时间，它就会以隐形的方式消耗你的计划——表现为大量隐性返工和接口处的沟通成本。团队能量预算就是把这些"正常但容易被忽略的消耗"提前计算进去。

"红色维度优先于新功能"——这是最难让Product Owner接受的一点。"先不排功能，先修团队能量"听起来像是在浪费时间。但设想一下：一个轮胎已经漏气的车，你给它加再多油也跑不快。先把轮胎补好，再考虑加多少油。

从可持续性的角度看，团队能量预算做的不只是"保证这个项目能做完"。它在保证"做完这个项目之后，团队还愿意、还能够做下一个项目"。PMBOK第8版把可持续性列为独立原则，但在敏捷项目中，可持续性不是一个需要额外管理的维度——它就是团队能量预算本身。

具体来说，能量预算中的"时间余量"对应着工作节奏的可持续性，"关键人备份"对应着知识的可持续性，"学习缓冲"对应着能力的可持续性，"信任存量"对应着协作关系的可持续性，"注意力余量"对应着决策质量的可持续性。这五个维度中任何一个亮红灯，都不是只影响当前迭代——它在透支团队的未来。

### 目的：

确保在排任何新功能之前，团队的能量储备处于健康水平。

### 重点：

- "信任存量"这一项是最被低估的维度。两个没有合作过的团队做接口开发，前2到3个迭代的效率和合作过的团队差距巨大。在团队能量预算中直接给出额外的缓冲期，不要期待他们天然就能高效对接。
- 红色维度优先于新功能——这是最难让Product Owner接受的一点。"先不排功能，先修团队能量"听起来像是在浪费时间。但一个团队能量多个维度亮红的团队，排再多功能也完不成。先止血，再打仗。
- 时间审计的结果不要用来追究谁不够忙——那不是目的。目的是发现"我们的时间都去哪儿了"从而系统性地去除浪费。

- **感知落差警报——"镜像时刻"**：当项目经理和团队对项目状态的感受出现系统性偏差时，"镜像时刻"专题可以把两张画面拼在一起。

  触发条件必须同时满足两条：项目经理的初判与团队匿名投票结果在2个以上维度出现跨级倒挂——你判绿（"+2"或"+3"）但团队校准结果为红（"-2"或"-3"）；且连续2个迭代出现同样的跨级倒挂。不满足任意一条，就只是偶发偏差，不需要启动。

  触发后在下一次迭代回顾会中插入15分钟专题，按以下脚本引导——

  **安全开场（30秒）：** SM先说一句话——"这个环节不讨论谁对谁错，不纳入任何绩效记录，唯一目的是把两张画面拼在一起。"

  **PM先写（2分钟）：** PM在白板上写下"我看到的信号"。用自己的观察列出项目状态判断，用事实描述而非评价。比如"这个迭代全部用户故事完成了、测试通过率在95%以上、团队没有人加班"，而不是"我觉得项目状态很好"。

  **团队匿名写（4分钟）：** 每人一张便利贴，匿名写下"我希望PM知道的一件事"。不是吐槽PM，是写"PM不知道但对项目重要的事"。比如"我们迭代中期发现接口设计有重大缺陷，为了不破坏进度线，团队自己加班重做了一遍。我们选择不说是因为知道PM一定会让我们暂停，而我们不想延期。"——这条信息不是指责，但它点明了信号不同步的根源。

  **交叉理解（4分钟）：** PM把全部匿名便利贴读一遍——不辩解、不追问来源、不反驳。读完说一句话："从这些便利贴里，我看到了______。"然后团队讨论：为什么PM看到的信号和团队实际感受之间差距这么大？

  **产出共识与行动（4分半）：** 团队共同写下三个问题的答案：(1)我们想要的状态是什么？(2)我们现在实际的状态是什么？(3)缩小差距的最小一步是什么？最后一条就是行动项。

  核心原则：不计入绩效评价，不用于追责，不证明谁对谁错。镜像时刻唯一证明的是两张画面都是真实的——PM看到的是PM座位上的画面，团队感受的是工位上的画面，两个都对，只是位置不同。镜像时刻不判断谁对，它只把两张画面拼在一起。

  后续跟进：行动项写入下一个迭代的Backlog，分配责任人。下一个迭代的Retro用5分钟回访——做了没有？效果怎么样？差距缩小了没有？回访本身也是闭环，不是做了就完了，是持续校准。

- **跨行业适配：** 硬件/制造行业增加维度：设备可用性（仿真服务器/测试台架排队时间）、供应商容错能力（关键物料供应商的交期波动缓冲）、工艺余量（良品率爬坡期的产量损失预备）。

## 步骤2.4 风险预映射——不是列风险清单，是画风险场

### 怎么执行：

**第一步：不做风险清单，做风险场地图**

传统风险管理的第一件事是列风险清单：列出所有可能出错的点，然后标概率和影响。这种做法最大的问题是——它假设风险是孤立的、独立的。事实是，项目中真正致命的从来不是单一风险事件，而是多个风险在时间上叠加共振。

正确做法：在一张白纸上从左上到右下画一条时间线（迭代1到迭代N）。

**第二步：标注风险源**

在时间线的上方和下方标注风险源：

- **时间线上方（外部风险源）：** 客户变更、市场变化、合规要求变化、供应商出问题、行业政策调整、竞品突然上新功能。
- **时间线下方（内部风险源）：** 关键人离职、技术不确定性、架构决策的长期影响、团队冲突、进度压力导致的质量妥协、估算失误。

每个风险源写一张便利贴，贴在时间线旁边。

**第三步：画可能性曲线**

对每个风险源，你觉得它在这个项目周期内什么时候最可能爆发？画一条曲线。比如：
- "关键人离职"的可能性曲线可能在项目中期（迭代4到6）达到顶峰——项目最忙的时候，团队承受的压力最大，也是离职风险最高的时候。
- "客户变更需求"的曲线可能在项目初期（迭代1到3）最高——客户还没看到实际可用的东西，容易提出各种修改。

**第四步：画传导路径——红色虚线**

这是整张图最关键的一步：用红色虚线标出"如果A发生，如何触发B"。比如：

- 架构决策失误（在迭代2）→ 技术债积累 → 修改成本指数上升（迭代4到5）→ 进度延迟 → 客户不满 → 干系人施压 → 团队加班 → 更多离职（迭代6到7）。

这条传导路径告诉你：看起来在迭代2做的一个小决策，可以在迭代7引发全面的危机。风险场地图让你一眼看到这些传导链。

**第五步：设定早期信号**

对传导路径中最上游的风险源——也就是最根本的原因，不是最末端的症状——设定早期信号。早期信号必须具体、可观察。例如：

- 不是"技术架构有问题"——而是"架构评审被推迟了第二次，评审会议的参与者减少到只有两个人"。
- 不是"团队压力太大"——而是"连续两个Sprint的Planning中没人主动说话，都在等PO分配任务"。
- 不是"客户不稳定"——而是"同一个用户故事在三次Sprint Planning中出现，且每次的表述内容都不一样"。

**第六步：风险场地图上墙**

把这张图挂在团队工作区——和方向卡片、团队协议放在一起。每次Retro时对照检查：有没有新的风险源出现？有没有传导路径上的某一步正在被激活？早期信号有没有被触发？

### 为什么这么执行：

传统风险清单最大的问题是它把风险当作孤立的、互不相关的独立事件，每个风险标一个概率和影响就完事了。但真实世界中，项目里的灾难几乎从来不是由单一风险造成的——它们总是多个风险源在时间上叠加、互相放大、像多米诺骨牌一样连锁触发的结果。

一个看似微小的上游疏忽——比如一次架构评审被跳过了——可能在三个迭代后引发一串连锁反应：技术方案有漏洞→底层设计需要修改→依赖这个模块的团队只能等待→进度延迟→客户不满→管理层施压→团队加班→核心人疲劳离职。从架构评审被跳过到团队核心离职，中间隔着五六步，每一步单独看都不致命，但串在一起就是灾难。

风险场地图让你看到这些传导路径。它的设计逻辑很简单：你不必预防所有风险——那是不可能的，也没有必要。你只需要在每个传导路径的最上游设置拦截点。因为越上游的干预成本越低。在架构评审漏掉时补一次评审，成本是一天。在传导链末端的项目延期和团队崩溃时再做干预，成本是数周甚至数月。

早期信号的设定是这个方法的关键。大多数人设定信号时喜欢写一些笼统的东西——"团队士气下降""客户需求不稳定"——但这些太模糊，无法触发行动。真正的早期信号必须具体到可以观察：不是"团队士气下降"，而是"Retro时主动发言的人从8个变成3个"。不是"需求不稳定"，而是"同一个故事在三次Planning中出现且每次内容不同"。具体到谁、什么时间、什么程度，你才能在问题还没变成灾难之前就采取行动。

### 目的：

从"事后被动应对风险"升级为"上游主动拦截风险"。

### 重点：

- 最容易画错的是"把症状当作风险源"。比如"项目延期"不是风险源——它是多个风险源共同作用的结果。追问自己：什么导致了延期？那个"什么"才是风险源。
- 早期信号必须具体、可观察。不能写"团队士气下降"——而是"Retro时主动发言的人从8个变成3个"。不能写"需求不稳定"——而是"同一个用户故事在三次Planning中出现且每次内容不同"。
- 风险场地图最大的敌人是"挂在墙上了但没人看"。要形成习惯：每次Retro时花5分钟看一下它。
- **跨行业适配：** 硬件行业增加风险源类型：单一供应商断供、芯片EOL停产、EMC认证失败。建筑行业增加：地质条件变化、天气延误、政府审批延迟。这些风险的共同特征：不可通过"多迭代几次"来解决——一旦发生就是硬阻断。

## 规划阶段检查点

每一条用"是/否"回答。有一条"否"就不应该开始第一个迭代。

1. 用户故事地图上有真实用户参与过，依赖关系线已标注，高风险接口已标记。
2. 迭代长度已确定（基于需求变化频率、团队反馈饥饿度、仪式时间占比三个因素），仪式时间窗口已固定到日历。
3. 团队能量五维评估完成，所有红色维度已经有明确的修复计划。
4. 风险场地图已上墙，上游风险源的早期信号已定义。
5. 团队能说出这个项目前三个迭代各自最关注的信息失真源是什么（用日常语言：最可能导致偏差的那个点）。

# 第三部分：项目执行——把计划变成可用的东西

执行阶段的主体是Sprint迭代。每个Sprint就是一次"集中产出周期"——团队在固定节拍内产出可用的增量，并在Sprint Review和Retro时接收真实反馈。执行阶段的质量不取决于团队做了多少，而取决于每个周期中方向校准的精度。

---

## 步骤3.1 迭代启动会 Sprint Planning——每次迭代开始时的方向校准

### 怎么执行：

Sprint Planning总时长控制在70分钟以内，分成6个时间段严格执行，不能跳段，不能超时。

**开场（5分钟）：** 一个人把项目真北卡片从墙上拿下来，大声读一遍。然后问团队一句话："经过上一个迭代，我们学到了什么？方向需要调整吗？"说完之后所有人沉默5秒钟——不要跳过这5秒沉默，它让真正有想法的人有机会开口。如果没人说话，继续。

**选范围（30分钟）：** 这是Planning的核心环节。PO从Product Backlog最上面开始，依次讲每个Story的"用户价值"。注意——PO不需要讲功能细节（"这里有一个下拉菜单，选A出现B"），只需要讲用户价值（"用户完成支付后，需要立即看到支付成功的确认页面，否则他会焦虑，可能重复支付"）。讲完后团队提问，PO澄清。然后每张Story做估算。但估算不是估"这个功能需要几个人天"——是估"我们对这个Story的理解有多清晰"。团队用T恤尺码（S/M/L/XL）来回答：S代表"方案清楚、依赖明确、做过类似的东西"；M代表"整体清晰但有几个细节需要在开发中确认"；L代表"知道大概方向但技术方案还没定"；XL代表"根本不清楚这是做什么的"。L和XL的Story需要在拆任务阶段特别关注，或者标记为需要先做技术探索。

**拆任务（20分钟）：** 选定的Story逐张拆成具体任务。每个任务必须小于1天的工作量。如果一个Story拆出来的任务里有超过1天的——说明对这个Story的理解还不够清晰。要么继续拆到每个任务都能在一天内完成，要么在任务卡上标注"技术探索"，在这个Sprint中专门分配时间来做验证。

**找接口（5分钟）：** 把拆完的任务卡全部摆在一起，找出需要两个人或两个模块之间信息同步的地方。比如"前端支付页面"和"后端支付接口"之间有一个接口——前端写页面时需要知道后端接口的返回格式。在任务卡右下角画一个小三角形，标记这些接口卡。这些卡在Daily Scrum时需要优先关注。

**能量检查（5分钟）：** 算一下这个Sprint计划总任务量相当于团队产能的百分之多少。答案是——不能超过80%。剩下20%是缓冲，用来吸收意外（Bug、临时需求、接口问题）。如果PO说"能不能塞满一点，我们这次任务不复杂"——答案是"不能。塞满了意味着任何一个意外都会导致整个迭代失败。"

**产出（5分钟）：** 写Sprint Goal——用一句话说清这个Sprint完成后用户能做什么事。贴在看板最上方。同时确认Sprint Backlog——所有拆完的任务卡移入看板第一列"待办"。

### 为什么这么执行：

Sprint Planning最常见的错误是把"选范围"做成了"搬运"——直接从Backlog最上面拖几张到Sprint里就算完了。这不是规划，是仓管。真正的规划包含两个不可省略的动作：第一，重新确认方向——经过一个迭代的真实世界反馈，判断我们是否还在正确的路径上。第二，识别这个迭代的高风险点——哪些任务是理解最模糊的、哪些接口最可能出问题。

估算"不确定性"而不是"工作量"，这个转换很关键：工作量描述的是"已经知道怎么做的事情需要多少投入"；不确定性反映的是"这个任务在执行中可能产生多少意外"。一个工作量大但方案清晰的任务，风险远低于一个工作量小但理解模糊的任务。项目出问题的根源，大多不是"做得慢"，而是"做的东西不对"。而那些理解模糊的任务，恰恰是最容易做错的地方。所以Sprint Planning的第一优先级不是确保"工作量满了"，而是确保"我们对要做的每一件事有足够清晰的理解"——理解越清晰，做错的可能性越低，项目偏离真实需求的概率越小。

### 目的：

每次迭代启动时，整个团队对这个迭代的目标（Sprint Goal）和主要风险（模糊任务、接口依赖）有相同的认知，并且留下20%的缓冲空间来吸收不确定性。

### 重点：

- "方向需要调整吗"这句话如果从来没人回答"需要"——大概率不是方向真的完美，而是团队不敢提出调整。在Retro时做一次匿名投票"你觉得项目方向需要调整吗"，结果往往出人意料。
- Planning超时的最大原因不是讨论不够快，而是在拆任务时突然发现对Story的理解和PO不一致，于是开始临时讨论需求。这说明这个Story不应该被拉进这个Sprint——它应该在Sprint开始前由PO和关键开发者提前澄清。把这种场景记下来，作为下次Planning改进的第一个信号。
- 80%容量规则不是建议，是纪律。把它写在团队协议里。连续两个Sprint超了80%——说明要么估算偏了，要么PO在施压——这两个问题都应该在Retro里提出来。
- **跨行业适配：** 非软件项目80%容量规则需下调至60-70%（硬件不确定因素更多）。Planning中"估不确定性"改为"估技术成熟度"——越是新技术、新工艺、新供应商的任务，缓冲越大。

## 步骤3.2 每日站会 Daily Scrum——每天15分钟的方向微调

### 怎么执行：

每天的站会，不是各人轮流汇报，而是团队共同看"工作在流动吗"。具体操作如下：

站在看板前。每个人手里拿一支马克笔（物理看板）或者打开看板工具（电子看板）。

**第一步（5分钟）——从右往左看：** 先看最右边一列"已完成"前面的那列（通常是"评审中"或"验收中"）。问：这里有没有哪张卡其实已经满足了完成条件，可以直接移过去？有——马上移过去。再看中间的"进行中"列。问：哪张卡停在同一个位置超过24小时了？这些是阻塞的卡。最后看最左边的"待办"列。问：这个Sprint Goal要求的核心任务有没有没有被认领的？

**第二步（5分钟）——只讨论卡住的卡，不讨论正常流动的卡：** 一张卡如果昨天在"进行中"、今天还在"进行中"而且有人正在做——不需要讨论它。一张卡超过24小时没有移动——必须讨论。讨论的方式是"阻塞三连问"：

1. 阻塞是什么？必须精确——不是说"等测试"，而是说"等张三对支付接口的返回格式做确认，超过24小时没回复"。
2. 谁在今天之内能解除它？这个人的名字必须说出来。是团队成员自己，还是需要找外部的人？
3. 解除它需要什么支持？不是"需要更多时间"这种空话，而是"需要李四在今天下午3点前帮忙跑一次测试环境"这种具体动作。

**第三步（5分钟）——确认今天的目标：** 所有人看一遍看板，然后每个人用一句话说"我今天要做的事，会让哪张卡离'已完成'更近一步"。如果有人说的事和看板上任何一张卡都没有直接关系——说明这个人的优先级和Sprint Goal不一致，会后需要单独对齐。

### 为什么这么执行：

传统的Daily Scrum用三个问题："昨天做了什么、今天做什么、有什么障碍"。这三个问题本身没有问题，但大量团队在使用中退化了——变成了"轮流述职"。每个人的回答之间没有关联，变成了"我向Scrum Master报告工作"。更严重的是，这三个问题引导大家关注"个人"而不是"工作流"。

从右往左看板的做法，把站会的焦点从"个人产出"转移到了"工作流动"。这不是一个随意的切换——它背后的逻辑是：项目的价值不在于每个人做了多少事，而在于工作项是否在系统性地从"待办"流向"已完成"。从右往左看之所以有效，是因为它先看瓶颈——最接近"完成"的地方是价值即将兑现的地方，如果那里卡住了，前面做再多都没有用。然后再向源头追溯，看有没有更多工作在涌向一个已经过载的环节。

站会的本质是"每天清理信息阻塞"——不让任何一个堵塞信号超过24小时不被处理。一天是信息偏差积累到可以被感知但尚未造成伤害的时间窗口。超过24小时的阻塞，意味着问题已经开始在不同的任务之间交叉感染——一个接口的延迟可能导致多个依赖任务出现偏离。

### 目的：

每天用15分钟确保工作流动的管道中没有堵塞，任何发现的堵塞在24小时内进入处置流程。

### 重点：

- 如果站会变成了"每个人向SM汇报"——说明你用的已经是错误的问题了。切换为从右往左看板的做法，第一天就会感觉到不同：大家的目光在看板上，不在Scrum Master脸上。
- 如果站会经常超过15分钟——原因不是信息太多，是有深层的矛盾被反复触碰但从未解决。比如同一张卡连续三天被提到卡在测试环境上——问题不在站会，在于"测试环境为什么总是不可用"这件事从来没有人去根因解决。把这种反复出现的问题从站会中抽出来，单独开一个30分钟的问题解决会。
- 站会的质量不取决于每个人说了多少话，而取决于散会时每个人是否确切知道"今天我要让哪张卡动起来"。
- **跨行业适配：** 非软件团队的站会频率和形式可调整——硬件在样机调试期可能每天需要；在等待供应商/实验室的时段可改为每周两次。核心不变的是"从右往左看瓶颈"。分布式大团队，站会用电子看板替代物理看板。

---

## 步骤3.3 看板与在制品限制——保护团队能量的日常机制

### 怎么执行：

**第一步——设置看板列。** 最简有效的列设计是五列，最多不超过六列：待办(Backlog) → 就绪(Ready) → 进行中(In Progress) → 评审中(In Review) → 已完成(Done)。列越多管理成本越高——每增加一列，每天移动卡片的时间就增加一分，而且越细的列越容易让人纠结"这张卡到底算A列还是B列"。少于四列又无法有效显示流动状态。五列是经过大量团队验证的最优平衡点。

**第二步——设定在制品限制。** 在制品限制写在每列的最上方，用红笔圈起来（物理看板）或用醒目颜色标注（电子看板）。

- "进行中"列的在制品限制 = 团队人数 × 1。如果团队有5个人，"进行中"列最多5张卡。每人同时只做一件事。在极少数情况下可以放宽到1.5倍（比如团队有6个人但有一人同时在两个项目中），但不能再高。
- "评审中"列的在制品限制 = 2。评审是同步活动——一个人同时只能评审一张卡，而且评审需要被评审者和评审人都有空。当"评审中"列满了，开发不能继续写新代码并把卡推进这一列。

**第三步——触及在制品限制后的规则。** 任何时候，任何一列的卡片数量达到了它的在制品上限——整个团队进入"瓶颈清理模式"。规则很简单：不再从左边拉新的卡进入这一列。所有人——包括正在做开发的——放下手头的新功能，转向帮助清理瓶颈。如果"评审中"满了，开发者放下键盘，去找测试同事或PO一起做评审。如果"进行中"满了（每个人都手上有一张卡了），不再从"就绪"列拉新卡进"进行中"。

**第四步——每天记录三个流度量。** 不需要复杂的系统，一个Excel表或看板工具自带的功能就够了：

- **Lead Time（前置时间）：** 一张卡从进入"待办"列到进入"已完成"列的总时间。它衡量的是"用户从提出需求到真正拿到价值用了多久"。
- **Cycle Time（周期时间）：** 一张卡从进入"进行中"列到进入"已完成"列的时间。它衡量的是"团队从开始做到做完需要多久"。
- **Throughput（吞吐量）：** 每周完成的卡片数量。它衡量的是"团队的交付能力"。

记录这些数据不是为了漂亮的报表——是为了回答一个关键问题："我们的交付能力是可预测的吗？"一个团队如果本周完成10张卡、下周完成2张卡——这是有严重流问题的信号。一个团队如果稳定地每周完成5-6张卡——流是健康的。可预测性比吞吐量本身更重要。

### 为什么这么执行：

在制品限制是敏捷实践中杠杆率最高的管理机制，同时也是最容易被忽略和破坏的。原因很简单：它违反直觉。在大多数人的直觉里，"想完成更多，就要同时做更多"——但实际结果恰恰相反。

认知科学有一个基本常识：人脑不是电脑，不能在多任务之间无缝切换。每切换一次任务，大脑需要15-25分钟才能重新进入深度工作状态。一个同时做5件事的人，每天的上下文切换成本可以消耗30-40%的有效时间。这意味着同时做5件事的人，真正有效的工作时间可能不到全职投入一项工作的60%。这就是为什么在制品限制如此重要——它不是"不让大家干活"，而是"让大家把有限的注意力集中在最少的事情上，做一件成一件"。

但在制品限制的价值不止于此。它还有一个更重要的功能——把"瓶颈"从隐性变成显性。在没有在制品限制的团队里，流程是这样的：开发拼尽全力写代码，写完了扔给测试，测试堆积如山，但没人知道——因为开发在继续写新的代码，看起来大家都很忙。而有了在制品限制后，当"评审中"列满了，开发不能再继续写新代码，于是不得不去帮助测试做评审。这个机制强迫整个团队面对系统性的瓶颈而不是绕过它。

同时进行的任务数，直接决定了团队精力余量的消耗速度。当一个团队成员同时做5件事时，看起来产出很高，但实际上精力正在被大量消耗在切换成本上——能量没有转化为价值，而是在"换脑子"的过程中浪费了。在制品限制通过限制同时在制的任务数量，直接降低了上下文切换的频率，保护了团队的能量不被无效消耗。三个流度量（Lead Time、Cycle Time、Throughput）就是工作流动状态的体检数据——它们告诉你"能量在系统中流动的速度和稳定性"。

### 目的：

确保工作真正在流动而不是堆积，流动的速度是可预测的，并且团队的能量不被多任务切换无效消耗。

### 重点：

- 在制品限制不是建议，是规则。如果"特殊情况"总是可以破例，那么在制品限制就形同虚设。一个在制品限制被不断破坏的团队，其表现和没有在制品限制的团队没有区别。
- 不要同时追踪超过三个流度量。对于大多数团队，只追踪Cycle Time和Throughput就已经足够。每增加一个指标，团队注意力就被分散一分，而且指标之间可能互相矛盾，导致分析瘫痪。
- 物理看板和电子看板必须同步。很多团队同时使用物理白板和Jira，但两者经常不同步——物理看板的卡移到了"已完成"但Jira没更新，或者反过来。一周之后，你会拥有两份互相矛盾的项目事实。解决办法是：选一个作为"唯一的真相源"，另一个只在必要时做镜像。通常是物理看板作为真相源（因为站会上大家围着它讨论），Jira在每天站会后花5分钟同步一次。
- **跨行业适配：** 硬件/制造业的WIP有双重含义——任务级WIP（设计任务卡）+ 物料级WIP（在制半成品数量）。物料级WIP是精益生产的核心指标。如果你的项目涉及实物产出，看板上需要增加一列"等待物料"。

## 步骤3.4 持续集成——不让团队各自走向不同方向

### 怎么执行：

**第一条规则——每天合并到主干。** 每个开发者至少每天一次把代码合并到主干分支。高频团队的做法是每次提交都自动触发构建和测试。这条规则没有例外——即使一个功能开发到一半还没完成，也先把完成的部分合并进去（用特性开关或分支标记功能来隐藏未完成的部分）。

**第二条规则——主干坏了，全员停手。** 如果主干构建失败（编译不通过或自动化测试失败），整个团队的第一优先级不是做新功能，是把主干修绿。这意味着：正在写新代码的人停下来，正在做Code Review的人停下来——先去帮忙修主干。这条规则不能有例外。主干红色持续的时间越长，累积的偏差越大。

**第三条规则——自动化测试三层覆盖。** 自动化测试必须覆盖以下三层，缺一不可：

- **单元测试：** 测试每个函数/方法的单独行为。确保每个最小单元按预期工作。
- **集成测试：** 测试模块之间的交互。确保A模块的输出和B模块的输入格式一致。
- **端到端冒烟测试：** 测试核心业务路径从开始到结束的完整流程。比如"用户登录→搜索商品→下单→支付→查看订单"这条路径能否走通。

测试不是"质量保证的最后一道防线"——它更重要的功能是"信息不一致的实时警报器"。一个测试失败了，它的含义是：你刚提交的代码和系统中已有代码对"系统应该怎么行为"的理解产生了偏差。测试越早发现这个偏差，修复成本越低。

**第四条规则——流水线状态可视化。** 在团队工作区挂一台显示器，或者在团队沟通频道上设置自动推送，让每个人都能实时看到主干构建的状态——绿色（正常）、黄色（构建中）、红色（失败）。可视化不是为了好看，是为了让"主干状态"成为团队默认关注的共享信息。

### 为什么这么执行：

两个开发者在各自的feature branch上分别工作了10天，然后合并——这种情况发生大规模冲突的概率接近100%。但浪费的不只是合并那一天的冲突解决时间，更严重的是两个人各自10天中积累的对系统行为的理解偏差。开发者A在这10天中假设"支付接口会返回这样的数据格式"，开发者B也在这10天中修改了支付接口。当他们的代码合并时，这个假设差异会以合并冲突的形式爆发。

问题的本质是：两个子系统的认知在各自的独立分支上自由漂移，漂移越久，碰撞时的差异越大，修复的难度呈超线性增长。什么是"超线性"？就是不是1+1=2。两个开发者各自工作一天后合并，成本是30分钟。两个开发者各自工作十天后合并，成本不是300分钟——很可能是三天甚至更久。为什么？因为漂移带来的不仅仅是代码冲突：还包括对业务逻辑理解的差异、对命名规范的分歧、对测试策略的不同假设、甚至对"这个功能应该怎么做"的完全不同的心智模型。所有这些差异的修复成本不是1+1=2，而是1+1=4。

每日集成把认知漂移的时间窗口限制在一天以内。即使出现了偏差，也只有一天积累的认知差异，修复成本一般不超过30分钟。自动化测试是检测这种偏差的传感器——没有自动化测试，你永远不知道两个人的代码对系统行为的理解是否一致，你只能等到部署到生产环境后由用户来告诉你。

### 目的：

确保所有人对系统当前状态的认知差距不超过一天，并在偏差发生的第一时间被检测到。

### 重点：

- 持续集成最重要的不是工具（Jenkins、GitLab CI、GitHub Actions都行），而是"主干坏了所有人停下工作先去修"这条文化规则。工具可以被完美配置，但如果团队文化允许主干红一整天（"等我先把这个功能写完再修"），那么持续集成已经失去了它的核心价值。
- 自动化测试如果覆盖面太低（只测试了Happy Path），那么你只是把问题从一个地方（合并冲突）移动到了另一个地方（生产环境故障）。测试覆盖率不需要追求100%（最后20%的投入产出比太低），但核心业务路径的覆盖率必须是100%。判断标准：如果核心路径的测试全部通过，系统的主要功能应该在理论上可以正常工作。
- 延迟合并的成本不是线性增长的。两个人隔一天合并的成本是30分钟，隔两周合并的成本可能是三天。越晚合并，成本增长越快。这个逻辑适用于任何需要多人协作的事情，不只是代码。

### 执行阶段检查点

每一条用"是/否"回答。有一条"否"就需要在下个迭代优先处理。

- [ ] Sprint Planning时团队是否先看了项目真北卡片、确认方向仍然有效后再选任务？
- [ ] Sprint Goal是否用一句话说清了这个迭代要达成的用户价值？
- [ ] 每个Sprint的计划工作量是否控制在团队容量的80%以内？（留了20%缓冲）
- [ ] Daily Scrum是否从右往左看板、只讨论卡住的卡片、没有退化为轮流汇报？
- [ ] 看板的在制品限制是否在生效？过去一周有几次在制品超限的情况？
- [ ] 主干构建是否保持绿色？过去一周有几次主干红了超过30分钟？
- [ ] 接口标注过的任务（有三角标记的）是否在Daily Scrum中被优先关注？

# 第四部分：项目监控——用数据看清项目的真实状态

监控与执行不是先后关系，而是并行关系——执行是在向前推进，监控是退后一步观察。两者的张力本身就是健康的：执行推动产出，监控确保产出的方向正确。监控就是持续采集项目的真实信号，发现信息偏差的源头，在问题从小幅可纠正变成大幅不可逆之前介入。

---

## 步骤4.1 燃尽图的正确用法

### 怎么执行：

每天更新剩余工作量的统计（通常以小时或Story Point为单位），生成燃尽线。但燃尽图的价值不在那条线本身，而在它的**形状变化趋势**。每天花30秒做以下三个趋势信号判断：

**信号一——夹角是否在扩大：** 燃尽线与理想线之间的夹角在扩大，意味着信息偏差在加速积累——实际完成速度离计划速度越来越远。这个信号出现时，大多数管理者的本能反应是"催团队快点"。但正确的反应应该是问："是什么在导致偏离？"是范围增加了（PO中间加了需求）？是遇到技术障碍了（方案比预期复杂）？是团队产能低于预期（有人被其他事情分流了）？找到原因比"催快点"有价值得多——因为催只会让团队在错误的方向上加快速度。

**信号二——突然走平（连续3天以上没有下降）：** 这通常预示着一个强阻塞的存在——或者某个关键决策卡住了（比如架构方案还没定下来），或者关键人请假了，或者发现了严重的技术问题需要暂停推进。走平是比夹角扩大更紧急的信号——夹角扩大说明"还在做，只是慢了"；走平说明"根本做不动了"。这时需要立刻检查累积流图，看哪一列在变宽——阻塞就藏在那里。

**信号三——突然陡降（一天降了三天的量）：** 这是最容易被误读为"好消息"的坏信号。如果燃尽线在一天内断崖式下降，首先要警惕"假完成"——可能的情况包括：测试被跳过了直接标记完成（质量妥协）、任务被悄悄切分了但剩余工作量没有相应调整（估算作弊）、或者有一批故事被PO移出了Sprint（范围缩水）。任何一个"陡降"都需要核对实际交付物——如果燃尽线下去了但用户看不到可用的功能，这个陡降就是假的。

**结合Sprint Goal看燃尽图：** 不是所有偏离理想线的信号都是问题。如果Sprint Goal已经可以确定实现（比如核心功能已经完成，剩下一堆优化任务），即使燃尽线还在理想线之上，也不意味着Sprint失败。燃尽图是工具，不是标准。

**燃尽图和累积流图必须一起展示：** 在同一块白板上，或者同一个Dashboard页面上。分开展示会丢失最关键的信息——燃尽图告诉你"完成了多少"，累积流图告诉你"卡在了哪里"。只看燃尽图你只能知道"慢了"，看了累积流图你才能知道"哪里慢了"。

### 为什么这么执行：

燃尽图是所有敏捷图表中被误解最严重的一个。大量团队把它当成了"进度催办工具"——看到燃尽线偏了就喊"大家加把劲"，看到燃尽线贴合理想线就满意地认为"项目正常"。这两种反应都浪费了燃尽图真正的价值。

燃尽图的价值不在于"当前偏离了多少"这个绝对值，而在于"偏离的加速度"这个趋势信号。一条均匀下降的燃尽线意味着节奏稳定——团队在每个时间单位内产出相对一致。一条锯齿状的燃尽线意味着工作不断被外部干扰打断——团队处于"刚进入状态就被拉出来"的模式。一条突然走平的燃尽线意味着存在一个未被发现的强阻塞源——可能某个环节的问题已经积累到了阻塞流动的程度。

这三个趋势信号（夹角扩大、突然走平、突然陡降）分别对应了三种不同的异常状态，而且它们比"进度完成了百分之多少"这个数字早2-5天发出预警。比如夹角扩大——在燃尽图上要3-5天才能形成明显的趋势，而此时实际进度的偏移可能还不到10%，用传统的进度报表完全看不出来。

### 目的：

用燃尽图诊断项目节奏的稳定性和偏航趋势，而不是用它来催进度。

### 重点：

- 燃尽图的最大误用就是用来催进度。当管理层看到燃尽线偏了就冲进团队说"加把劲"——团队很快学会一个对策：在迭代中期偷偷修改剩余工作量的估算，让燃尽线看起来更贴近理想线。这叫"优化数字而非优化现实"——团队学会了让报表看起来漂亮，但项目的真实状态没有任何改善，反而因为数据失真导致决策者离真相越来越远。燃尽图的正确用法是诊断——它告诉你"出了什么问题"，而不是"谁做得不够好"。
- 燃尽图对"质量妥协"是完全盲的。一个团队可以连续每天标记3张卡"已完成"，但每张卡都有一堆隐藏Bug没有修复、有一堆自动化测试没写，燃尽线看起来完美无缺。但产品已经变成了一个纸糊的房子——外表看着完整，一碰就塌。所以燃尽图绝不能单独使用。必须和以下数据一起看：代码审查通过率、自动化测试通过率、Bug reopen率。任何一个数据异常，即使燃尽线完美，也需要深挖。
- 如果一个团队的燃尽图永远完美地贴着理想线——这个概率比中彩票还低。更大的可能性是：估算被"动态调整"了，或者团队为了保持燃尽图好看而牺牲了诚实。这是最危险的信息失真——数字被优化了，但现实没有改变。
- **跨行业适配：** 如果任务无法拆解为Story Points（硬件/建筑），改为里程碑趋势图（Milestone Trend Analysis）：横轴是报告日期，纵轴是预计完成日期。如果一条里程碑线在持续向右移动，说明延迟在积累。再结合挣值管理（EVM）的实物完成百分比（不是任务数百分比）做交叉验证。这种替代方案不依赖Story Point估算，而是基于可验证的实物进度——样机完成度、施工面积、测试用例通过率——来判断项目真实状态。

## 步骤4.2 累积流图——工作到底卡在哪了

### 怎么执行：

每天记录看板上各列的卡片数量，累积成一张累积流图。图的横轴是日期，纵轴是卡片数量。不同颜色的色带代表不同的看板列——最低层是"已完成"，往上依次是"评审中"、"进行中"、"就绪"、"待办"。色带的宽度代表某一列当前的卡片堆积量。

不做复杂的数学分析，只盯着两个核心信号：

**信号一——某个色带持续变宽（超过3天）：** 说明这个列是当前的瓶颈。比如"评审中"的色带连续4天在变宽——意味着卡片在这里排队等待评审。这时很多管理者会去催"加快评审"，但更好的做法是问两个更深的问题：第一，为什么有这么多卡片同时涌入了评审环节？是开发端没有控制好流入速度（比如在制品限制没有被遵守，开发者一下子完成了太多卡推了过去）？第二，评审环节的处理能力是否不够？是评审人时间不够，还是评审流程本身太耗时？找到根本原因后，解决方案可能是限制开发端的流入速度，而不是催评审人加班。

**信号二——所有色带都在变宽：** 这叫做"在制品全面膨胀"，是所有信号中最需要警惕的。当所有色带同时变宽——意味着团队同时在做的卡片数量在全面增加。这个场景通常发生在管理层说"大家加把劲"之后，团队为了显得在多做事情，同时启动了更多卡片——但结果恰恰相反。在制品增加导致每个人在不同任务之间频繁切换，每张卡的Cycle Time变长，所有人感觉"更忙了"但是"什么都没做完"。最终的Throughput不仅没有增加，反而可能下降。

**在Retro时用累积流图引导讨论：** 把累积流图投影出来，不给任何解读，先问团队一句话："你们觉得这张图在说什么？"让团队自己观察、自己解读。这个过程比任何分析报告都更有价值——因为团队自己发现的问题，他们会更主动地去解决。

### 为什么这么执行：

燃尽图和累积流图是互补的。燃尽图回答的问题是"还剩多少时间来完成目标"，累积流图回答的问题是"工作在哪里堵住了"。只看燃尽图就像开车只看油表不看路况——你知道还能跑多久，但不知道前面有没有堵车。只看累积流图就像只看路况不看油表——你知道哪里堵，但不知道你还有多久能到目的地。

累积流图是团队能量在各个工作阶段流动状态的可视化。当"评审中"的色带变宽时，意味着大量的精力已经被投入到了开发阶段（开发完成了这些卡），但在评审阶段被卡住了——能量已经投入但还没有转化为"已完成"的结果。能量卡在半路的时间越长，它被浪费的比例越高（因为上下文切换、重复理解、等待中的遗忘）。累积流图让你能够看到团队能量在哪里被消耗但没有产生产出——这就是为什么它比燃尽图更能揭示"问题出在哪里"。

### 目的：

识别项目中的流程瓶颈，在瓶颈导致团队系统性效率下降之前及时干预。

### 重点：

- 累积流图需要至少两周的连续数据才有分析价值。三天的图除了告诉你"我们在记录数据"之外什么也看不出来。刚开始使用时，先积累数据，不要急于做分析判断。
- 发现瓶颈之后最常见的错误反应是"给瓶颈环节加人"。如果测试是瓶颈就加一个测试人员——这听起来合理，但实际上通常让事情更糟：新人需要老人花时间带、沟通渠道增加、工具和环境配置需要时间。老人指导新人的时间实际上减少了瓶颈环节的处理能力。在大多数情况下，更有效的做法是"限制流入瓶颈的速度"——严格遵守在制品限制，不让开发把更多卡片推给已经过载的测试。堵住出水口不如先关小进水口。

## 步骤4.3 升级版挣值管理——不只测量"做了多少"，还测量"信息有多准"

### 怎么执行：

传统的挣值管理计算两个指标：进度绩效指数（SPI）和成本绩效指数（CPI）。这两个指标是有价值的，但它们只测量了"表面层"——做了多少事、花了多少钱。本手册增加两个新的指标，四个一起用。

**进度绩效指数和成本绩效指数：** 照常计算，方式不变。进度绩效指数 = 挣值 / 计划值。大于1.0表示进度超前，小于1.0表示进度落后。成本绩效指数 = 挣值 / 实际成本。大于1.0表示成本低于预算，小于1.0表示超支。

**信息准确度：** 信息准确度 = |周报中的完成百分比 - 燃尽图实际完成百分比|。采集方式：每周末请各模块负责人用一句话真实地汇报"完成百分比"。同时从燃尽图中提取实际燃尽的百分比。两个数字的差的绝对值就是信息准确度的反向读数——差越大，说明信息越不准。

- 差值 < 0.10：正常。汇报和实际基本一致。
- 差值在0.10-0.15之间：黄色预警。部分信息可能正在被美化或过滤。
- 差值 > 0.15：红色预警。要么有人在刻意美化数据，要么负责人真的不知道实际进度——两种情况都危险。
- 差值持续上升趋势（连续三周上涨）：管理失真正在系统性扩大。决策者看到的世界和真实世界的差距在拉大。

**团队能量水平：** 团队能量水平 = 当前迭代的实际产出 / 过去3个迭代的平均产出。用完成的Story Point数或卡片数来计算。

- 团队能量水平 > 1.0：团队能量处于上升期。但要注意区分——是效率真正提升了，还是团队在赶工消耗未来的精力？如果这个指标连续两个迭代都大于1.3，同时加班时间也在上升——这可能是透支而非效率提升。
- 团队能量水平在0.8-1.0：正常范围。团队能量稳定。
- 团队能量水平 < 0.7持续两个迭代：即使进度绩效指数看起来正常，团队能量已经在系统性下降。需要干预——不是"加把劲"这种干预，是"找出能量消耗源并消除它"这种干预。

**四象限汇报法：** 当你向管理层或干系人做月度汇报时，一页幻灯片展示四个指标。左侧是传统指标（进度绩效指数和成本绩效指数），右侧是领先指标（信息准确度和团队能量水平）。在讲解时用一句话解释："我们不仅看项目做了多少（左侧），也看我们获取的信息有多真实、团队的能量水平是否可持续（右侧）。"

### 为什么这么执行：

传统的挣值管理只测量表面层的东西——做了多少工作、花了多少钱。但一个项目的健康度不能只看表面层。一个进度绩效指数=1.05（进度超前）的项目，可能同时运行在信息严重失真（周报全绿但真实进度惨淡）和团队已经透支的状态下。管理层看到进度绩效指数=1.05，会继续开心地投入更多工作——直到两个月后团队能量崩了、人员开始离职、进度绩效指数断崖下跌。到这个时候再干预，成本已经是早期干预的10倍。

信息准确度和团队能量水平是两个领先指标。它们在进度绩效指数和成本绩效指数出现异常之前2-4周就会发出预警。为什么？因为信息失真和能量储备是深层变量——它们的变化发生在表面层变化之前。就像一个人：体重秤上的数字（表面）还没有变化，但精力下降、睡眠变差、食欲不振（深层）已经持续了一个月。信息准确度就是精力下降的信号——信息不再准确传递。团队能量水平就是睡眠变差的信号——团队的能量在消耗但没被补充。等到进度绩效指数和成本绩效指数也出了问题，那已经是症状外显到了体重的层面了。

四象限汇报法是为了解决一个实际问题：管理层习惯于只看进度绩效指数和成本绩效指数做判断，突然增加两个新指标可能导致困惑。把它们放在同一页上，左侧是"他们习惯看的"，右侧是"他们应该看的"，用简单的话解释两者的关系——"左侧告诉你做得怎么样，右侧告诉你还能这样做多久"。

### 目的：

在进度绩效指数和成本绩效指数显示项目出问题之前，从信息质量和团队能量两个维度发现早期预警信号，给管理者2-4周的提前干预时间。

### 重点：

- 信息准确度和团队能量水平不需要精确到小数点后两位。趋势比绝对值重要得多。差值从0.10慢慢升到0.25——这是问题，不管0.25精确与否。团队能量水平连续两个迭代低于0.7——这是问题，不管0.69还是0.71。面对管理层时，不要陷入"这个数字到底是多少"的争论，要展示趋势。
- 信息准确度的采集面临一个深层悖论：在信息失真严重的组织里，这个指标最容易失效。因为周报的完成百分比往往是按领导期望"倒推"出来的——"领导希望听到85%，我就报85%"。同时燃尽图的剩余工作量也容易被团队在迭代中期"动态调整"——在冲刺结束前三天，把剩余时间从40小时改成20小时，让燃尽线看起来在正常轨道上。如果两处数据源都存在人为美化，那么|周报完成% - 燃尽图完成%|算出来的差值可能依然很小——两个都假但假得很一致，你会看到一条漂亮的绿线，而项目实际上已经在漂移。这就叫"虚假的正常"。

**破解方法——用不可伪造的系统数据做交叉验证。** 不要把信息准确度的计算只依赖两种人工汇报数据。在每次做"信息准确度"评估时，从以下四个客观源头提取数据来校验人工汇报的可信度：

1. **代码实际合入情况：** 从版本控制系统（Git）直接提取——这个Sprint计划了多少个用户故事、每个故事对应的分支有没有真正合并到主干？不是"开发说做完了"，是"代码仓库显示分支已合并且构建通过"。
2. **自动化测试通过率：** 从持续集成流水线直接提取——本Sprint新增的测试用例数量、失败用例的修复速度。测试通过率不会撒谎——如果核心业务路径的测试一直挂着红色，不管周报怎么写，质量就没有到位。
3. **实际部署次数：** 从部署工具直接提取——本Sprint向测试环境或预发布环境部署了多少次？部署频率直接反映了代码从"写完了"到"能跑了"的真实进度。
4. **代码审查周转时间：** 从代码审查工具直接提取——Pull Request从提交到合并的平均时间。如果这个数字在持续上升但周报显示一切正常，你面对的就是系统性信息失真。

**实操建议：** 做月度汇报时，不要只报一个"信息准确度=0.12"的数值。在数值旁边列出上述四个客观指标的摘要。例如："本Sprint合并到主干的Story分支共8个（应合并12个），自动化测试通过率从87%升到91%，向预发布环境部署了4次，PR平均审查时间从6.2小时降到4.8小时。"——这一串客观数据构成了一道"防伪栅栏"。如果周报完成百分比和这四项客观数据的方向一致（周报说做完了且代码也合了、测试也过了、部署也走了），你的信息准确度就值得信任。如果周报说做完了但代码没合、测试挂了、部署没走——差值再小也是在说谎。

**与测算公式的关系：** 上述客观数据的校验不是替代|周报完成% - 燃尽图完成%|的计算公式，而是给它加上一个"可信度标签"。在四象限汇报幻灯片上，如果客观数据交叉验证通过了，给信息准确度数值打一个绿色√；如果客观数据和人工汇报差距大，打一个红色⚠——提醒决策者：下面这四个数字可能需要打折。这同时解决了采集自身被污染的问题：因为客观数据不可被人工修改，即使信息失真环境再严重，你总是有一个来自系统的锚定参照。
- 进度绩效指数/成本绩效指数 + 信息准确度/团队能量水平四象限的价值在于：让你避免单独任何一个指标的误判。进度绩效指数低但团队能量水平高——可能是遇到了一次性困难而不是系统性衰退。进度绩效指数高但信息准确度差——看到的进度可能是假的。只有四个指标一起看，才能拼出完整的画面。

- **四个系统指标的健康底线**：以下四个指标不要单独看，要配合它们的约束指标一起判断。

  **Git合入情况**配合PR规模一起看——单次PR净变更（新增行数+删除行数）不超过400行、不低于50行才计入有效统计。合并次数高但平均每次不到50行——碎片化刷量的信号，有人用小提交刷数据。合并次数低但单次超过400行——审查难度太大。

  **CI测试通过率**配合Bug重新打开率一起看。通过了但Bug反复被打开——测试只覆盖了正向路径，边界条件没测到。不稳定的测试（flaky test）禁止用CI自动重试来掩盖——一次失败就算失败，发现问题就要修复，不能让自动重试把问题隐藏起来。

  **部署次数**配合回滚率一起看。同一模块48小时内部署超过3次——暂停并说明原因。回滚率超过8%——部署前验证不足。部署成功率低于85%也值得调查。

  **PR审查周转时间**配合审查评论数一起看。审查快但零评论——走了过场。"LGTM"三个字母不构成有效审查。一个健康的审查至少要有审查者对代码逻辑提的一个追问或建议。连续两个迭代"零评论审查"超过30%，SM需要介入讨论审查质量标准。

- **度量分层提示**：四个客观校验源（Git合入、测试通过率、部署次数、审查周转）不需要每次月度汇报全部看一遍。日常站会只花2分钟看三个信号：在制品有没有超限、燃尽斜率是否正常、有没有卡住超过一天的阻塞项。每周回顾如果有异常再往下深挖。月度汇报时从四个校验源里选一个做交叉验证，轮流抽样就行。如果某个指标连续两周没有导致任何行动改变——看了它既不调整计划也不产生决策——它就是噪音，暂时停用，把注意力集中到真正驱动行动的数据上。

## 步骤4.4 迭代回顾会 Retrospective——让每个迭代都增强团队的学习能力

### 怎么执行：

Retrospective总时长控制在45分钟以内，严格五段议程。SM（Scrum Master）负责计时和引导。

**第一段（5分钟）——安全规则：** SM在开场时申明一条规则：今天Retro中说的任何一句话，不会出现在任何人的绩效评估中，也不会被记录到任何人的档案里。然后SM先分享自己在这个迭代中犯的一个错误——真实的、不修饰的错误。这个动作不是形式，它是在发出一个信号：这里是安全空间，承认错误不会被惩罚。如果SM自己不示范脆弱性，团队成员不会相信这个空间是安全的。

**第二段（10分钟）——数据回顾：** 把这个迭代的燃尽图、累积流图、Cycle Time数据投影出来。先让数据说话——团队安静地看两分钟数据。然后SM问两个问题：第一，"数据告诉我们发生了什么？"第二，"有什么是数据没有说但你在迭代中感受到了的？"第二个问题比第一个更重要——数据是客观的但有限，感受是主观的但往往包含了数据没捕捉到的信号。

**第三段（15分钟）——根因诊断：** 团队讨论以下三个问题，每个问题5分钟：

1. "这个迭代中，我们收到的信息里，哪个信息来得太晚？如果早一周知道，我们会做什么不同的决定？"（诊断信息延迟——信息来晚了，说明信息传递链路有堵塞点）
2. "这个迭代中，什么消耗了最多能量但产生了最少价值？"（诊断能量消耗源——可能是无效会议、过度设计、返工、等审批、或者重复解决同一个问题但从未根因处理）
3. "这个迭代中我们应对变化的方式，和上上个迭代相比有什么不同？"（诊断团队适应力——团队是在不断进步还是原地踏步）

**第四段（10分钟）——选一个改进项：** 团队投票选出**一个**改进项，在下个迭代落地。直接讨论三个要素：

- 具体改什么：不能说"提高代码质量"，要说"每个Pull Request必须有至少一个同行Review才能合并"。
- 谁来负责：说出具体的名字。
- 下次Retro时怎么验证：下次Retro时大家看什么数据或证据来判断这个改进确实落地了。

**第五段（5分钟）——感谢：** 每人用一句话感谢团队中的一个人做的一件具体的事。不是泛泛的"感谢大家的努力"，而是"感谢李四在上周三晚上帮我一起解决了那个支付Bug"。公开的、具体的感谢是建立团队信任成本最低的方式。

### 为什么这么执行：

Retrospective的本质不是"吐槽会"也不是"表扬会"——它是团队级别的学习能力锻炼。团队从冲击中恢复并从中学习的能力，就是靠Retro在练习。每个迭代Retro就是对学习能力的一次系统训练：团队回顾过去的工作方式，发现可以改善的地方，选一个最小的改进项落地，下一个迭代验证它是否有效。这个过程持续循环——每个迭代团队的"从经验中学习并适应的速度"就提高一点点。

为什么只选一个改进项？这是变革承受力在团队变革管理中的应用。变革承受力指的是一个团队能同时承受的改进数量上限。如果Retro选出三个改进项，同时推动三个改变——团队的注意力和能量会被分散在三个方向上，结果往往是三个改进没有一个能做到。一个改进项，真正落地了，在下次Retro中验证了，然后再选新的——这才是"持续改进"的真实含义。持续改进不是"每次改进很多"，而是"每次改进一点点，但从不停止"。

根因诊断的三个问题围绕着三个核心维度：信息传递的准确度（第一个问题——信息延迟意味着信息失真的积累）、团队能量的维持水平（第二个问题——能量消耗在不产出价值的地方意味着精力在被浪费）、团队从错误中学习的速度（第三个问题——比上次做得更好吗）。每个迭代从这三个维度做一次快速诊断，团队就能在问题积累到不可逆之前发现并处理它。

为什么要有"感谢"这个环节？因为在日复一日的迭代压力中，人们很容易只看到"还有多少没做完"，而忽略"谁做了什么让事情更好"。公开的、具体的感谢是组织场域中最良性的正向信号之一——它不花一分钱、不占用什么时间，但它传递的信号是"你的努力被看见了"。

### 目的：

每个迭代结束时，团队的学习能力比迭代开始时高一点点——在复利效应下，四个迭代后这个差异会非常显著。

### 重点：

- Retro如果变成了"吐槽大会"——说明SM没有做好安全规则的建立。吐槽本身不是问题，问题在于吐槽只宣泄情绪不产生行动方案。SM的职责是引导团队从吐槽中提炼出可操作的改进项。
- Retro如果变成了"沉默大会"——说明心理不安全感比以前想象的要严重。如果连续两个Retro大家都不愿意说真话，问题不在Retro流程本身，而在项目的环境中存在系统性的恐惧（比如有人因为说了真话而受到过惩罚）。这时SM需要把问题拿到更高级别的对话中去解决。
- 改进项选了但没有在下一个Retro验证——等于没有选。改进项选了但到了下一个迭代所有人都忘了要改什么——也等于没有选。改进项必须是一个真实的任务，出现在下个迭代的看板上，分配了工时、有在制品限制、有人负责。如果一个改进项能在迭代中无声无息地消失，说明它从一开始就没有被认真对待。
- 改进项必须SMART：具体（Specific）、可衡量（Measurable）、可达成（Achievable）、与目标相关（Relevant）、有明确时间限制（Time-bound）。"加强沟通"不SMART。但"每天的Daily Scrum从汇报式改为从右往左看板方式，连续执行一个迭代，Retro时投票比较变化"——这才是SMART。

---

### 监控阶段检查点

每一条用"是/否"回答。有一条"否"就需要在下个迭代优先处理。

- [ ] 燃尽图是否和累积流图在同一个地方展示（同一块白板或同一个dashboard）？
- [ ] 最近一个迭代中，燃尽线是否出现了超过3天连续走平或突然陡降（需排查原因）？
- [ ] 累积流图是否有超过两周的连续数据可以分析趋势？
- [ ] 信息准确度（周报完成%与燃尽图实际完成%的差距）是否小于0.15？
- [ ] 团队能量水平（当前迭代产出/过去3迭代平均产出）是否大于0.7？
- [ ] 最近一次Retro是否产出了一个SMART改进项且它已经出现在下个迭代的看板上？
- [ ] 连续两个Retro的参与度是否没有明显下降（发言人数和吐槽深度是早期信号）？

# 第五部分：项目收尾——善终比善始更难

---

## 步骤5.1 方向验证——我们不只要做完，我们要做对

### 怎么执行：

第一步，把启动阶段做的那张核心目标卡片从墙上取下来——它应该还在，如果已经不见了，本身就是第一个信号。把卡片贴在白板正中央，团队围过来，逐条逐字对照。每一条问三个问题：这条我们做到了吗？做到的标准是什么？有什么证据可以证明做到了？

第二步，邀请至少一位真实用户参与验收。不是产品经理替用户说"应该没问题"，不是领导拍肩膀说"辛苦了"。是那个真正在使用产品的人。约45分钟，你坐在他旁边看他操作。不要引导他用——让他自己摸索。问三个问题："这个功能解决了你最开始时最痛苦的问题吗？""如果满分10分，你现在打几分？""差的那几分是什么？"注意听他的措辞。如果他说"嗯……还行""差不多""挺好的"，然后停顿——那不是满意，那是礼貌。追问："如果你要推荐给你的同行用，你会怎么介绍？"如果他说不出来，那就是没有打动他。

第三步，把"团队内部发现但未修复的问题"和"用户验收时发现的差距"分开列在两个清单里。左边的清单是已知技术债，右边的清单才是真正刺痛用户的地方。两个清单的性质不同：左边是"我们知道但没时间修"，右边是"我们以为做了但实际上没做到"。右边的每一条都要标注：用户原话是什么、这个差距对用户的影响有多严重、修复的大致工作量。

第四步，写方向验证报告。格式只有四行：目标A——完全实现（证据：用户验收记录编号XXX）。目标B——部分实现（用户指出哪里差了——附用户原话）。目标C——未实现（原因：当时的假设被证伪了、资源被挪用了、或者根本是错的目标）。结论——这个项目做到了什么、没做到什么。不要写"项目圆满完成"，除非每一条都拿到了真实用户的确认信号。如果核心目标没有完全实现，它在验证报告里是加分的——诚实承认差距说明团队有判断力，写"圆满完成"反而说明团队没有理解什么是真正完成。

### 为什么这么执行：

没有方向验证的收尾，只是"我们做完了"而不是"我们解决了问题"。项目启动时团队定下了"我们要解决用户的什么痛苦"这个核心目标。项目收尾就是回去确认这个目标是否真的实现了。如果收尾时不回去问用户的真实感受，你就永远不知道当初设定的目标是否正确、是否打动了用户。团队的自我评估和用户的真实感受之间通常有显著的差距——这不是团队不努力，而是团队在开发过程中慢慢形成了自己的内部视角，对产品的问题不再敏感。真实用户参与验收的本质，是用外部视角对内部视角做一次校准。用户说"还行"不是通过——那是客气话，不是真实的认可。真正的验证信号是用户主动说"这个功能帮我解决了那个问题"，或者是用户主动提出"如果这里能改一下就更好了"——后者说明他已经把自己当成了使用者，不是旁观者。

### 目的：

为项目画上一个诚实的句号——让团队知道我们到底做成了什么、没做成什么、差距在哪里。这个句号不是为了存档，是为了让团队带着清晰的认知进入下一个项目，而不是带着"好像做完了又好像没做完"的模糊感觉结束。

### 重点：

第一，"用户说还行"不算验证通过。用户真满意的信号只有两个：他会主动推荐给别人吗？他会持续用下去吗？如果这两个答案都是否定的，那"还行"只是社交礼仪。第二，方向验证最怕的是形式化验收——发一个问卷给用户填，用户填了"满意"，你就归档了。问卷只能告诉你"用户没有公开不满"，不能告诉你"用户其实没有真正用起来"。第三，如果在验证时发现"我们当初定的核心目标本身就是错的"——这不是失败，这是宝贵的发现。把它写进验证报告，下一个项目就不会再犯同样的错误。

## 步骤5.2 经验归档——让下一个项目团队不吃同样的亏

### 怎么执行：

不做又长又没人看的项目总结报告。改用三张轻量归档，每张不超过一页A4纸。

**第一张：决策日志。** 在一张A4纸上，列出项目中做过的5到10个最关键决策。每个决策只写三行。第一行：当时面临的选择是什么——要写清楚选项A、选项B分别是什么，不要写"我们决定用微服务架构"，要写"我们当时在单体架构继续演进和拆微服务之间二选一"。第二行：我们选了哪一个。第三行（最重要的一行）：为什么选这个——当时的约束条件是什么、我们放弃了什么、有什么信息在当时是缺失但做决定后又才发现的。这一行最忌讳事后美化。不要写"我们经过充分调研后做出了正确决策"——要写"当时有两个选择，A方案成本低但扩展性差，B方案扩展性好但团队不熟悉。因为上线时间只剩三周，我们选了A，约定上线后三个月内补技术债。"三行。每行不超过两句话。5到10个就够了。一张A4纸正好装下。

**第二张：重要偏差记录。** 记录项目中发生的2到3个最重大的方向偏差事件。所谓方向偏差，就是"团队以为在做的事和实际应该做的事之间出现了显著偏离"。每个事件写四段话，每段一句话。第一句：什么时候发现偏航的——精确到迭代。第二句：怎么发现的是谁用什么方式发现的（是数据先报警——比如看板上的卡片堆积变宽了、还是人的直觉先报警——比如有人说了句"我觉得我们方向不对"）。第三句：从发现到纠正用了多长时间。第四句：如果重来一次，我们会在哪里更早干预——"如果我们在迭代2的Review时就直接给用户看原型而不是等到迭代4做完了再展示，可以早两周发现方向问题"。两到三个事件，一张A4纸。

**第三张：团队工作模式总结。** 这张纸记录的是这个团队作为一个集体的"工作模式DNA"，帮助下一个和这个团队合作的人最快上手。内容包括：什么迭代长度下这个团队最有效率（经过X周的尝试，Y周的节奏最适合我们）。什么类型的任务适合并行做、什么类型必须串行做（前端和接口可以并行，但数据库变更和现有功能迁移必须串行）。团队在什么情况下决策最快（是时间压力下还是在充分讨论后？如果是后者，安排决策时必须留出至少一天的缓冲时间）。以及一个特别提示——只用一句话写："下一个和这个团队合作的人，最应该知道的一件事是______"。比如："这个团队的后端长期处于人手不足状态，任何涉及后端的任务规划时至少要留出30%的缓冲。"或者"这个团队的习惯是下午效率最高，早上的会议尽量推迟到10点后。"

三张纸完成后，放在新项目启动时能直接访问到的地方。不是放进某个人电脑的文件夹，不是发到某个没人看的Wiki四级子页面。放在项目的根目录，第一眼就能看到。推荐的做法是：在新项目的Wiki首页加一个侧边栏链接叫"上一项目经验——决策日志/重要偏差记录/团队工作模式"，或者在新项目的Shared Drive根目录放一个名为"前置阅读-上一个团队的经验"的文件夹。

### 为什么这么执行：

归档的价值不是"存档"，是"让下一个项目启动时不吃同样的亏"。一个新团队接手旧项目的维护，或者一个新项目组建了部分老人加部分新人的混合团队，这两种场景下，决策日志和团队工作模式总结能大幅减少"从零摸索"的时间和"重复犯同样错误"的概率。传统项目总结报告为什么没人读？因为它太长、太宏观、太安全——没有人会从一个两万字的总结中提取出"后端团队长期人手不足所以排期要多加30%缓冲"这条关键信息。而三张A4纸的归档之所以有效，是因为它强制了信息密度的最大化——你只能写最重要的东西，所以每句话都是高信息量的。经验归档的本质，是把项目中的实践经验转化为可传递给下一个团队的知识。一个组织如果每个项目结束时都做经验归档，它的学习能力和恢复力会随着项目数线性增长；如果不做，这些能力只存在于离职的人的大脑中。

### 目的：

让下一个项目团队不必从零开始积累判断力。这个项目花三个月才发现的坑，下一个项目应该在一周内就知道。

### 重点：

第一，不是字数越多越好，是上下文越清晰越好。三张A4纸，写满了不删减，那比写短篇还难。好的归档是用最少的字传递最多的上下文——"后端人手不足所以排期+30%缓冲"比"考虑到后端团队的资源配置情况，建议在项目规划时预留适当的额外时间"好十倍。第二，决策日志最忌讳的是事后美化。"我们当时深思熟虑后做出了英明的决策"——这句话没有任何信息量。真实的决策往往是在信息不全、时间紧迫、团队意见不统一的情况下做出的。写出当时的犹豫、分歧和妥协——这些才是对后来者最有价值的信号。第三，归档不只为了"存档"，是为了"传递"。如果写完三张纸之后，没有人知道它的存在，那它就等于没有写。所以发布环节和撰写环节同样重要。

## 步骤5.3 团队能量恢复——项目结束后必要的休整

### 怎么执行：

项目交付后，不要立即解散团队，不要立即启动下一个项目。安排至少1到2周的过渡期。过渡期内的三件事按顺序执行。

第一件事：技术债清偿（3到5天）。把项目中那些"为了赶进度先这样后面再改"的地方，现在改掉。打开代码库或文档库，找出所有标注了TODO、FIXME、HACK的地方。按优先级排：影响线上稳定性的先修，不影响但每次开发都踩坑的其次，纯代码风格的放最后。不用全部修完，修到"核心路径上没有未处理的TODO"就行。技术债清偿不仅是技术动作，也是心理动作——团队需要一个"我们把事情做干净了"的感觉来结束项目。这个感觉比技术上的收益更重要，因为它带着正向的情绪进入下一个项目，而不是带着"还有一堆烂摊子"的愧疚感。

第二件事：学习周（3到5天）。每个人选一个和项目相关但当时没时间深入的方向——可以是项目中踩过的坑对应的技术（"这次因为不熟悉消息队列导致了好几次线上问题，这周深入学习一下"），可以是项目中好奇但没时间研究的工具或框架（"这次竞品有个功能做得很好，我想研究一下他们怎么实现的"），也可以是完全不相关但个人感兴趣的技术方向。学习周不是培训——没有课程表、没有讲师、没有考核。就是一个人自己学，每天下午4点团队花30分钟做一个内部分享。目标是每个人离开这个项目时，比进来的时候多了一个"Aha，这个我终于弄懂了"。

第三件事：团队散伙饭（半个晚上）。不是形式主义的团建——不是找一家贵餐厅、领导讲一段感谢的话、大家吃完各自回家。正确的做法是：找一个能让人放松说话的环境（不要大圆桌，要小桌或者有沙发的空间），菜上了之后，不是领导先说话，而是每个人依次回答两个问题：这个项目里你最骄傲的一个瞬间是什么？这个项目结束后，你接下来想做什么？第一个问题是让正向的情绪有机会被表达（在项目压力期大家很少有机会说"我很骄傲"），第二个问题是让每个人把注意力从"结束了"转向"接下来"。这是情感层面的能量恢复——让团队在情感上完成对这个项目的告别，而不是突然就各奔东西。

第四件事（如果团队分散到不同项目）：软交接。不搞"周五结束A项目，周一出现在B项目组"。给一周的交叠期——上午处理A项目的收尾（写文档、交接代码、回答新人问题），下午开始了解B项目的背景。交叠期的存在让"结束"和"开始"之间有一道缓冲带，而不是一个硬切换。这道缓冲带吸收的是注意力和精力的冲击——完全切换到一个新项目的大脑认知成本很高，一个交叠期可以让这个切换从"跳跃"变成"过渡"。

### 为什么这么执行：

在项目型组织中，团队能量恢复是最容易被砍掉的环节。理由听起来总是很充分——"下一个项目等着呢，别休息了，直接上"。但连续跳过能量恢复的后果在三到六个月后显现：核心成员开始频繁请假、工作效率下降、对同事失去耐心、对项目失去热情。这不是"工作态度问题"——这是团队的精力被持续消耗而没有补充的结果。当一个人的能量被持续消耗而没有机会恢复，最终会导致状态崩塌。团队能量恢复不是"放假"，它包含技术债清偿（把项目中遗留的技术问题清理干净）、学习周（让每个人在项目中积累的经验变成真正的能力提升）、散伙饭（给团队成员一个情感上的圆满收尾）。这三件事虽然没有产出任何"可见的功能"，但它们在做一件更重要的事——让团队带着清零的状态进入下一个项目，而不是带着上一个项目的残余疲劳和未处理的信息偏差。

### 目的：

让团队成员以健康的状态离开这个项目，在进入下一个项目之前完成能量的恢复和清零。

### 重点：

第一，团队能量恢复这个环节对项目经理来说是最难推行的，因为它不产出任何管理层想看到的"交付物"。说服管理层的话术是："如果我们跳过能量恢复，三个月后你会花三倍的时间来填补离职和返工的缺口。一个团队连续三个项目不做能量恢复，核心成员离职率会从正常水平的10%升到40%以上。你算算那个成本是1周过渡期的多少倍。"第二，能量恢复不是闲置。团队没有产出新功能，但做了技术债清偿、内部学习、文档整理——这些事情虽然不叫"交付功能"，但它们是"让团队有能力继续交付功能"的前提条件。第三，如果管理层实在无法接受一周的过渡期，最低底线是：至少两天。一天做技术债核心修复，半天做内部分享交流，半天做简单的散伙饭加软交接第一天。两天比零天好，两周比两天好。但零天累积三次，核心人就走了。

### 收尾阶段检查点

每一条用"是/否"回答。有一条"否"就不能算项目真正完成。

- [ ] 核心目标卡片上每一条声明的验证结果都有明确记录（做到了/部分做到/未做到）？
- [ ] 方向验证是否有至少一位真实用户参与（而不只是团队自评）？
- [ ] 决策日志是否已写好（至少5个关键决策，每个包含"为什么"）？
- [ ] 重要偏差记录是否记录了项目中2-3个最重大的方向偏差及其纠正过程？
- [ ] 团队工作模式总结是否已归档到新项目能直接访问的地方（Wiki首页/Shared Drive根目录）？
- [ ] 团队能量恢复过渡期是否已排入日程（至少两天，低于两天需管理层书面豁免）？

## 第六部分：项目维护——从冲刺模式切换到长跑模式

---

## 步骤6.1 模式切换——从冲刺模式切换到长跑模式

### 怎么执行：

第一步，定义一个明确的"模式切换标志事件"。这个事件必须是可观测的、不模糊的。比如"核心交易功能上线满30天且无严重线上事故（P0级事故为零）"。不是"产品稳定了"——太模糊。"上线满30天无P0事故"——每个人都能判断是对还是错。这个事件一触发，团队的运行节奏就应该正式切换。

第二步，调整迭代节奏。如果项目期是1到2周的迭代，维护期改为4周迭代，或者直接切换为Kanban连续流模式。Kanban连续流的含义是：去掉固定的迭代边界，不设Sprint，用在制品限制和拉式系统管理工作流入。这个改变的原因是：维护期的工作性质和项目期完全不同。项目期的需求是有明确边界的（发布V1.0必须包含这些功能），而维护期的需求是持续涌入的（用户反馈修复、小功能增强、技术债清理、安全更新）。用固定迭代的节奏来响应持续涌入的需求，要么导致迭代边界频繁被打破（每次用户反馈一个紧急问题就要开放当前迭代的边界），要么导致维护期团队的精力被持续消耗（每个迭代都为了在固定时间内塞入各种不同类型的任务）。4周迭代或Kanban连续流的好处是：它允许团队根据工作性质调整节奏，而不是被一个固定的时间盒子驱动。

第三步，审视团队结构的变化及重新画关系图。维护期的团队通常比项目期更小，可能是3到5人的核心维护团队加部分弹性支持人员。团队结构变化意味着关键人物分布变化——之前项目期的关键决策者可能已经去了其他项目，新的维护期负责人可能不是原来的Tech Lead。所以需要重新画一张简版的关系图，用三个同心圆：每天参与维护的成员放最内圈，不每天参与但能决定维护方向的人（比如产品负责人上线了产品但不做维护决策了，这个角色可能变成了"值班架构师"或"运维负责人"）放中圈，需要定期同步但不参与日常维护的人（下游系统负责人、用户支持团队）放外圈。维护期的关系图比项目期的简单，但必须重新画——因为角色的职责和影响力在维护期已经发生了变化。

第四步，切换度量体系。项目期的进度绩效指数、成本绩效指数、燃尽图——两个迭代之后这些指标就失去了意义，因为维护期的"进度"不是按照"完成了多少功能"来衡量的。切换到运营度量体系。核心指标三个：平均修复时间（平均修复时间——线上问题从发现到修复部署的平均时长）、用户满意度（通过NPS评分或用户在反馈中的关键词情感分析来测量）、系统可用性（通过在线率指标——两周达到了多少个9）。额外可以加一个季度级指标：技术债减少量（有多少个遗留TODO被清理了）。

### 为什么这么执行：

用冲刺的节奏跑长跑，团队会在维护阶段持续处于"项目压力"的状态，即使实际上没有迫在眉睫的截止日期。这种节奏错配本身的危害是什么？是团队能量的持续、慢性消耗。团队习惯了每个迭代结束前冲刺交付的节奏，在维护期没有"迭代结束"的压力下，要么不断给自己制造虚假的Deadline来维持冲刺感（自己卷自己），要么在"没有Deadline"的情况下慢慢失去节奏感（生产力下降）。这两种状态都不是健康的运营节奏。项目期和维护期的工作性质本质区别在于节奏不同。项目的节奏是脉冲式的：高能量集中在一个点（冲刺交付），然后释放。运营的节奏是稳态式的：能量持续、小幅度流动，伴随周期性的调整。工作的性质变了，管理的节奏也应当跟着变。用项目期的管理方式来管理运营期，本质上是向一个已经改变的系统输入错误的管理信号。

### 目的：

让管理节奏匹配系统当前的真实阶段——项目结束了就是结束了，不要再拿冲刺的心态来跑维护。

### 重点：

第一，模式切换最大的陷阱是名义上进入了维护但实际上所有人还在用项目期的KPI衡量团队。项目经理还在看燃尽图，管理层还在问"这个月完成了多少Story Point"，产品负责人还在用Sprint Planning的方式排需求——这些惯性力量会让维护期的团队持续处于"项目状态"而没有真正切换到"运营状态"。第二，过渡不是一步到位。第一个维护期迭代建议采用折中方案——4周迭代，但前两周作为"过渡观察期"，如果团队觉得节奏还是太快或太慢，可以在第一个月结束后再调整。第三，不要在切换后立即改变所有的流程和工具。先改节奏（迭代长度从1-2周变4周或Kanban），再改仪式（Retro从每迭代一次变每月一次），最后再改度量体系（进度绩效指数切换到平均修复时间/用户满意度）。改太快团队会失焦，改太慢团队会继续在旧模式中自动运行。

## 步骤6.2 运营期的持续监控

### 怎么执行：

建立三个层次的轻量级监控机制，每个层次的时间成本不同但信息互补。

第一层：月度快速检查。每月一次，30分钟，不要更多。团队成员围坐在一起，只看三个维度的数据。第一个维度：用户反馈中高频关键词的变化趋势。打开过去一个月的用户反馈来源（客服工单、应用商店评论、群聊记录、用户访谈记录），把用户最常说的高频词列出来。是正向词变多了（"好用""快""稳定"），还是负向词变多了（"卡""报错""找不到""慢"）？如果一个新词突然进入了高频列表——比如过去三个月用户从来没说过"支付失败"，这个月出现了三次——这是一个重要的早期信号。不需要做什么复杂的自然语言处理分析，用Excel统计一下词频，或者甚至就是把评论一条条读过去然后凭直觉写一个趋势判断，也行。第二个维度：平均修复时间的趋势。从发现到修复部署一段线上问题的平均时长。这个数字如果在变长，系统中可能存在未处理的技术债在积累——修复一个问题需要的时间越来越长了，因为代码越来越不易于理解、测试覆盖越来越低、或者部署流程越来越复杂。第三个维度：团队加班趋势。不是精确统计，而是一个简单的绿色黄色红色的判断：过去一个月内，每周平均加班天数是否超过了1天？连续两周加班？黄色警报。连续三周加班？红色警报。维护期的加班比项目期的加班更危险——项目期的加班是有终点的（发布之后就能休息），维护期的加班没有终点，所以它一旦开始就会无限持续下去。

第二层：季度深度回顾。每季度一次，一个完整的两小时的迭代回顾。沿用和项目期一样的三问框架：这个季度中，信息来源最不准确的领域是哪里（什么数据你觉得不太可信）？团队精力的最大消耗源是什么（什么东西消耗了大家的精力但产出的价值最少）？团队的恢复力是上升还是下降（相比上一个季度，团队应对变化的速度变快了还是变慢了）？不同的是，这场回顾不需要全员参加——维护期的团队超过半数参加即可。维护期的回顾不需要像项目期那样严格，但必须产出两件事：一个明确的改进项（下个季度要解决的一个具体问题），和一个"值得警惕"的信号（团队觉得需要关注但没有立刻行动的想法）。

第三层：系统问题热点图维护。这不只是一次性的产出，它需要持续维护。创建一个简单的文档，记录系统中哪些地方最常出问题、最不被团队成员理解、最容易产生信息不对称。可以是一个简单的表格：模块名称、最近一次出问题的时间、出问题的频率、问题类型（技术债回归、理解偏差、外部依赖变更、文档过时）、风险等级（低/中/高）。这个热点图是维护期的"敌人地图"：你知道哪里最容易出事，哪里最容易被忽略，哪里最值得投入有限的维护资源。热点图不需要很复杂。用一张Wiki页面维护就行。每月月度快速检查时花两分钟看一眼：这个月有没有新的热点冒出来？老热点有没有降温？

### 为什么这么执行：

维护期的监控和项目期监控的核心区别在于：项目期的监控是为了"在交付过程中发现偏航"（高投入、高强度），维护期的监控是为了"在漫长的稳定期中保持感知"（低投入、低强度）。一个系统上线后，如果你用和项目期一样的管理强度去对待它，管理成本本身就会消耗掉本来可以用来维护系统的资源。这不是在做维护，是用管理本身消耗团队——结果是磨损了团队而系统没有变好。维护期的三个层次监控之所以有效，是因为它信息互补但不冗余：月度检查管的是"有没有什么正在变坏"（趋势扫描），季度回顾管的是"我们有没有持续改善"（系统性回顾），热点图管的是"哪些地方是我们最应该关注的"（战略性注意力分配）。维护期的监控本质是"以最低成本保持对系统状态的感知，以便在恶化信号出现时及时干预，而不至于在恶化完全暴露之前对它毫不知情"。持续监控的目的不是防止所有的恶化（那不可能），是在恶化开始的时候，你能比系统崩溃早一个月感知到。

### 目的：

在漫长的维护期中保持对系统健康度的感知，防止恶化在无人注意的情况下默默积累到不可逆的程度。

### 重点：

第一，维护期的监控必须比项目期更轻量，否则就本末倒置了。如果维护阶段的管理开销和开发阶段一样大，那就不是在管理维护——是在消耗本应被保护的能量。管理开销高于实际维护工作的团队，很快就会陷入"没时间维护系统，因为所有时间都花在了汇报系统状态上"的怪圈。第二，不要过度自动化监控。一个群发的自动告警邮件如果没有人真正去看，那么它的存在只是制造了一个"我们已经有监控了"的虚假安全感。第三，系统问题热点图的价值不取决于它有多精确，而取决于它有没有被定期维护和参考。如果一个热点图在创建后六个月没有任何更新，它已经变成了电子垃圾。宁愿不要热点图，也不要一个过时的热点图——因为它会给你一种虚假的安全感。

每一条用"是/否"回答。有一条"否"就需要在下个月优先处理。

- [ ] 是否已明确定义了"项目模式切换为维护模式"的标志事件（如核心功能上线满30天无P0事故）？
- [ ] 迭代节奏是否已从项目期节奏（1-2周）调整为维护期节奏（4周或Kanban连续流）？

- [ ] 是否有月度快速检查机制（30分钟内完成，不超时）？
- [ ] 系统问题热点图在过去一个季度内是否有更新？
- [ ] 过去一个月，平均修复时间是否没有明显上升趋势？

# 第七部分：工具箱——每种工具帮你看到什么

| 工具名 | 帮你看清什么 | 什么时候用 | 什么时候不用 | 最常见误用 |
|--------|-------------|-----------|-------------|-----------|
| **燃尽图** | 帮你看到项目的完成速度是在加快还是减慢——不是看还剩多少，是看趋势。曲线向下弯说明在加速，向上弯说明在减速。趋势变化比当前剩余量早好几天发出预警。 | 每个迭代内追踪进度的稳定性。结合迭代目标一起看——如果目标已可达成，偏离理想线不一定有问题。 | 不能跨迭代比较（不同迭代体量不同），也不能用来判断质量好坏。也不要用来对比两个团队谁更高效——不同团队的估算方式不可比。 | 用来催进度。管理者指着曲线说"偏了大家加把劲"——这破坏了工具的用途。燃尽图是用来诊断偏航原因的，不是用来施压的。越催，团队越会偷偷修改剩余工作量估算，让数据变得不可信。 |
| **燃起图** | 帮你看到项目范围在变大还是变小，以及实际完成了多少。用两条线（已完成量 和 总范围）展示动态关系：总范围线上移说明范围在增加，已完成线上移说明真在推进。 | 需求频繁变化的项目。能清晰展示"范围膨胀了多少"和"完成了多少"之间的动态关系。特别适合给管理层看的汇报场景——一张图说清"为什么进度看起来没变但范围多了"。 | 固定范围的简单项目。两条线几乎平行，产生不了额外洞察，此时燃尽图更简单直接。 | 两条线不分开画——总范围线和已完成线用相同颜色或样式，让人分不清究竟是范围变了还是真的在推进。区分两条线的视觉样式是燃起图的全部意义。 |
| **累积流图** | 帮你看到工作卡在哪个环节了。不同颜色的色带代表看板的不同列。某个色带持续变宽意味着这个环节在堆积——工作流到这里就堵住了。 | 识别流程瓶颈。团队超过4人且工作流有多个阶段时，累积流图是最直观的"哪里堵住了"诊断工具。维护期监测系统健康度同样适用。 | 团队小于4人，流动路径太简单，累积流图提供不了比直接观察更多的信息。短期探索性项目也不用——数据还没积累完项目已结束。 | 数据不连续就分析趋势。累积流图需要至少两周连续数据才有意义。只记录了三天的数据就开始指着色带说"这里有瓶颈"——那是噪音不是信号。另一个误用：发现瓶颈后立即给瓶颈环节加人，而不是先限制上游流入。 |
| **看板（在制品限制）** | 帮你看到团队同时在做的任务是不是太多了。每列的最大卡片数就是"水位线"——超过这条线说明团队精力被分散在多任务切换中，实际产出反而下降。 | 运维、持续交付、维护期等需要连续流的场景。当团队不需要固定迭代节奏也能稳定交付时，看板比Scrum更合适。也适合支持类团队——他们的工作是被事件驱动的，不是按迭代排期的。 | 需要固定迭代节奏的大版本交付场景。如果每个版本有明确的截止日期和范围承诺，Scrum的迭代结构比看板更合适。新产品从零开始也不适合全看板——需要迭代的节奏感来建立反馈回路。 | 有看板但没有在制品限制。没有在制品限制的看板只是"任务列表"，不具备任何流管理能力。另一个常见误用：在制品限制被"特殊情况"反复打破，最终形同虚设。再一个：列太多（超过6列），管理看板的成本超过了看板带来的价值。 |
| **用户故事地图** | 帮你看到用户从开始到结束的完整体验，以及任务之间在哪里需要对接。横向是用户旅程的时间顺序，纵向是优先级分层，接口线标注了协作风险。 | 项目启动规划、每个大版本发布前的需求探索。特别适合需要跨团队协作的复杂项目——多个团队可以在一张地图上看到各自的依赖关系。 | 日常迭代执行追踪。用户故事地图是"地图"不是"GPS导航"——迭代执行阶段应该用看板和燃尽图。也不适合用来做详细的任务估算——那是规划阶段的事。 | 产品负责人一个人画完然后讲给团队听。用户故事地图的价值在于集体讨论中暴露出的假设分歧，不是在终稿有多好看。没有真实用户参与的地图是内部视角的猜谜。另一个误用：画完就收起来，没人再根据用户实际行为调整地图。 |
| **升级版挣值管理** | 同时看进度是否正常+成本是否正常+报告是否可信+团队是否在透支。传统挣值管理只看前两个，加上"信息准确度"（周报和实际的差距）和"团队能量水平"（当前产出和历史平均的比值）。 | 干系人月度汇报、项目预警。当管理链超过两层时，信息准确度指标的价值尤为突出——它能量化你看到的报告和实际情况之间的差距。团队能量水平在项目中期数据充足后最能发挥作用。 | 2-5人的小团队。管理开销不值得——小团队的信息失真可以通过站会直接发现。纯探索型项目也不适用——没有明确的"计划值"作为基线。 | 只看进度和成本，忽略信息准确度和团队能量水平。一个进度正常的项目可能信息严重失真、团队已经透支。等到进度指标下滑再行动，已经晚了。还常见误用：把信息准确度和能量水平算到小数点后两位当精确数据——趋势比绝对值重要，关注方向而不是精度。 |
| **循环时间** | 帮你看到从开始做到做完一件事平均要多久。循环时间越稳定，团队交付的可预测性就越高。数据是集中还是分散，告诉你工作流的稳定性。 | 持续改进和预测交付日期。有4周以上连续数据后，可以用循环时间的统计来做交付日期预测。也是识别流程改进效果的客观指标——改前改后的对比告诉你改进是否真的有效。 | 项目刚开始（需要至少4周数据才有统计意义）。前两周的循环时间主要反映"团队还在磨合"，不是正常工作节奏。团队流程发生重大变更时也需要重新积累数据。 | 追求循环时间越来越短——这可能导致以牺牲质量来换取速度。"短而稳定"比"越来越短"更有价值。循环时间的稳定性比绝对值重要。另一个误用：拿不同复杂度的工作项混在一起算平均循环时间，失去了分析意义。 |
| **吞吐量** | 帮你看到团队每周/每迭代能稳定交付多少。吞吐量的波动幅度直接告诉你团队的工作节奏是否健康——波动太大说明流程不稳定。 | 预测未来交付能力。结合循环时间的分布，可以用概率统计做交付日期预测。容量规划场景也很适合——管理层问"下个季度能做多少"时，吞吐量历史数据是最客观的回答依据。 | 团队处于剧烈变化期（人员变动、技术栈更换）。吞吐量在变化期没有统计意义的规律。单一项目的极小型团队也不适合——样本量太小，统计意义有限。 | 不同周的吞吐量直接比较但忽略了工作项大小差异。一张"集成支付网关"的卡和一张"改个按钮颜色"的卡在吞吐量里都算"1"，直接比较会误导决策。需要配合工作项大小标准化来解读。 |
| **迭代回顾会** | 帮你看到团队在持续改进中是否真的在进步。从一个迭代到下一个迭代，团队的自我修正速度有多快——能系统性地发现问题并转化为行动吗？还是每次都说同样的问题？ | 每个迭代结束、项目里程碑。每一个迭代都应有一次。遇到重大事故后也可以加开一次专项回顾——不需要等到迭代结束。 | 适用所有场景——没有"不适合做回顾"的情况。即使只跑了三天的短迭代也值得花30分钟做一次。即使项目快要解散了，做一次"收尾回顾"也比不做有价值。 | 改进项太多一个都落不了地。一次回顾选3-5个改进项，等于一个都没选——团队精力被稀释。只选一个，落地了，下个迭代再选新的。另一个误用：选了改进项但下次回顾不验证，等于没选。再一个：回顾变成了吐槽大会而没有引导到建设性的改进方案。 |
| **物理看板vs电子看板** | 帮你看到不同信息展示方式对团队沟通效率的影响。物理看板的优势是"路过就能看见"，不需要登录系统。这种低摩擦的信息可见性让团队对齐成本更低、信息传递失真更少。 | 同地团队用物理看板（信息在全员的直接视野中，更新零摩擦，站会时自然对齐）。分布式团队用电子工具（选择团队用得最顺手的一个，不要为了"标准化"强制换工具）。 | —— | 两者同时维护但不同步。一旦物理看板的卡片状态和电子工具不一致，团队就拥有两张互相矛盾的项目视图。必须选一个作为唯一真相源，另一个只在必要时做镜像。另一个常见问题：工具换得太频繁，团队刚适应一个就换下一个，每次切换都消耗团队精力。最好的工具是团队愿意用的那个。 |

## AI时代的项目管理

PMBOK第8版新增了AI在项目管理中的应用附录。本章从实操角度出发，回答一个实际问题：AI到底能帮项目经理做什么，不能做什么。

### AI能帮PM做什么（具体）

**自动生成周报草稿。** 基于看板数据（哪些卡完成了、哪些卡阻塞了）和代码提交记录，AI可以自动生成周报初稿。PM只需要核对它有没有写错，省掉整理周报的重复劳作，把精力留给更有价值的事情。

**识别燃尽图异常模式并提前预警。** 燃尽图的趋势变化往往在SPI下降前2-3天就已经显露出模式——斜率变平、锯齿状波动加大、突然走平。AI可以持续跟踪这些模式，在偏差还未被肉眼发现时就给出预警信号。

**自动汇总用户反馈中的高频关键词变化。** 来自客服工单、应用商店评论、用户访谈的信息量很大，人工筛选成本高。AI可以自动提取高频关键词并分析变化趋势——比如"支付失败"出现频率突然翻了三倍，AI可以标注出来提醒PM核实。

**模拟影响分析。** "如果延迟这个功能两周，对整体进度的影响是什么？"AI可以基于依赖关系图快速推演出影响范围——哪些下游任务会被阻塞、关键路径是否会延长。这个分析不需要精确到天，但"大概会延后一到两周"这个级别的判断，AI可以在几秒内给出。

**代码审查辅助。** AI可以识别高风险变更——修改了核心路径却没有更新测试、改了接口返回格式却没有通知下游依赖团队、删除了一个函数却没有检查调用方。这些在人工审查中容易遗漏的隐患，AI可以作为第一道防线。

### AI不能替代PM做什么

**判断方向的真伪。** 数据能告诉你方向偏了，但"偏差是因为走错了还是因为环境变了"——这个判断需要人的判断力。AI看到的是数字偏差，但理解偏差背后的原因需要PM对业务、团队、市场的综合感知。

**建立信任和安全感。** AI可以处理信息，但项目管理中最重要的信任建立、安全感维护、冲突化解——这些发生在人与人之间的微妙互动中，不是AI能介入的。

**做"不做什么"的取舍。** AI能列出所有选项和利弊分析，但做取舍的后果需要人来承担。当你说"放弃功能A"时，你在放弃一部分用户价值——这个选择带来的后果，不能让AI背锅。

**感知未被表达的恐惧。** 站会上所有人都在说"正常"，但你能感觉到不对劲——某个人连续三天没有主动发言、一个平时话多的人突然安静了。这种感觉AI没有，因为AI分析不了你没告诉它的东西。

**复原决策日志中的真实妥协。** AI无法准确复原决策当时的妥协与真实原因。使用AI整理会议纪要时，决策日志中"为什么选这个方案"的"妥协的真实原因"——比如"因为当时人力不够，只能选这个简单方案先上线"——必须由PM或团队骨干手动校准和书写。严禁让AI用"出于架构扩展性和长期战略考虑"这类官话和漂亮的总结词来替代真实的现场教训。后来者最需要的不是美化过的理由，而是"当时我们在什么约束下做了这个取舍"的诚实记录。

### 给PM的实操建议

**先把项目数据整理好。** 看板数据不准确、代码提交信息不规范、用户反馈没有系统记录——先花时间把这些基础打牢。没有干净的数据，AI给出的建议是有害的噪音。

**从低风险场景开始用。** 让AI帮你做周报草稿、汇总用户反馈关键词。不要在初期就让AI做排期决策或范围决策。你对AI的信任应该像对人的信任一样——从低风险场景开始逐步建立。

**AI的建议永远是假设，不是指令。** AI说"根据历史数据这个功能需要5天"——把它当作"有一个可能，但我还要看团队实际状态和技术复杂度再做判断"。你才是那个承担判断责任的人。

**把自己定位为"承担判断责任的人"。** AI是不知疲倦的分析助手——处理数据、发现模式、生成初稿。但每个决策的判断权在你手上。项目管理从来不是信息处理的速度游戏，而是判断质量的较量。

# 第八部分：常见场景快速诊断（5个高频问题）

## 场景1：站会开了、看板用了、迭代跑了，但团队觉得和之前瀑布没区别，就是多了几个会

**症状：** 团队在执行"敏捷流程"但没有敏捷思维。站会是轮流汇报"昨天做了什么今天做什么"——和瀑布的晨会没有本质区别。规划会是产品负责人分配任务而不是团队承诺目标。回顾变成了吐槽大会但没有后续行动。形式在跑，但实质的协作方式没有变。团队觉得"敏捷就是多开了几个会"——这种感受本身就是最强烈的预警信号。

**诊断：** 项目方向没有被团队真正理解和接受。团队仍然在等别人告诉"做什么"，而不是根据对项目目标的理解去判断"应该做什么"。核心问题不是流程执行不到位，而是团队没有获得自主决策的框架——他们缺少的不是流程，是一个共同认可的指南针。当团队在做决策时不是先看项目目标再判断，而是先问"产品负责人想让我做什么"时，敏捷就退回了瀑布。

**第一步行动：** 下次迭代规划时，先不动需求列表。拿出你们在项目启动时写的那张"项目目标"卡片，问团队一个问题："这个迭代结束时，用户能用我们的产品完成一件什么他们现在做不了的事？"如果团队讨论这个话题时的状态和讨论任务分配时完全不一样——更投入或更沉默——都说明同一件事：项目目标没有被共享。更沉默说明团队不敢说"我不确定"。更投入说明他们渴望但缺乏这样的对话。下一步：花一个下午重新校准项目目标，让每个人亲手修改，而不是被动接受。

---

## 场景2：周报全绿，但直觉告诉我项目偏了

**症状：** 所有正式指标（燃尽图、Bug计数）都在合理范围内。燃尽图贴着理想线，Bug数量甚至比预期的还少。但你总觉得哪里不对劲——可能是一种说不清的"安静"，也可能是一些零散的信息拼在一起不对劲："听说那个模块的设计方案在团队内有很大分歧"和"测试说那个模块的代码质量不太好"，可这两件事都没有出现在任何正式报告里。

**诊断：** 信息的失真度正在升高。正式报告渠道的信息和项目真实状态之间的差距在扩大。坏消息被逐层过滤掉了，只剩下好消息向上传递。周报的绿色不是原始数据的颜色，是经过多层润色之后的颜色。层级越多，过滤越严重。每层传递的人都有动机不让坏消息在自己手上"爆炸"，所以会包装、淡化、推迟上报。那个"直觉"不是玄学——是你作为经验丰富的管理者，潜意识已经接收到了一些被正式渠道过滤掉的真实信号。

**第一步行动：** 下周不看周报。随机选两个正在开发的功能，直接坐到开发者旁边，让他把当前版本跑给你看。不提前通知、不找经理陪同、不带着"检查"的姿态——就说"我想看看现在长什么样"。你看到的东西和周报描述之间的差距，就是当前信息失真度的真实读数。如果差距显著（比如周报说"核心功能已开发完成"，但实际跑起来页面都打不开），那么你的首要任务不是管项目进度，是修复信息传递链——和团队约定一条"坏消息直达通道"：任何人在发现当前状态和周报不符时，有权直接向你报告而不经过中间管理层。

---

## 场景3：核心开发者离职后，项目瘫痪了三周

**症状：** 一个人离职后，团队发现所有人都不完全知道系统某些模块的逻辑，不知道某些接口和谁对接，不知道某些决策为什么这样做。代码有注释的地方不多，没有决策文档，外部联系人的名单在他个人的邮箱里。项目陷入事实上的停工——团队成员不敢改代码，怕改坏了自己不知道的东西。三周后才逐渐恢复。

**诊断：** 团队对某个关键人物的依赖太深，没有备份。这个人不是"一个成员"——他承载了太多只有他知道的信息：关键决策原因、业务逻辑、外部联系人、技术判断。他的离开不只带走了一个人，还带走了项目方向的关键"导航记忆"。这些信息没有一个地方有备份。他一旦消失，整个团队的协作网络就断裂了。这是项目管理中最危险的单点故障——一个人脑子里装着太多别人不知道的东西，而团队没有能力自行恢复。

**第一步行动：** 立刻做两件事。第一件事：对剩下的成员做一个"知识地图"——用一张白纸列出所有模块和外部接口，每个人用红笔标出"这个模块/接口只有我一个人知道全貌"的点。这些红点就是下一个单点故障风险。针对每个红点，安排一个"知识备份周"——让持有者带着另一个人完整地走一遍整个逻辑链。第二件事：从今天开始，所有重要决策附带2-3句话的"为什么"，写入一个共享文档（不是邮件，邮件会沉）。这个习惯在小团队里感觉是多余的——直到核心成员离开的那一天，你会感谢这个习惯。

## 场景4：每个迭代都在做，但做了半年用户说"这不是我想要的"

**症状：** 迭代在滚动、功能在交付、工作量在累积。燃尽图在迭代内显示正常，团队在评审会上展示了功能，产品负责人说"很好"。但产品上线后，用户反馈冷淡甚至负面——"这不是我想要的"、"操作太复杂了"、"我真正需要解决的问题你们没有碰"。团队很沮丧：我们明明每天都在冲刺。

**诊断：** 从用户真实需求到最后交付的功能，中间经过了层层传递：用户说给销售/客服听，销售/客服说给产品经理听，产品经理写进需求文档，团队按需求文档做。每一层传递中，信息都在不知不觉中变了样。用户说"我要更快地导出报表"，到了团队可能变成"做一个导出按钮"——但用户真正的需求是"我不想每天花两小时手动处理数据"，一个导出按钮解决不了根本问题。更糟的是，每个评审会都没有真实用户参加，团队一直在用"内部验证"替代"真实反馈"——反馈回路断了。偏差在这个断裂处持续累积，直到上线时才一次性爆发出来。

**第一步行动：** 下个迭代的评审会必须邀请至少一位真实用户参加。如果用户不能来，让客服或销售带着用户的原始录音或聊天记录来。评审的焦点不是"我们做了哪些功能"——而是把功能展示给用户（或其代表），问一个开放问题："这个功能解决你一天中的哪个具体时刻的什么问题？"如果回答是"嗯……还行吧"——说明你解决的要么不是他真正的痛点，要么解决方式不对。用户的"还行吧"和"太棒了这就是我要的"之间的差距，就是方向偏了多少。差距不可怕，可怕的是用下一轮迭代继续在错误的方向上加大输出。

## 场景5：回顾每次都有改进项，但下个迭代照样发生

**症状：** 回顾有流程、有参与、有产出。团队在回顾上很活跃，白板上贴满了改进建议——"加强前端后端的沟通"、"提高代码质量"、"减少需求变更"。每个成员都签字同意要改。但到了下个迭代，相同的问题再次出现——前端后端的接口还是没对齐，代码审查还是流于形式，需求还是在中途变更。团队开始认为"回顾没什么用"。

**诊断：** 团队对改变是有上限的。改进项太多了，每个改进项都分散了注意力，结果一个都没落地。更深层的问题是：改进项本身可能是不可执行的。像"加强沟通"这个改进项——没有具体的人、时间、触发条件、验证标准，它不是一个行动，是一个愿望。愿望不占用工时、不进入看板、没有负责人，所以它永远不会被执行。团队在回顾上投入了时间讨论，但没有转化为任何真实的工作方式改变——精力花了，改变没有发生。

**第一步行动：** 下次回顾的最后一步，严格坚持"只选一个改进项"规则。这个改进项必须满足三件事：第一，具体到"谁在什么时间前完成什么事"（例如"张三在下周三前把前端和后端的接口文档补完"）；第二，在下次回顾时能用可见的证据验证（"到时候我们看文档有没有被审查过"）；第三，这个改进项是下个迭代看板上的一张真实任务卡（占用工时、有人负责）。一个落地的改进项，比五个在回顾纪要里睡觉的改进项有用得多。

## 附录一：项目健康度快速评估表

对每个问题打分1-5（1=完全不，5=完全符合）。总分低于30分需要重点关注。

| # | 诊断问题 | 打分(1-5) |
|---|---------|-----------|
| 1 | 每个核心成员能用自己的话说出项目成功的定义吗？（不是背文档，是用自己的语言，像在跟朋友解释一样） | |
| 2 | 最关键的三个隐性干系人，你知道他们最近对这个项目的真实看法吗？（不是"你觉得他们怎么想"，是你最近和他们聊过） | |
| 3 | 上次周报中标记为"正常"的事项，你随机抽查过一个是真实的吗？（不是全部查，是随机抽一个验证） | |
| 4 | 团队目前的加班频率是否不超过每月2天？（含周末主动学习也算加班） | |
| 5 | 如果项目最关键的那个人明天请假两周，项目会受多大影响？（1=完全停滞；5=几乎不受影响） | |
| 6 | 最近一个迭代中，有真实用户在评审会上给了直接反馈吗？（不是产品负责人转述，是用户亲自开口说的） | |
| 7 | 回顾选出的改进项，上一次真正落地是什么时候？（1=从没落地过；5=上上次回顾的改进项已验证完成） | |
| 8 | 从发现一个线上问题到修复部署完成，平均多长时间？（1=超过一周；5=当天内） | |
| 9 | 跨团队或跨模块的对接处，过去一个月发生过因为信息没对齐导致的返工吗？（1=发生过3次以上；5=没有发生过） | |
| 10 | 团队中有人在过去两周内提出过一个改进建议并被认真回应了吗？（"认真回应"=讨论了可行性、安排了执行、或明确解释了为什么不执行） | |

**分数解读：**

- **45-50分：** 项目状态健康。保持现有管理节奏。
- **35-44分：** 存在局部隐患。针对低分问题，在下个迭代制定专项改进。
- **25-34分：** 存在系统性问题——可能信息失真度高、团队能量在透支。建议暂停新功能规划1-2个迭代，集中修复低分项。
- **低于25分：** 项目状态已经比较危险。不建议继续按原计划推进，先做一次全面的项目健康度诊断——重新校准项目目标、测量信息失真程度、检查团队能量水平。
