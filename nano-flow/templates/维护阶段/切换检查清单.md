# 维护期切换检查清单

> **目标：** 确认项目已准备好从"冲刺模式"切换到"维护模式"。切换是一次性的——做了就不回头。

---

## 切换条件

以下条件至少满足一个才能切：

[ ] 主要交付物已交付给用户满 30 天，且无 P0 缺陷
[ ] 其他切换标志：[ ]

---

## 切换前检查清单

### 代码 & 基础设施

- [ ] 核心路径上的 TODO 已清理（或记录在技术债清单中）
- [ ] 构建/部署流程有文档（新人能照着跑通）
- [ ] 关键模块有备份人（不要求能做，但知道找谁、看哪份代码）
- [ ] 监控/告警已配置（如果有的话）

### 文档

- [ ] 接口约定全部归档到 `archive/`
- [ ] 问题列表中所有活跃问题已处理（解决/搁置/调目标）
- [ ] 团队模式总结已完成（`templates/收尾阶段/团队模式总结.md`）
- [ ] 决策日志已完成（`templates/收尾阶段/决策日志.md`）

### 人员

- [ ] 每个仍在维护的模块有明确负责人
- [ ] 团队知道维护期的沟通规则（即时 @ → 月度检查）
- [ ] 发起人确认进入维护期

---

## 切换后的节奏调整

| 项目期 | 维护期 |
|--------|--------|
| 按周推进 | 按需响应 |
| 周会（1 小时） | 月度检查（30 分钟） |
| 每天看 00-项目状态.md | 每周看一次即可 |
| Sprint / 里程碑驱动 | Kanban 连续流 |
| 盯进度和阻塞 | 盯用户反馈和 MTTR |

---

## 切换后的度量切换

| 项目期你盯什么 | 维护期你盯什么 |
|-------------|-------------|
| 做完多少功能 | 用户有没有在骂我们（反馈关键词趋势） |
| 进度快还是慢 | Bug 修复有没有变慢（MTTR 趋势） |
| 计划完成率 | 团队有没有在加班（维护期加班比项目期更危险） |
