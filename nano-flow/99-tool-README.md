# 极简协同工具箱

> **这套工具的设计原则就是它自己推广的方法论：给定当前团队配置，最小动作集合是什么。**

本手册提炼自 ZeroProj 的核心理念，但不是让你全部搬过来。它是一个"什么时候才该加东西"的判断清单。

---

## 定位

`99-tool/` 是**通用参考工具箱**——可封装为 skill，供多个项目复用。不包含任何本项目特有的模板（那些在 `99-current/`）。

## 内容

```
99-tool/
├── README.md                       ← 你在这
├── 敏捷项目-RST.md                  ← 完整实操手册：6 阶段 × 6 核心洞察 × 50+ 步骤
├── 极速稳定交付-实战手册.md          ← 极简实战版：面向小团队的"最小动作集合"
└── templates/                      ← 阶段模板（填表式，拿起来就能用）
    ├── 启动阶段/（6 个模板）
    ├── 规划阶段/（3 个模板）
    ├── 收尾阶段/（4 个模板）
    └── 维护阶段/（2 个模板）
```

### 两本手册怎么选

| | 敏捷项目-RST | 极速稳定交付-实战手册 |
|---|---|---|
| **定位** | 完整的方法论教材 | 面向当前团队配置的实操指南 |
| **长度** | ~1800 行，通读约 3 小时 | ~600 行，通读约 1 小时 |
| **适合** | 学习"为什么这样做"、建立判断框架 | 直接查"现在该做什么" |
| **内容** | 6 个核心洞察 + 6 阶段完整步骤 + 每个步骤的"为什么" | 团队配置诊断 + 最小动作表 + 模板速查 |
| **关系** | 骨架——告诉你每个阶段承担什么功能 | 肌肉——告诉你当前配置下具体做什么 |

**建议：** 先通读"极速稳定交付-实战手册"了解当前该做什么，遇到"为什么这样做"的问题时翻"敏捷项目-RST"对应章节。

## 核心原则

### 不做什么，比做什么更重要

绝大多数项目管理问题不是因为工具太少，而是仪式太多。站会变成了轮流汇报，看板卡片停了一周没人问，燃尽图画得漂亮但核心风险没人碰。

**在加任何东西之前，先问自己：这个工具让团队的决策更快了还是更慢了？**

## 当前配置 & 最小动作

你现在有：

```
00-proj/              ← PM 每周写一次项目状态
01-artifacts/         ← 产物存放
02-sync/              ← 对接记录
03-meeting/           ← 会议材料（PM 会前汇总，会中更新，会后留档）
10-item/{name}/       ← 每人每周写一次个人状态
99-tool/              ← 工具箱（你在这）
```

**这就是全部。不要主动加更多。**

每周五：各人写自己的 `10-item/{name}/YYMMDD.md`，PM 汇总到 `03-meeting/YYMMDD.md`，同步更新 `00-proj/YYMMDD.md`。

## 阶段模板速查

`templates/` 下的模板按项目阶段组织，填表式设计——拿起来就能用。

| 阶段 | 模板 | 什么时候用 |
|------|------|----------|
| **启动** | 团队变量分析、方向红绿笔、三条红线、干系人地图、恐惧清单、团队协议 | 项目第 1-2 天，所有人刚聚齐 |
| **规划** | 角色依赖图、风险预映射、里程碑设定 | 方向确认后、开动前 |
| **收尾** | 方向验证、决策日志、偏差记录、团队模式总结 | 主要交付物完成，开始归档 |
| **维护** | 切换检查清单、月度检查记录 | 切换时一次 + 每月一次 |

**缩减版（紧急启动或团队 < 3 人）：** 至少做方向红绿笔 + 三条红线 + 里程碑设定。

## 痛点 → 最小行动速查

**永远不是"先建好等着"——是痛了才动。**

| 现象 | 可能原因 | 最小行动 | 别做的事 |
|------|---------|---------|---------|
| 同一张卡三天不移动 | 存在未报出的真实阻塞 | 私下问"需要帮忙吗"，**不问"为什么还没做完"** | 不要上燃尽图、不要开复盘会 |
| 没人记问题 | 心理不安全 | **PM 自己先记一条自己的问题** | 不要定"必须记问题"的规则 |
| 周报全绿但直觉不对 | 纸面好看，真实状态被美化 | **不看周报，坐到一个开发者旁边看当前版本** | 不要加更多 KPI |
| 接口对接反复返工 | A 改了输出没通知 B | 在 `02-sync/` 建一个对接文件，谁发现新信息谁更新 | 不要上 Confluence、不要定签字流程 |
| 有人卡住但不好意思说 | 群聊 @ 人心理负担重 | PM 单独私聊问一句 | 不要强迫群聊 @ |
| 知识只在一个人脑子里 | A 给 B 讲半天，下次换人还得讲 | 当事人在 `02-sync/` 写 5 行总结，**只记坑和结论** | 不要建知识库、不要评分体系 |
| 兼职成员信息严重滞后 | 不在群里、不看周报 | 周五下午打电话当面同步，**只问"什么卡住了"** | 不要拉更多群、不要日报 |
| 周会变成轮流汇报 | 没人准备、没人做决策 | **取消周会。改成 PM 扫一遍各人文件，群聊发三句话** | 不要设议程模板、不要轮流发言 |
| 一个文件夹里文件太多 | 历史自然积累 | **不处理。文件多了是好事——说明在跑** | 不要建归档规则、不要自动清理 |

## 什么时候才该加新工具？

### 反模式——不要因为这些理由加

- "别人都有" —— 别人团队跟你不一样
- "万一以后需要" —— 不需要的东西预先建好 = 负担
- "反正文件夹空着也是空着" —— 空的工具给人一种"在用"的错觉，反而掩盖真正缺的东西
- "书上写了要这样做" —— 书上的团队不是你们的团队

### 何时加——三个硬条件

| 条件 | 判断标准 |
|------|---------|
| 痛点反复出现 | 同一个问题连续出现了 3 周以上 |
| 现有工具明确不够 | 不是"用起来不方便"，是"信息传不到" |
| 加一个比不加一个省更多时间 | 维护新增工具的代价 < 现在承受的损失 |

**同时满足三个条件才动手。** 动手也只加一个——不要一次加三个。

### 一票否决

- 你们团队**有人明确反对**——那就不加。
- 加了之后**有人不用的**——那是加错了，删掉。
- 用的过程**比没加之前更痛苦**——那是信号错了，退回去。

---

## 最小修改原则

当确实需要加新工具时：

1. **只加一个。** 试两周再看要不要加第二个。
2. **从最简单的方式开始。** 如果一行 Markdown 够用，别上 JSON。如果一张表够用，别建文件夹。
3. **不改动已有文件的结构。** 新东西追加在末尾、新建文件、新建文件夹——**永远不重构旧文件**。
4. **如果两周后发现没用，删掉。** 不留"万一有用"的东西。

## 什么时候从极简升级到完整版

以下是当前项目（5 人，星形枢纽，有兼职）的扩展信号：

| 信号 | 引入什么 | 在哪 |
|------|---------|------|
| 团队扩展到 7 人以上 | 接口契约活文档（CON-xxx.md） | 敏捷项目-RST |
| 跨角色知识传递频繁卡住 | 知识沉淀体系 | 敏捷项目-RST |
| 多个项目同时跑，资源冲突 | 跨项目诊断面板 | 敏捷项目-RST |
| PM 肉眼扫不过来了 | AI 聚合脚本 | 敏捷项目-RST |
| 用户角色 ≥ 2 种，需求依赖复杂 | 用户故事地图 | 敏捷项目-RST 步骤 2.1 |
| 需要预测交付日期 | 吞吐量统计 + Cycle Time | 敏捷项目-RST |
| 团队能量明显下降但不知道为什么 | 团队能量五维评估 | 敏捷项目-RST 步骤 2.3 |
| 项目进入维护期 | 维护期月检记录 | templates/维护阶段/ |
| 第一次用本工具箱 | 启动阶段 5 个模板全部走一遍 | templates/启动阶段/ |

**原则：** 痛点没到，不要回头看。工具是河对岸的药铺——你还在河的这边，先把桥建了再想药的事。
